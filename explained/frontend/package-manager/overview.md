# [UNVERIFIED] 패키지 매니저는 무엇이며 어떤 역할을 하는가?

## 도입

현대 JS 개발에서 `npm install` 한 줄이 수백 개의 의존성을 자동으로 설치해주는 것을 당연하게 여기지만, 이 자동화를 제공하는 것이 패키지 매니저다. 단순히 파일을 다운로드하는 것을 넘어 버전 관리와 호환성 검증까지 포함한다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**기본 역할.** 패키지 매니저는 다른 개발자가 만든 코드(패키지)를 다운로드·설치·관리하는 도구다. npm(Node Package Manager), yarn, pnpm이 대표적이다.

**버전 관리.** 패키지마다 버전이 있고, 프로젝트가 요구하는 버전 범위(`^18.0.0` 등)와 실제 설치 버전 사이의 관계를 관리한다. 버전 범위 표기(`^`, `~`, 정확한 버전)에 따라 어떤 업데이트를 허용할지 결정한다.

**호환성 체크.** `peerDependencies`로 선언된 호환성 요구사항을 자동으로 검증한다. 예: `react-dom`은 `react`와 같은 버전이어야 한다는 제약을 PM이 확인한다.

---

## 종합

패키지 매니저는 "다른 사람의 코드를 내 프로젝트에 안전하게 가져오는 인프라"다. PM 없이 `node_modules`를 직접 관리하면 수백 개 패키지의 버전·의존성을 손으로 추적해야 한다. 이것이 다음 질문에서 자세히 다룰 내용이다.

# [UNVERIFIED] 패키지 매니저가 없으면 어떤 문제가 생기는가?

## 도입

npm이 없던 시절을 상상하면 PM의 가치를 이해할 수 있다. 모든 것을 수동으로 처리하면 어떤 일이 생기는지가 PM이 해결하는 문제를 그대로 보여준다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**수동 다운로드.** 라이브러리를 GitHub에서 zip으로 받아 프로젝트에 복사해야 한다. 그 라이브러리가 의존하는 다른 라이브러리도 같은 방식으로 직접 찾아서 받아야 한다. 의존성 트리가 깊어지면 이 과정이 기하급수적으로 복잡해진다.

**버전 충돌.** 두 라이브러리가 같은 패키지의 다른 버전을 요구할 때, 어느 버전을 써야 하는지 직접 파악하고 해결해야 한다.

**환경 의존성 문제.** "내 컴퓨터에서는 됐는데 서버에서 안 된다"는 문제는 서로 다른 버전이 설치됐을 때 발생한다. lockfile 없이는 `npm install`이 팀원마다 다른 버전을 설치할 수 있다.

```
PM 없는 세계
개발자 A: lodash 4.17.20 수동 설치
개발자 B: lodash 4.17.21 수동 설치
CI 서버: lodash 4.17.19 (다른 날 세팅)
→ "내 컴퓨터에서는 됐는데..."
```

---

## 종합

PM이 해결하는 문제의 핵심은 "재현 가능한 환경"이다. `package.json`과 lockfile이 있으면 어떤 컴퓨터에서 `npm install`을 실행해도 동일한 의존성 트리가 만들어진다. 이 재현성이 팀 협업과 CI/CD를 가능하게 하는 기반이다.

# [UNVERIFIED] 패키지 매니저가 주는 효과는 무엇인가?

## 도입

패키지 매니저가 없을 때의 문제를 해결하는 것이 PM의 효과다. 핵심은 설치 자동화, 호환성 검증, lockfile 기반 재현성이다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**설치 자동화.** `yarn install` 또는 `npm install` 한 줄로 `package.json`에 선언된 모든 의존성과 그 의존성의 의존성까지 자동으로 설치한다. 수백 개의 패키지를 직접 다운로드하지 않아도 된다.

**호환성 검증.** `peerDependencies`에 선언된 제약을 자동으로 체크한다. 예를 들어 `react-dom@18`은 `react@18`을 요구하는데, 설치된 `react`가 다른 버전이면 경고 또는 오류를 낸다. 런타임에 조용히 깨지는 것을 설치 시점에 방지한다.

**lockfile 기반 재현성.** `package-lock.json`(npm), `yarn.lock`(Yarn), `pnpm-lock.yaml`(pnpm)이 정확한 버전과 의존성 트리를 고정한다.

```json
// package.json: 버전 범위 선언
{
  "dependencies": {
    "react": "^18.0.0"   // 18.x 허용
  }
}

// yarn.lock: 실제 설치 버전 고정
react@^18.0.0:
  version "18.2.0"       // 팀 모두 동일 버전
  resolved "..."
```

lockfile을 git에 커밋하면 팀 전체, CI, 배포 환경이 동일한 버전을 받는다.

---

## 종합

패키지 매니저의 세 효과(자동화, 호환성, 재현성)는 서로 맞물린다. 자동화가 없으면 재현성을 유지하기 어렵고, 재현성이 없으면 호환성 문제를 추적하기 어렵다. `npm install`이 단 한 줄로 끝나는 당연함 뒤에 이 세 가지가 동시에 동작하고 있다.
