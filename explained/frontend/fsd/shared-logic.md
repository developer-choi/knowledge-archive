# [UNVERIFIED] 두 슬라이스가 같은 로직을 공유해야 할 때 어떻게 처리하는가?

## 도입

같은 레이어의 슬라이스끼리는 직접 참조가 금지된다. 그런데 두 슬라이스가 동일한 로직을 필요로 하는 상황이 생기면 어떻게 해야 할까? FSD는 "공유 방향"에 따라 세 가지 해결책을 제시한다.

---

## 본문

이 질문의 OA는 검증되지 않은(AI Answer) 내용이다. FSD 공식 문서 기반 원칙에서 도출한 접근 패턴이다.

세 가지 해결책:

**1. Shared 레이어로 내리기**

공유하려는 로직이 특정 비즈니스와 무관한 순수 유틸리티라면 Shared 레이어로 이동한다.

- 판단 기준: 이 코드가 user, product, post 같은 도메인 개념 없이도 의미가 있는가?
- 예: 날짜 포매터, HTTP 클라이언트 설정, 범용 UI 컴포넌트(Button, Input)

**2. 하위 레이어(Entities)로 내리기**

두 Features가 동일한 데이터 모델을 공유한다면, 그 로직을 Entities 레이어의 관련 슬라이스로 옮긴다.

- 판단 기준: 이 코드가 특정 비즈니스 엔티티(user, product 등)를 다루는가?
- 예: user 타입, user 관련 API GET 함수, user 데이터를 보여주는 UI

**3. 상위 레이어(Widgets/Pages)에서 조합**

두 슬라이스를 직접 연결하지 않고, 상위 계층에서 각각을 가져와 로직을 조립한다.

- 판단 기준: 두 슬라이스의 상호작용이 특정 화면 맥락에 의존하는가?
- 예: Post 목록과 User 프로필을 함께 보여주는 화면 — features/post-list와 entities/user를 Pages에서 조립

```
두 Features가 공유 로직 필요 시 판단 흐름:

로직이 도메인 없는 순수 유틸?
  → Shared로 내리기

로직이 특정 엔티티 데이터를 다룸?
  → Entities로 내리기

상호작용이 특정 화면에 종속?
  → Pages/Widgets에서 각각 import해 조합
```

---

## 종합

FSD에서 "공유"의 방향은 항상 아래 레이어 또는 위 레이어를 향한다. 같은 레이어 슬라이스끼리 직접 공유하는 길은 없다. 어떤 코드가 두 곳에서 필요하다면 그것은 더 낮은 추상화 레벨에 속하거나(Shared, Entities), 더 높은 레이어에서 조립되어야 할(Widgets, Pages) 코드라는 신호다. 이 제약이 처음엔 불편하게 느껴지지만, 코드베이스가 커질수록 "어디에 있어야 하는가"가 명확한 덕에 탐색 비용이 줄어든다.
