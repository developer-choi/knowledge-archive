# MSW를 셋업해두기만 하고 어떤 요청에 대한 핸들러를 안 적었다면, 그 요청은 어떻게 처리되는가?

## 도입

MSW(Mock Service Worker, 네트워크 요청을 가로채 가짜 응답을 돌려주는 목킹 도구)를 처음 붙일 때 흔히 이렇게 상상한다 — "일단 MSW 서버를 켜두면(listen 시작), 앱에서 나가는 모든 네트워크 요청이 MSW의 손아귀에 들어와 막히거나 가로채질 것이다." 그래서 "핸들러를 아직 안 적은 요청은 어디에도 못 나가고 붕 뜨는 것 아닌가" 하고 걱정한다.

실제 동작은 이 상상과 정반대다. 이 질문은 "핸들러를 안 적어둔 요청"이 MSW 아래에서 어떤 취급을 받는지를 다룬다.

## 본문

### network-first — 명시하지 않으면 간섭하지 않는다

공식 문서는 MSW의 기본 철학을 이렇게 한 문장으로 못 박는다.

> MSW embraces a network-first approach, which means that it will not interfere with the network unless you explicitly say so in your handlers.

"MSW는 network-first(네트워크 우선) 접근을 취한다. 즉, 당신이 핸들러에서 **명시적으로 그렇게 하라고 말하지 않는 한** 네트워크에 간섭하지 않는다."

여기서 network-first란 "기본값은 진짜 네트워크, 목킹은 예외"라는 뜻이다. MSW 서버를 켜는 것(셋업)은 "이제부터 요청을 가로챌 준비가 됐다"는 대기 상태를 만드는 것일 뿐, 그 자체로 트래픽을 잡아채기 시작하는 스위치가 아니다. 실제로 어떤 요청을 잡아챌지는 전적으로 당신이 핸들러에 적어둔 목록이 결정한다.

### 핸들러에 적은 요청만 가로채고, 나머지는 그대로 통과한다

핸들러(handler)란 "이 URL·이 메서드의 요청이 오면 이렇게 목(mock)해라"라고 적어두는 규칙 하나하나를 말한다. MSW는 나가는 요청을 볼 때마다 이 핸들러 목록과 대조한다.

- 핸들러에 **명시적으로 적어둔** 요청 → MSW가 가로채서 당신이 지시한 대로(가짜 응답 등) 처리한다.
- 핸들러에 **없는** 요청 → MSW가 간섭하지 않고 원래 가려던 실제 네트워크로 그대로 내보낸다.

이 "적어두지 않은 요청은 손대지 않고 실제 서버로 그대로 흘려보내는" 기본 동작을 passthrough(패스스루, 그대로 통과)라고 부르며, 이것이 MSW의 **기본값(default)**이다. 즉 핸들러가 없다고 요청이 실패하거나 막히는 게 아니라, MSW가 애초에 관여하지 않은 것처럼 진짜 서버까지 무사히 나간다.

그래서 앞서 도입에서 말한 "MSW를 켜두면 모든 요청이 가로채진다"는 직관은 뒤집어야 한다. 정확히는 이렇다 — **명시하지 않으면 아무 일도 일어나지 않는다.** 목킹은 당신이 핸들러로 하나씩 지목한 요청에 대해서만 발동하는, 선택적으로 켜는 기능이다.

## 종합

MSW는 network-first(네트워크 우선) 도구라서, 서버를 켜두기만 하고 핸들러를 안 적은 요청은 가로채지 않고 실제 네트워크로 그대로 통과(passthrough)시킨다 — 이 통과가 기본값이다. "MSW를 켜두면 모든 요청이 막히거나 가로채진다"는 흔한 오해와 반대로, 핸들러에 명시적으로 지목한 요청만 목킹되고 나머지는 간섭 없이 진짜 서버로 나간다.

# 하나의 요청이 여러 핸들러에 매칭될 때, 어떤 핸들러가 그 요청을 처리하는가?

## 도입

핸들러를 여러 개 등록하다 보면 같은 URL·같은 메서드를 겨냥한 핸들러가 둘 이상 생길 수 있다. 예를 들어 `GET /user`를 잡는 핸들러가 세 개 나란히 있다면, `GET /user` 요청이 들어왔을 때 이 세 핸들러가 모두 발동해 뒤죽박죽 처리될까? 아니면 그중 하나만 골라질까? 골라진다면 무슨 기준으로 고를까?

이 질문은 "한 요청 ↔ 여러 매칭 핸들러"의 관계에서 실제로 누가 그 요청을 책임지는지를 다룬다.

## 본문

### fallthrough — 핸들러 목록을 위에서 아래로 훑어 첫 매칭을 찾는다

공식 문서의 설명이다.

> Every intercepted request "falls through" the list of your handlers, looking for the first matching handler to return an instruction on how to handle the request (mock response, response-patch, passthrough, do nothing). A single request may match multiple handlers at the same time but only one handler can be responsible for handling it.

"가로챈 모든 요청은 당신의 핸들러 목록을 따라 '흘러내리며(falls through)', 요청을 어떻게 처리할지에 대한 지시를 돌려주는 **첫 번째 매칭 핸들러**를 찾는다. 하나의 요청이 동시에 여러 핸들러에 매칭될 수는 있지만, 그 요청을 책임지고 처리할 수 있는 핸들러는 오직 하나뿐이다."

fallthrough(폴스루, 흘러내림)란 가로챈 요청이 핸들러 배열을 위에서 아래로 한 칸씩 훑어 내려가는 동작을 말한다. MSW는 각 핸들러에 요청을 대보며 "이 핸들러가 이 요청을 어떻게 할지 **지시(instruction)를 돌려주는가**"를 본다. 지시의 종류는 네 가지다.

- **mock response(목 응답)** — 가짜 응답을 만들어 돌려준다.
- **response-patch(응답 패치)** — 실제 응답을 받아 일부만 덮어써서 돌려준다.
- **passthrough(패스스루)** — 손대지 않고 실제 서버로 그대로 통과시킨다.
- **do nothing(아무것도 안 함)** — 지시를 내지 않고 다음으로 넘긴다.

핵심 규칙은 뒤 문장이다 — 한 요청이 여러 핸들러에 **매칭**될 수는 있어도, 실제로 그 요청을 **책임지고 처리하는** 핸들러는 정확히 하나다. 위에서부터 훑다가 처음으로 지시(위 네 가지 중 하나)를 돌려주는 핸들러를 만나면 거기서 처리가 확정되고, 그 아래 핸들러들은 매칭이 되든 말든 아예 도달하지 못한다.

### 코드로 따라가기 — "One"은 찍히고 "Three"는 안 찍힌다

문서의 예시가 이 규칙을 그대로 보여준다.

```ts
import { http, HttpResponse } from 'msw'

export const handlers = [
  http.get('/user', () => console.log('One')),
  http.get('/user', () => HttpResponse.json({ name: 'John' })),
  http.get('/user', () => console.log('Three')),
]
```

세 핸들러 모두 `GET /user`에 매칭된다. `GET /user` 요청이 오면 위에서부터 흘러내리며 이렇게 진행된다.

- **첫 번째 핸들러**: 매칭된다. 하지만 `console.log('One')`만 실행하고 아무 값도 반환하지 않는다 — 즉 "어떻게 처리하라"는 지시를 돌려주지 않았다. 그래서 이 요청은 아직 처리되지 않은 채 계속 흘러내린다. (콘솔에 "One"이 찍힌다.)
- **두 번째 핸들러**: 다시 매칭된다. 이번엔 `HttpResponse.json({ name: 'John' })`을 반환한다 — 목 응답이라는 지시를 돌려준 것이다. 여기서 이 요청의 처리가 **확정·종료**된다. 요청자는 `{ name: 'John' }`이라는 가짜 JSON을 응답으로 받는다.
- **세 번째 핸들러**: 매칭 대상이긴 하지만 요청이 이미 두 번째에서 끝났으므로 **도달조차 하지 않는다.** `console.log('Three')`는 실행되지 않아 "Three"는 콘솔에 찍히지 않는다.

정리하면 콘솔에는 "One"만 찍히고, 응답은 두 번째 핸들러의 목 JSON이며, "Three"는 나타나지 않는다. "매칭된 핸들러가 셋"이어도 "책임지고 처리한 핸들러는 두 번째 하나"인 것이다.

## 종합

가로챈 요청은 핸들러 배열을 위에서 아래로 흘러내리며(fallthrough) 처리 지시(목 응답·response-patch·passthrough·아무것도 안 함)를 돌려주는 첫 번째 매칭 핸들러를 찾는다. 한 요청이 여러 핸들러에 동시에 매칭될 수는 있어도 실제로 책임지고 처리하는 핸들러는 오직 하나이며, 그 첫 매칭 핸들러가 지시를 반환하는 순간 처리가 끝나 뒤 핸들러들엔 도달하지 않는다. 그래서 세 개의 `GET /user` 핸들러 예시에서 "One" 로그는 찍히고 두 번째의 목 JSON이 응답되며 "Three"는 실행조차 되지 않는다.
