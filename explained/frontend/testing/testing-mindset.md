# 테스트 코드가 필요한 이유는 무엇인가?

## 도입

[UNVERIFIED] 표시된 질문으로, User Answer 기반이다. 테스트 코드가 없으면 코드를 수정할 때마다 기존 기능이 여전히 동작하는지 사람이 손으로 확인해야 한다. 이 확인 비용이 누적되면 변경에 두려워지고 개발 속도가 느려진다.

---

## 본문

수정한 코드가 문제가 없는지를 검증하기 위해서다. 사람이 매번 같은 시나리오를 손으로 확인할 수는 없으므로, 코드로 코드를 검증한다.

테스트 코드의 핵심 역할은 두 가지다.

첫째, **회귀 방지**: 새 기능을 추가하거나 기존 코드를 수정할 때 이미 동작하던 기능이 깨졌는지 자동으로 감지한다.

둘째, **변경 자신감**: 테스트가 통과하면 수정이 기존 기능을 망가뜨리지 않았다는 확신을 가지고 배포할 수 있다.

---

## 종합

테스트 코드가 없으면 "내가 이 함수를 고쳐도 괜찮을까"라는 질문에 손으로 직접 확인하거나, 확인을 생략하고 배포했다가 프로덕션에서 버그를 발견하는 두 가지 선택지만 남는다. 테스트는 이 확인 비용을 한 번 투자로 자동화한다 — 이후 매번 변경할 때마다 무료로 재실행된다. 코드베이스가 클수록, 팀이 클수록 이 자동화의 가치는 커진다.

# QA가 있는데도 왜 테스트 코드가 필요한가?

## 도입

[UNVERIFIED] 표시된 질문으로, User Answer 기반이다. QA팀이 있다면 "테스트 코드가 왜 필요한가"라는 질문이 자연스럽게 나온다. 테스트 코드는 QA를 대체하는 것이 아니라 서로 다른 종류의 결함을 잡는다.

---

## 본문

테스트 코드는 QA를 대체하는 것이 아니라 보완하는 것이다. 두 활동은 목적, 시점, 비용이 다르므로 서로 다른 종류의 결함을 잡는다.

```
항목          QA                     테스트 코드
----------    -------------------    ---------------------------
주체          사람 (QA 엔지니어)     자동화된 코드
시점          기능 완성 후           코드 작성과 동시에 / CI
검증 대상     시나리오, UX, 시각     함수·컴포넌트 단위, 회귀
반복 비용     매번 사람이 재수행     한 번 작성 → 자동 반복
잡는 결함     사용성·UX·예상 외 흐름   로직 오류·회귀
```

---

## 종합

QA는 사용자 시나리오 전체를 흐름으로 검증하고, 시각적 요소와 UX 감각을 평가할 수 있다. 테스트 코드는 함수 단위 로직을 빠르고 반복적으로 검증하며, 회귀를 자동으로 잡는다. 버튼 색이 이상한 것은 테스트 코드가 잡기 어렵지만, 할인 계산 로직이 1원 틀리는 것은 테스트 코드가 정확하게 잡는다. 두 활동을 병행할 때 커버리지가 가장 넓어진다.

# 어떤 기능이 테스트코드를 작성할 만큼 중요한가?

## 도입

[UNVERIFIED] 표시된 질문으로, User Answer 기반이다. 모든 기능에 테스트를 쓰는 것은 비현실적이다. 우선순위를 정하는 4가지 기준이 있다.

---

## 본문

모든 기능에 동일한 비중으로 테스트를 작성할 수는 없으므로, 다음 4가지 기준으로 우선순위를 정한다.

- **비즈니스적으로 더 중요한 기능**: 결제, 로그인, 핵심 구매 흐름처럼 서비스의 수익과 직결되는 기능. 커뮤니티 댓글보다 결제 프로세스가 우선이다.
- **코드가 복잡한 기능**: 조건문이 많아 경우의 수가 많은 코드. `if/else`가 5개 이상 겹쳐있는 할인 계산 로직 같은 것.
- **과거에 한 번 이상 오류가 발생했던 기능**: 회귀 가능성이 높다. 한 번 깨진 곳은 또 깨진다.
- **많은 곳에서 사용하는 공통 함수**: 영향 범위가 넓어 깨질 때 파급력이 크다. `formatDate`, `calculatePrice` 같은 유틸 함수.

---

## 종합

테스트를 작성하는 비용은 한정적이므로 투자 대비 효과를 극대화해야 한다. 위 4기준은 "깨졌을 때 피해가 큰 것"과 "깨질 가능성이 높은 것"을 교집합으로 가리킨다. 처음 테스트를 도입하는 팀은 이 기준으로 목록을 만들고 상위 5개만 먼저 테스트를 쓰는 것으로 시작하면 된다. 100% 커버리지는 목표가 아니라 결과다.

# 부모 함수가 자식 함수를 호출할 때, 테스트를 어떻게 분배해야 하는가?

## 도입

[UNVERIFIED] 표시된 질문으로, User Answer 기반이다. 함수 A가 함수 B를 내부적으로 호출하는 구조에서, 두 함수를 독립적으로 모두 철저하게 테스트하면 중복이 생기고 유지보수 비용이 커진다.

---

## 본문

부모 함수가 자식 함수를 내부적으로 호출하는 구조라면, 다음 3단계로 테스트를 분배한다.

- **부모 함수를 많이 테스트한다**: 자식 함수의 동작을 포함하여 실제 사용 흐름을 검증한다. 부모를 통해 자식의 동작도 간접적으로 커버된다.
- **자식 함수에서는 부모로부터 잘 전파되는지만 짧게 확인한다**: 부모에서 이미 검증한 케이스를 자식에서 중복 테스트하지 않는다.
- **자식 함수에서 확장한 부분만 따로 테스트한다**: 자식 고유의 추가 로직만 별도로 커버한다.

```ts
// 예시: validateString → validateIncludeString 구조

// validateString: 문자열인지 확인
// validateIncludeString: validateString + 특정 문자열 포함 확인

// 부모 (validateIncludeString) 테스트 — 많이
test('빈 문자열이면 invalid', ...);     // 자식(validateString) 동작도 커버
test('특정 문자 포함 시 valid', ...);   // 부모 고유 로직
test('특정 문자 미포함 시 invalid', ...);

// 자식 (validateString) 테스트 — 적게
test('숫자가 들어오면 invalid', ...);   // validateString 고유 케이스만
// '빈 문자열' 케이스는 부모에서 이미 커버했으므로 중복 불필요
```

---

## 종합

부모를 더 많이 테스트하고 자식은 확장 로직만 테스트하면 중복이 줄고 테스트 파일이 가벼워진다. 자식 함수를 철저히 테스트했어도 부모가 자식을 잘못 호출하면 버그가 생긴다 — 그래서 부모 테스트가 실제 사용 흐름을 반영하는 것이 중요하다. "테스트는 코드의 사용 흐름을 문서화한다"는 관점에서 보면, 자식보다 부모의 관점에서 쓴 테스트가 더 많은 정보를 전달한다.
