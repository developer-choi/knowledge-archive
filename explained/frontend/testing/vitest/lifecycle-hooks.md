# 매 테스트마다 반복되는 준비·정리 코드를 복붙하지 않으려면 Vitest에서 무엇을 쓰는가?

## 도입

테스트를 여러 개 쓰다 보면 "매번 똑같이 준비하고, 끝나면 똑같이 치우는" 코드가 반복된다. 데이터 초기화, 데이터베이스(DB) 연결, 테스트용 서버 켜기 같은 것들이다. 이걸 테스트마다 복붙하면 지저분하고, 한 군데 고칠 때 전부 고쳐야 한다. Vitest는 이 문제를 위해 "생명주기 훅(lifecycle hooks)"을 제공한다.

---

## 본문

먼저 단어부터 정리하자. 여기서 말하는 **"훅(hook)"은 "정해진 시점에 내 코드를 대신 실행해주는 자리(슬롯)"** 다. 예를 들어 "각 테스트가 시작되기 직전"이라는 시점에 걸어둔 슬롯이 있고, 거기에 내 준비 코드를 넣어두면 Vitest가 그 시점마다 알아서 실행해준다. 즉 훅이 답하는 질문은 딱 하나, **"내 코드가 언제 실행되는가?"** 다.

이 "훅"이라는 말을 **"픽스처(fixture)"** 와 헷갈리기 쉬운데, 둘은 답하는 질문이 다르다. 훅은 "언제 실행하나"를 정하는 자리이고, 픽스처는 "테스트에 무슨 재료(데이터·객체)를 넣어주나"에 가깝다. 지금은 픽스처는 잊고, **훅 = 실행 시점 슬롯** 이라는 것만 잡으면 된다. 예를 들어 `beforeEach`는 하나의 훅(=슬롯)이다.

Vitest에 어떤 훅들이 있는지 가족 단위로 이름만 먼저 훑어두자. 뒤 질문에서 하나씩 자세히 다룬다.

- `beforeEach` / `afterEach` — 각 테스트 하나하나의 전/후
- `beforeAll` / `afterAll` — 파일 전체에서 딱 한 번씩, 맨 앞/맨 뒤
- `onTestFinished` / `onTestFailed` — 테스트 함수 안에서 쓰는 특수한 것들

이 훅들에 준비·정리 코드를 한 번만 등록해두면, 각 테스트에 복붙할 필요 없이 Vitest가 알맞은 순간에 실행해준다.

---

## 종합

생명주기 훅은 "정해진 시점에 내 코드를 자동 실행해주는 자리"다. 준비(초기화·DB연결·서버기동)와 뒷정리를 훅에 한 번만 등록하면 매 테스트에 복붙하지 않아도 된다. 훅은 "언제 실행되나"를 정하는 것이고, "무슨 재료를 넣나"의 픽스처와는 별개 개념이다.

---

# 파일 안 모든 테스트에 각각 준비·정리를 걸려면 어떤 훅을 쓰며, 각각 언제 실행되는가?

## 도입

한 테스트 파일 안에 테스트가 5개 있다고 하자. 각 테스트가 시작하기 전에 깨끗한 상태를 만들어주고, 끝난 뒤 뒷정리를 해주고 싶다. 이럴 때 쓰는 게 `beforeEach`와 `afterEach`다.

---

## 본문

- **`beforeEach`** — 모든 테스트 각각의 실행 **직전** 에 매번 실행된다.
- **`afterEach`** — 모든 테스트 각각의 실행 **직후** 에 매번 실행된다.

```ts
import { beforeEach, afterEach, test } from 'vitest'

beforeEach(() => {
  // 각 테스트 시작 전마다 실행: 상태를 깨끗하게 준비
})

afterEach(() => {
  // 각 테스트 끝난 후마다 실행: 뒷정리
})
```

이 두 훅 덕분에 각 테스트는 **"알려진 깨끗한 상태에서 시작"** 하는 것이 보장된다. 앞 테스트가 뭘 어질러 놨든, `beforeEach`가 새로 준비하고 `afterEach`가 치워주니 서로 간섭하지 않는다.

---

## 종합

`beforeEach`는 각 테스트 직전에, `afterEach`는 각 테스트 직후에 매번 실행된다. 결과적으로 모든 테스트가 같은 깨끗한 상태에서 출발하게 된다.

---

# 테스트가 도중에 실패해도 afterEach(정리 훅)는 실행되는가?

## 도입

`afterEach`에 뒷정리를 걸어뒀는데, 테스트가 중간에 실패하면 그 정리는 건너뛰어질까? 실패한 테스트일수록 남긴 찌꺼기를 치워야 하니, 이 보장 여부가 중요하다.

---

## 본문

**`afterEach`는 테스트가 도중에 실패해도 실행된다.** 테스트가 실패했다고 뒷정리를 건너뛰면, 치우지 못한 찌꺼기가 다음 테스트로 새어 나가기 때문이다. 그래서 Vitest는 성공·실패와 무관하게 `afterEach`를 보장해서 실행한다.

---

## 종합

`afterEach`는 성공이든 실패든 매 테스트 후 반드시 실행되므로, 실패한 테스트의 뒷정리도 건너뛰지 않는다. 덕분에 한 테스트의 실패가 남긴 상태가 다음 테스트로 새는 것을 막는다.

---

# 한 파일의 테스트들이 매번 상태를 리셋하지 않으면 어떤 문제가 생기며, 훅이 이를 어떻게 해결하는가?

## 도입

훅을 안 쓰고 테스트들이 상태를 공유한 채로 두면 어떤 사고가 나는지 구체적으로 보자. 이건 실무에서 아주 흔한 버그의 원인이다.

---

## 본문

예를 들어 여러 테스트가 하나의 공유 배열을 같이 쓴다고 하자.

```ts
const items: string[] = []

test('A', () => {
  items.push('a')
  expect(items.length).toBe(1) // 통과
})

test('B', () => {
  expect(items.length).toBe(0) // 실패! A가 밀어넣은 'a'가 남아 있음
})
```

테스트 A가 공유 배열에 값을 넣었는데 아무도 치우지 않으면, 그 흔적이 뒤 테스트 B로 **새어 나간다(leak)**. B는 빈 배열을 기대했는데 A의 찌꺼기 때문에 실패한다.

더 고약한 건, 테스트 실행 순서나 어떤 테스트가 먼저 돌았느냐에 따라 결과가 달라진다는 점이다. 이렇게 **"이전에 무엇이 돌았느냐에 따라 됐다 안 됐다 하는 테스트"를 플레이키 테스트(flaky test)** 라고 부른다. 원인을 잡기 매우 까다롭다.

훅은 이 문제를 근본에서 없앤다. `beforeEach`에서 상태를 매번 새로 초기화하거나, `afterEach`에서 매번 치우면, 각 테스트가 **깨끗한 상태에서 독립적으로** 시작한다. 서로 간섭할 여지가 사라진다.

```ts
let items: string[]
beforeEach(() => {
  items = [] // 매 테스트 시작 전 새 배열로 리셋
})
```

### 대표 사례 — mock의 호출 기록도 같은 상태다

이 "상태 누수"의 가장 흔한 실제 사례가 **mock(가짜 함수)**이다. mock은 자기가 몇 번·어떤 인자로 불렸는지를 스스로 적어두는 "호출 수첩"을 상태로 갖는데, 이 수첩도 위의 공유 배열과 똑같이 리셋하지 않으면 다음 테스트로 새어 든다. 그래서 mock 역시 `beforeEach`/`afterEach`에서 매 테스트마다 비워, 각 테스트가 깨끗한 상태에서 출발하게 한다.

즉 여기서 다루는 "테스트가 남긴 상태를 매 테스트마다 리셋한다"는 원리는 공유 배열에만 국한되지 않고 mock의 호출 기록에도 똑같이 적용된다. mock의 수첩을 되돌리는 구체적 수단(`mockClear`/`mockReset`/`mockRestore`, 전역 `restoreMocks` 설정 등)은 이 원리를 mock에 적용한 것으로, [`../mocking/overview.md`](../mocking/overview.md)와 [`../mocking/mock-functions.md`](../mocking/mock-functions.md)에서 다룬다.

---

## 종합

상태를 리셋하지 않으면 한 테스트가 건드린 공유 상태가 다음 테스트로 새어 나가, 실행 순서에 따라 됐다 안 됐다 하는 플레이키 테스트가 된다. `beforeEach`/`afterEach`로 매 테스트마다 상태를 초기화·정리하면 각 테스트가 독립적으로 돌아 이 문제가 사라진다. 이 원리는 공유 배열뿐 아니라 mock의 호출 기록 등 "테스트가 남기는 모든 상태"에 똑같이 적용되며, mock에 대한 구체적 리셋 수단은 `mocking/overview.md`·`mocking/mock-functions.md`가 다룬다.

---

# DB 연결·서버 기동처럼 매 테스트마다 반복하기엔 너무 비싼 준비 작업은 어떤 훅으로 처리하며, beforeEach와 무엇이 다른가?

## 도입

`beforeEach`는 편하지만 "매 테스트마다" 실행된다. 그런데 DB에 연결하거나 서버를 켜는 작업은 한 번 하는 데도 시간이 꽤 걸린다. 테스트 100개마다 100번 연결하면 테스트가 한없이 느려진다. 이럴 때 쓰는 게 `beforeAll`/`afterAll`이다.

---

## 본문

- **`beforeAll`** — 파일 전체에서 **딱 한 번**, 모든 테스트가 시작되기 전에 실행된다.
- **`afterAll`** — 파일 전체에서 **딱 한 번**, 모든 테스트가 끝난 뒤에 실행된다.

`beforeEach`와의 차이는 "몇 번 실행되느냐"다.

| 훅 | 실행 횟수 |
|----|-----------|
| `beforeEach` | 테스트 개수만큼 (매 테스트) |
| `beforeAll` | 파일당 1번 |

DB 연결, 서버 기동, 큰 파일 읽기처럼 **반복하기엔 너무 비싼** 준비는 `beforeAll`에서 한 번만 만들어두고, 모든 테스트가 그 자원을 공유해서 쓰고, 파일이 끝날 때 `afterAll`에서 닫아주면 된다.

```ts
import { beforeAll, afterAll } from 'vitest'

let db: Db
beforeAll(async () => {
  db = await connectDb() // 파일 전체에서 한 번만 연결
})
afterAll(async () => {
  await db.close() // 모든 테스트 끝나면 한 번만 닫기
})
```

주의: `beforeAll`로 만든 자원은 모든 테스트가 **공유** 하므로, 테스트가 그 자원의 상태를 바꿔놓으면 다시 앞의 플레이키 문제가 생길 수 있다. "비싸고 읽기 위주라 공유해도 안전한 것"은 `beforeAll`, "테스트마다 깨끗해야 하는 상태"는 `beforeEach`로 나누는 게 요령이다.

---

## 종합

`beforeAll`/`afterAll`은 파일 전체에서 딱 한 번씩 실행된다. DB 연결·서버 기동처럼 매 테스트마다 반복하기엔 비싼 준비는 여기서 한 번만 하고 공유한 뒤 파일 끝에 닫는다. 매 테스트마다 도는 `beforeEach`와는 실행 횟수가 근본적으로 다르다.

---

# onTestFinished 훅은 무엇을 하며, beforeEach·afterEach와 달리 어디에서 호출해야 하는가?

## 도입

지금까지의 훅들은 테스트 함수 **바깥** 에 미리 등록해뒀다. 그런데 자원을 테스트 함수 **안** 에서 만드는 경우도 있다. 그때 뒷정리를 어디에 두느냐가 문제인데, `onTestFinished`가 이걸 깔끔하게 푼다.

---

## 본문

**`onTestFinished`는 테스트 함수 "안에서" 호출** 해, "이 테스트가 끝나면 이걸 치워라"라고 정리 작업을 등록하는 훅이다. `beforeEach`·`afterEach`가 테스트 바깥에 등록되는 것과 정반대다. 테스트 본문이 실행되는 도중에 호출해야 하고, **테스트 본문 바깥에서 부르면 에러를 던진다.**

왜 필요할까? 자원을 테스트 안에서 만드는 상황을 보자. 정리를 `afterEach`로 빼면, 만드는 곳(테스트 안)과 치우는 곳(바깥 `afterEach`)이 멀리 떨어져 읽기 어렵다. `onTestFinished`를 쓰면 **만드는 바로 그 자리에서 정리를 등록** 할 수 있어 코드가 한눈에 들어온다(응집도가 높아진다).

```ts
test('DB 조회', () => {
  const db = connectDb()
  onTestFinished(() => db.close()) // 만든 자리 바로 옆에서 정리 등록

  // ... db로 테스트 ...
})
```

정리하면, `onTestFinished`는 **"멀리 떨어진 `afterEach`를, 자원을 만든 그 자리의 정리로 바꿔주는"** 도구다. 세부 성질도 알아두자.

- 등록한 정리는 테스트가 끝난 뒤 **항상** 실행된다.
- 실행 시점은 `afterEach` 훅들 **이후** 다. (`afterEach`가 테스트 결과에 영향을 줄 수 있으므로 그 뒤)
- 콜백은 테스트 문맥 정보인 **TestContext** 를 인자로 받는다.

---

## 종합

`onTestFinished`는 테스트 본문 안에서 호출해 "테스트 종료 후 실행할 정리"를 등록하는 훅이며, 본문 밖에서 부르면 에러가 난다. 자원을 만든 바로 그 자리에서 정리를 등록해, 멀리 떨어진 `afterEach`보다 코드 응집도가 높다. 등록된 정리는 `afterEach` 이후에 항상 실행된다.

---

# onTestFinished는 어떤 점에서 유용한가?

## 도입

앞에서 "만든 자리에서 정리한다"는 응집도 이점을 봤다. `onTestFinished`에는 그 외에도 두 가지 실질적인 강점이 더 있다.

---

## 본문

**첫째, 재사용 로직에 좋다.** 자원을 만들어주는 헬퍼 함수가 있을 때, 헬퍼가 **자기 정리를 스스로 등록** 하게 만들 수 있다. 그러면 이 헬퍼를 쓰는 쪽은 뒷정리를 신경 쓸 필요가 없다.

```ts
function getTestDb() {
  const db = connectMockedDb()
  onTestFinished(() => db.close()) // 헬퍼가 스스로 정리 예약
  return db
}

test('...', () => {
  const db = getTestDb() // 부르기만 하면 정리는 알아서 됨
  // ...
})
```

**둘째, 도중에 검증(assertion)이 실패해도 정리가 항상 실행된다.** 이게 왜 중요한지 원리로 보자. 검증 함수(`expect(...)` 같은 것)가 실패하면 **에러를 던지고(throw), 그 줄에서 실행이 멈춘다.** 그래서 정리 코드를 테스트 **맨 끝 줄** 에 써두면, 중간 검증이 실패한 순간 거기까지 도달하지 못해 **정리가 건너뛰어진다.**

```ts
test('맨 끝 정리 - 위험', () => {
  const db = connectDb()
  expect(something).toBe(true) // 여기서 실패하면 아래로 못 감
  db.close() // 실행 안 됨! 연결이 안 닫힌 채 남음
})
```

반면 `onTestFinished`로 등록한 정리는 **테스트 앞부분에서 이미 예약** 돼 있으므로, 이후 검증이 실패하든 말든 Vitest가 테스트 종료 시 반드시 실행한다.

```ts
test('onTestFinished - 안전', () => {
  const db = connectDb()
  onTestFinished(() => db.close()) // 미리 예약됨
  expect(something).toBe(true) // 실패해도 위 정리는 실행됨
})
```

이 "검증 실패와 무관하게 정리 보장"이 맨 끝 정리 대신 `onTestFinished`를 택하는 핵심 이유다.

(참고: 위 예의 mock/spy 같은 가짜 자원도 마찬가지다. **spy** 는 원래 함수를 감싸 "어떻게 호출됐는지 기록하는 가짜 껍데기"를 말한다.)

---

## 종합

`onTestFinished`는 두 가지로 유용하다. 첫째, 헬퍼가 자기 정리를 스스로 등록하게 해 호출자가 뒷정리를 신경 쓸 필요가 없다. 둘째, 중간 검증이 실패해 실행이 멈춰도 미리 예약된 정리는 반드시 실행된다 — 맨 끝 줄 정리가 건너뛰어지는 문제를 막아준다.
