# [UNVERIFIED] 프로젝트 관리 방식의 역사는 어떻게 흘러왔는가?

## 도입

코드베이스를 어떻게 구성할지는 팀 규모와 공통 코드 비중에 따라 최적해가 달라진다. 역사적으로 "모놀리식 → Polyrepo → Monorepo"의 순서로 변화해왔고, 각 단계는 직전 단계의 한계를 해소하려는 시도였다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**모놀리식(Monolithic).** 모든 코드가 하나의 코드베이스에 있다. 초기 개발은 단순하지만, 모듈화가 없어서 어느 코드가 어느 부분에 영향을 주는지 파악하기 어렵다. 변경의 영향 범위가 넓고 예측이 힘들다.

**Polyrepo(Multi-repo).** 도메인이나 서비스마다 별도 git repository를 만든다. 각 팀이 독립적으로 릴리스·배포할 수 있고, 소유권이 명확해진다. 단점은 공통 코드(유틸, 타입, 디자인 시스템)를 공유하기 위해 매번 publish → install 사이클을 거쳐야 한다는 것이다.

**Monorepo.** 여러 패키지를 하나의 repository에 모은다. 공통 코드는 install 없이 바로 참조할 수 있고, 한 PR로 여러 패키지를 동시에 수정·검증할 수 있다. 단점은 빌드·배포 도구 체인이 복잡해진다는 것이다.

```
역사 흐름
모놀리식                    Polyrepo                    Monorepo
한 repo, 모든 코드          서비스마다 별도 repo          한 repo, 여러 패키지
├─ 단순한 초기 개발           ├─ 팀별 독립 릴리스            ├─ 공통 코드 공유 쉬움
└─ 영향 범위 예측 어려움       └─ 공통 코드 공유 번거로움      └─ 빌드 도구 복잡도 증가
         문제 해소 →                    문제 해소 →
```

---

## 종합

세 방식은 "더 나쁜 것 → 더 좋은 것"의 단순 발전이 아니라, 각 조직의 상황(팀 규모, 공통 코드 비중, 독립 배포 필요성)에 따라 최적해가 다르다. Monorepo가 최신 트렌드지만, 공통 코드가 적고 팀이 완전히 독립적이라면 Polyrepo가 여전히 좋은 선택일 수 있다.

# [UNVERIFIED] 모노레포가 유리한 경우와 폴리레포가 유리한 경우는?

## 도입

"왜 모노레포를 쓰는가"를 결정할 때 핵심 변수는 공통 코드의 비중이다. 공통 코드가 많을수록 모노레포가 유리하고, 적을수록 폴리레포의 독립성이 더 큰 가치를 발휘한다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**모노레포가 유리한 경우 (공통 코드 비중 높음).**
- 여러 앱이 디자인 시스템, 공통 유틸, 도메인 모델을 공유하는 상황.
- 공통 코드 비중이 70~100%라면, 공통 코드 변경 시 모든 앱에 즉시 반영해야 하는 경우가 많다.
- Polyrepo에서는 공통 코드를 수정할 때마다 publish → 각 앱에서 install → PR 머지 사이클을 반복해야 한다.

**폴리레포가 유리한 경우 (공통 코드 비중 낮음).**
- 각 서비스가 독립적이고 공통 코드 비중이 10~30%인 상황.
- 팀별 독립 릴리스 사이클, 권한 분리, 배포 파이프라인 단순함이 더 중요한 가치.

```
의사결정 기준: 공통 코드 비중
공통 코드 비중 70%+                공통 코드 비중 30%-
      ↓                                   ↓
    Monorepo                          Polyrepo
공통 코드 즉시 반영 가능             팀별 독립 릴리스 용이
한 PR로 여러 패키지 동시 수정        빌드 복잡도 낮음
```

---

## 종합

모노레포를 선택할 때 "요즘 다들 쓰니까"가 아니라 "공통 코드 비중이 높아서"가 근거여야 한다. 공통 코드 없이 모노레포를 쓰면 Turborepo 같은 도구 설정의 복잡도만 늘어나고 이점은 없다. 반대로 공통 코드가 많은데 폴리레포를 쓰면 publish/install 사이클 반복으로 개발 생산성이 크게 떨어진다.

# [UNVERIFIED] 코드 공유와 수정 측면에서 모노레포와 폴리레포는 어떻게 다른가?

## 도입

공통 코드를 수정하는 시나리오에서 두 방식의 차이가 가장 극명하게 드러난다. 폴리레포는 변경이 여러 단계를 거쳐야 전파되고, 모노레포는 한 PR에서 끝난다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**Monorepo에서의 코드 공유.** 같은 repo 안의 패키지를 workspaces가 심볼릭링크로 연결해준다. `import { Button } from '@myorg/ui'`는 npm에서 설치한 게 아니라 `packages/ui/src`를 직접 참조한다. publish·install 없이 수정이 즉시 반영된다.

**Polyrepo에서의 코드 공유.** 다른 패키지를 쓰려면 npm 레지스트리에 publish되어 있어야 한다. 공통 코드를 수정하면 다음 단계를 거쳐야 한다.

```
Polyrepo에서 공통 코드 변경 시
공통 패키지 수정
  → 버전 올리기 (semver)
  → npm publish
  → 각 앱에서 install (package.json 업데이트)
  → 앱마다 별도 PR 생성·머지
```

작은 변경 하나가 여러 repo에 걸쳐 4~5단계의 사이클을 만든다.

**Monorepo에서의 동일 시나리오.**

```
Monorepo에서 공통 코드 변경 시
공통 패키지 수정
  → 동일 PR에서 영향받는 앱도 함께 수정·검증
  → PR 1개로 모든 변경 완료
```

---

## 종합

모노레포의 핵심 가치는 "공통 코드 변경과 사용처 검증을 한 PR에서"다. 폴리레포에서 공통 유틸 함수 하나를 바꾸면 N개 앱 각각에서 install·테스트·PR 사이클을 반복해야 하는 것과 대비된다. 이 사이클 비용이 높다고 느끼는 팀이 모노레포로 전환하는 주된 동인이다.
