# [UNVERIFIED] 번들러는 왜 등장했는가?

## 도입

번들러 등장의 동인은 세 가지다. 브라우저의 모듈 시스템 미지원, HTTP/1.x의 동시 요청 한계, 빌드 자동화·최적화 요구. 세 동인은 서로 독립적이지 않고, 각각 "파일을 하나로 합쳐야 한다"는 같은 방향을 가리킨다. 각각의 세부는 아래 질문에서 이어진다.

---

## 본문

세 동인 모두 User Answer 기반이므로 인용 없이 해설한다.

**브라우저의 모듈 시스템 미지원.** ES6(ES2015)에서 `import/export` 문법이 표준화됐지만, 한동안 브라우저는 ESM을 직접 실행하지 못했다. 개발자가 모듈 단위로 코드를 작성하면 브라우저가 이해하는 형태로 변환·병합하는 도구가 필요했다. 그 역할이 번들러다.

**HTTP/1.x의 동시 요청 한계.** 브라우저-호스트당 동시 연결 수는 대략 6개다. 모듈이 수백 개면 요청이 직렬화되어 초기 로딩이 느려진다. 여러 모듈을 소수의 파일로 합치면 요청 수가 줄어든다.

**빌드 자동화·최적화 요구.** 번들러는 단순 파일 합치기를 넘어, Tree Shaking, Minification, CSS·이미지·TypeScript 처리까지 플러그인 시스템으로 통합하는 자동화 플랫폼이 됐다.

```
세 동인
├── 브라우저 ESM 미지원    → import/export를 브라우저 이해 형태로 변환
├── HTTP/1.x 요청 한계    → 모듈 N개를 파일 수십 개로 합쳐 요청 수 절감
└── 빌드 자동화 요구       → Tree Shaking·Minification·비-JS 자원 처리 통합
```

---

## 종합

번들러는 "브라우저가 이해하지 못하는 모듈 시스템"과 "HTTP/1.x의 요청 수 제약"이라는 두 현실적 제약에서 출발했다. 거기에 코드 최적화·자동화 요구가 더해지면서 번들러는 단순 변환 도구에서 빌드 파이프라인 전체를 아우르는 플랫폼으로 진화했다. 번들러 없이 수백 개의 `import` 파일을 그대로 서빙하면 초기 로딩이 느리고, 최신 TS 문법이나 JSX를 브라우저가 직접 실행하지 못한다. 현재의 React·Vite 개발 환경이 당연하게 느껴지는 이유는 번들러가 이 복잡도를 모두 처리해주기 때문이다.

# [UNVERIFIED] 브라우저의 모듈 시스템 미지원이 번들러를 어떻게 요구했는가?

## 도입

ES6 표준이 `import/export` 문법을 추가했지만, 브라우저 지원이 뒤따르기까지 시간이 걸렸다. 그 공백 기간 동안 개발자는 ESM으로 코드를 작성하고 싶었지만, 브라우저는 CommonJS·AMD·IIFE 같은 형태만 실행할 수 있었다. 번들러가 그 간극을 메우는 역할을 맡았다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**ESM과 브라우저의 간극.** `import React from 'react'` 같은 ESM 문법을 브라우저가 이해하지 못하면, 개발자는 모듈 단위 개발을 포기하거나 변환 단계를 거쳐야 한다. 번들러가 소스의 `import`를 분석해 의존성 그래프를 만들고, 그래프를 브라우저가 실행할 수 있는 형태로 평탄화한다.

**모듈 그래프 평탄화.** 수십~수백 개의 ESM 파일을 하나의(또는 소수의) JS 파일로 합쳐 `<script src="bundle.js">`로 넣으면, 브라우저는 모듈 시스템을 전혀 몰라도 실행한다.

```
개발자 소스 (ESM)          →  번들러  →  브라우저 실행 파일
src/index.ts                           bundle.js (IIFE / CJS)
  └── import Button from './Button'
  └── import utils from './utils'
```

---

## 종합

모듈 시스템 미지원 문제는 번들러 등장의 가장 근본적인 동인이다. 브라우저가 ESM을 완전히 지원하는 지금도 번들러가 남아 있는 이유는, 변환이 필요 없더라도 최적화(Tree Shaking, 코드 분할)와 자동화(TypeScript 컴파일, CSS 처리) 역할이 번들러에 남아 있기 때문이다.

# [UNVERIFIED] HTTP/1.x의 한계는 번들러를 어떻게 요구했는가?

## 도입

브라우저는 하나의 호스트에 동시에 열 수 있는 TCP 연결 수를 약 6개로 제한한다. 모듈을 파일마다 개별 요청으로 받으면 이 한계에 금방 부딪힌다. 번들러가 파일을 합쳐 요청 수를 줄이는 것이 네트워크 측면에서 번들링의 핵심 이유다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**동시 요청 수 제한.** 브라우저가 호스트당 6개의 연결만 허용하면, 모듈 100개는 약 17번의 라운드트립으로 나뉘어 순차 전송된다. 사용자 입장에서는 초기 화면이 뜨는 데 오래 걸린다.

**3-way handshake 비용.** HTTP/1.x는 요청마다 TCP 연결을 새로 수립한다. 모듈이 많을수록 handshake 오버헤드가 누적된다.

**번들링의 해결 방식.** 수백 개의 모듈을 3~5개의 번들 파일로 합치면 요청 수가 95% 이상 줄어든다. 같은 네트워크 조건에서 초기 로딩이 극적으로 빨라진다.

```
번들링 전 (HTTP/1.x)              번들링 후
모듈 100개 → 요청 100개            bundle.js 1개 → 요청 1개
호스트당 6개 동시 → ~17 라운드트립    단일 요청 → 1 라운드트립
```

---

## 종합

HTTP/1.x 제약은 "모듈 수 = 요청 수"를 직접 비용으로 만들었다. 번들러가 파일을 합치는 것만으로 초기 로딩 속도가 크게 개선되는 이유다. HTTP/2·3이 이 제약을 완화했지만, 번들링은 그 이외의 이유(Tree Shaking, 코드 분할)로 여전히 유효하다. 다음 질문에서 HTTP/2·3 환경에서 번들링이 여전히 유효한 이유를 다룬다.

# [UNVERIFIED] HTTP/2·3에서도 번들링이 여전히 유효한가?

## 도입

HTTP/2·3은 멀티플렉싱으로 동시 요청 한계를 상당 부분 해소했다. "그럼 번들링이 필요 없어지지 않았나?"는 합리적인 의문이다. 결론은 여전히 유효하다는 것이고, 이유는 HTTP 효율화와 별개의 최적화 작업이 번들링 단계에서 이뤄지기 때문이다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**멀티플렉싱이 해소하는 것과 해소 못하는 것.** HTTP/2·3의 멀티플렉싱은 TCP 연결 수 제한을 우회해 다수의 요청을 병렬 처리한다. 그러나 "브라우저가 파싱·처리해야 하는 모듈 수"는 줄어들지 않는다. 모듈이 1000개면 브라우저는 1000개의 파일을 파싱하고 실행해야 한다.

**번들링이 여전히 주는 것들.**
- Tree Shaking: 사용하지 않는 코드를 번들 단계에서 제거한다. 파일을 그대로 서빙하면 불필요한 코드가 포함된 채 전달된다.
- Minification: 변수명 단축, 공백 제거로 파일 크기를 줄인다.
- Chunk Splitting: 자주 바뀌지 않는 vendor 코드와 앱 코드를 분리해 캐시 효율을 높인다.

---

## 종합

HTTP/2·3은 번들러 등장 동인 중 하나인 "요청 수 제한"을 해소하지만, 번들러의 다른 역할(코드 최적화·분할·캐싱 전략)은 HTTP 버전과 무관하게 남는다. Vite도 production 빌드에서 Rollup 번들링을 유지하는 이유가 여기에 있다. "HTTP/2 쓰면 번들링 필요 없다"는 오개념이 흔하니 주의가 필요하다.

# [UNVERIFIED] 최적화·빌드 자동화는 번들러를 어떻게 요구했는가?

## 도입

번들러는 모듈을 합치는 도구로 시작했지만, 현재는 빌드 파이프라인 전체를 통합하는 자동화 플랫폼이다. Tree Shaking부터 TypeScript 컴파일, 이미지 최적화까지 번들러의 플러그인 시스템으로 하나의 명령에서 처리된다.

---

## 본문

User Answer 기반이므로 인용 없이 해설한다.

**코드 최적화.** Tree Shaking은 정적 분석으로 사용되지 않는 `export`를 제거한다. Minification은 변수명을 단축하고 공백을 없애 번들 크기를 줄인다. 이 두 작업 모두 전체 모듈 그래프를 본 뒤에야 가능하므로 번들러 단계에서 처리된다.

**비-JS 자원 처리.** TypeScript(`.ts`), CSS 모듈(`.module.css`), Sass(`.scss`), 이미지(`.png`) 등은 브라우저가 직접 실행하지 못한다. 번들러는 로더·플러그인을 통해 이들을 브라우저가 다룰 수 있는 형태로 변환한다.

```
빌드 파이프라인
소스                     번들러 처리                  출력
.ts  ─────────────────→  esbuild/tsc 변환   ─────→
.module.css ──────────→  CSS 모듈 변환       ─────→  bundle.js
.png  ────────────────→  Asset hash 처리    ─────→  assets/
index.tsx ────────────→  Tree Shaking + Minify ──→
```

---

## 종합

번들러가 "파일 합치기 도구"를 넘어 빌드 자동화 플랫폼이 된 이유는, 현대 프론트엔드가 TypeScript·CSS Modules·이미지 등 브라우저가 직접 이해하지 못하는 자원들을 광범위하게 사용하기 때문이다. 번들러 없이 이 모든 변환을 수동으로 처리하는 것은 현실적으로 불가능하다. `vite build` 한 줄로 TypeScript 컴파일, Tree Shaking, 이미지 최적화가 모두 돌아가는 것이 번들러의 자동화 가치다.
