# 문자열(String)이란 무엇인가?

## 도입

문자열은 가장 자주 다루는 자료구조 중 하나다. CS 관점에서 문자열은 배열과 닮은 구조를 가진다 — 문자들이 순서대로 나열된 시퀀스다.

---

## 본문

> A string is a sequence of characters.

"문자열은 문자들의 시퀀스(순서 있는 나열)다."

- **sequence**: 순서가 있고 인덱스로 특정 위치에 접근할 수 있다. `"hello"[1]`은 `"e"`다. 배열처럼 연속 메모리에 저장된다.
- **characters**: 문자 하나하나. JS에서는 내부적으로 UTF-16으로 인코딩된 코드 유닛이다.

```js
const str = "hello";
str[0]  // "h"
str[4]  // "o"
str.length  // 5
```

---

## 종합

문자열이 "시퀀스"라는 정의에서 배열과의 유사성이 나온다. 실제로 많은 언어에서 문자열은 문자 배열의 형태로 구현된다. JS에서 문자열 인덱싱(`str[i]`)이 O(1)인 이유도 연속 메모리에 저장되어 있기 때문이다 — 배열의 랜덤 접근과 같은 원리다. 단, 문자열은 대부분의 언어에서 배열과 달리 불변(immutable)이라는 중요한 차이가 있다.

---

---

# 문자열이 불변(Immutable)이라는 것은 무엇을 의미하는가?

## 도입

JS에서 `let name = "Hello"`라고 쓰고 `name[0] = "J"`를 시도하면 에러 없이 조용히 무시된다. 문자열이 불변이기 때문이다 — 한 번 만들어진 문자열의 내용은 바꿀 수 없다.

---

## 본문

> Strings are typically immutable in most of the programming languages like Java, Python and JavaScript. In Python, Java and JavaScript, strings are immutable.

"문자열은 Java, Python, JavaScript를 포함한 대부분의 프로그래밍 언어에서 일반적으로 불변이다."

- **immutable**: 생성 후 내용을 변경할 수 없다. 변경하려면 새 문자열을 만들어야 한다.

User Annotation이 짚듯:
```js
let name = "Hello";
name = "Another";
// "Hello"가 저장된 메모리를 직접 수정하는 게 아님
// "Another"를 담는 새 메모리 공간을 만들고 name이 그쪽을 가리킨다

let arr = [1, 2, 3, 4, 5];
arr = [5, 4, 3, 2, 1];
// 배열 재할당과 동일한 패턴
```

문자열에 특정 문자를 삽입·삭제하는 것도 기존 문자열을 수정하는 게 아니라 새 문자열을 생성한다.

```js
const str = "hello";
const newStr = str.slice(0, 2) + "X" + str.slice(3);
// "heXlo" — str은 여전히 "hello", 새 문자열이 만들어짐
```

---

## 종합

불변성 때문에 문자열을 반복 수정하면 매번 새 문자열이 생성돼 메모리와 시간 비용이 누적된다. 예를 들어 for 루프 안에서 `result += str`을 n번 반복하면 이론적으로 O(n²) 복사가 발생한다 (매번 새 문자열 생성). JS V8 엔진은 내부 최적화로 이를 개선하지만, 대량 문자열 조합이 필요하다면 배열에 담아 마지막에 `.join("")`을 쓰는 패턴이 권장된다.

# [UNVERIFIED] 문자열이 불변인 것이 왜 중요한가?

## 도입

단순히 "못 바꾼다"는 제약처럼 보이지만, 불변성은 세 가지 핵심 이점을 가져다준다 — 안전성, 메모리 효율, 해시 자료구조 신뢰성.

---

## 본문

User Answer를 기반으로 세 가지를 해설한다:

**1. 안전성 (Thread-Safety)**

여러 스레드가 같은 문자열 데이터를 동시에 읽어도, 불변이라 수정이 불가능하므로 데이터가 꼬일(race condition) 걱정이 없다. Node.js는 싱글 스레드 이벤트 루프지만, Worker Thread를 쓰는 경우에도 문자열 공유가 안전하다.

**2. 효율성 (String Pool / String Interning)**

"hello"라는 문자열을 코드 100군데서 사용해도, 실제 메모리에는 "hello"를 딱 하나만 저장하고 100개의 변수가 그 주소를 공유할 수 있다. 불변이니까 공유해도 한 곳에서 수정할 걱정이 없다.

```js
const a = "hello";
const b = "hello";
// a와 b가 같은 메모리 주소를 가리킬 수 있음 (엔진 최적화)
// 불변이라 둘 중 하나가 수정해도 다른 쪽에 영향 없음
```

**3. 해시 자료구조 (Hash Key로 안전하게 사용)**

`Map`이나 `Set`의 키(Key) 값으로 문자열을 안심하고 쓸 수 있다. 해시값은 키의 내용을 기반으로 계산되는데, 키가 변할 수 없으니 해시값도 변하지 않는다 — 저장 후 키가 변해서 해시가 달라지는 사태가 없다.

```js
const map = new Map();
const key = "user123";
map.set(key, { name: "Alice" });
// key가 불변이므로 map.get("user123")이 항상 같은 버킷을 찾는다
```

---

## 종합

문자열의 불변성은 단순한 설계 결정이 아니라, 성능·안전성·신뢰성을 동시에 얻기 위한 트레이드오프다. 수정 비용(새 문자열 생성)을 지불하는 대신 공유·캐싱·해시 키 사용에서 이점을 얻는다. React에서 props로 문자열을 내려줄 때 자식 컴포넌트가 임의로 수정할 수 없다는 보장도 불변성에서 나온다.

# [UNVERIFIED] 문자열은 메모리에 어떻게 저장되는가?

## 도입

문자열이 `str[i]`로 O(1)에 특정 문자에 접근할 수 있는 이유는 메모리 저장 방식과 직결된다.

---

## 본문

Python, Java, JavaScript에서 문자열의 문자들은 배열처럼 **연속된(contiguous)** 메모리 위치에 저장된다.

```
"hello" 메모리 표현:
주소:  1000  1002  1004  1006  1008
       'h'   'e'   'l'   'l'   'o'
       (JS는 UTF-16이라 문자당 2바이트)
```

연속 저장의 결과:
- `str[3]` 접근 → `base + 3 × 2 = 1006번지` → O(1)
- 배열과 동일한 랜덤 접근이 가능하다.

Linked List 방식이었다면 `str[3]`을 얻으려면 head부터 3번 포인터를 따라가야 해서 O(n)이 된다 — 연속 저장이기 때문에 O(1)이다.

---

## 종합

문자열의 연속 메모리 저장은 두 가지를 동시에 가능하게 한다 — O(1) 인덱스 접근과 CPU 캐시 친화성. 알고리즘에서 문자열을 배열처럼 인덱스로 접근하는 코드가 자연스럽게 O(1)인 이유다. 단, 불변이라 배열처럼 특정 위치를 직접 수정할 수는 없다는 점을 함께 기억한다.

# 문자열의 길이를 구하는 시간 복잡도는?

## 도입

`str.length`가 O(1)인 언어도 있고 O(n)인 언어도 있다. 차이는 "길이를 미리 저장해두는가"에 달려 있다.

---

## 본문

> The length of a string refers to the total number of characters present in it, including letters, digits, spaces, and special characters.

"문자열의 길이는 문자, 숫자, 공백, 특수문자를 포함한 총 문자 수를 뜻한다."

User Annotation이 짚은 언어별 차이:

**C언어 `strlen()`**: 문자열 끝을 알리는 `\0`(null terminator)를 만날 때까지 문자를 하나씩 세기 때문에 O(n).

**JavaScript `.length`**: 문자열 객체 내부에 길이를 미리 저장해두므로 O(1). 생성 시 한 번 계산하고, 이후 접근은 저장된 값을 반환한다.

```js
const str = "hello";
str.length  // 5 → O(1) (저장된 값 반환)

// C:
// char str[] = "hello";
// strlen(str)  → h→e→l→l→o→\0 순회 → O(n)
```

---

## 종합

`.length`가 O(1)인 JS에서는 for 루프 조건에 `str.length`를 매번 써도 성능 걱정이 없다. 문자열 처리 알고리즘에서 길이를 반복해서 참조하는 경우 변수에 캐시할 필요도 없다. 단, JS에서 이모지처럼 서로게이트 쌍(surrogate pair)으로 표현되는 문자는 `.length`가 문자 수가 아니라 코드 유닛 수를 반환한다는 점은 주의가 필요하다 ("😀".length === 2).

# [UNVERIFIED] 두 문자열의 동일성 비교(equality check)의 시간 복잡도는?

## 도입

`"hello" === "hello"`처럼 두 문자열이 같은지 비교하는 것은 단순해 보이지만, 내부적으로는 두 문자열을 한 글자씩 비교해야 한다.

---

## 본문

두 문자열의 `===` 비교는 O(n)이다. 문자를 하나하나 비교해야 하기 때문이다.

```
"hello" === "hello":
  h === h → e === e → l === l → l === l → o === o → true
  → 5번 비교 → O(n)

"hello" === "world":
  h === w → false → 즉시 종료
  → 1번 비교 → O(1) 조기 종료 (최선)
```

**최선**: 첫 문자부터 다르면 즉시 false → O(1).
**최악**: 두 문자열이 같거나 마지막 문자에서 차이가 남 → O(n).

**길이 비교 최적화**: 두 문자열의 길이가 다르면 즉시 false → O(1). 대부분의 언어가 이 최적화를 먼저 수행한다.

```js
"hello".length !== "hi".length  // 5 !== 2 → 즉시 false
```

---

## 종합

문자열 비교가 O(n)이라는 사실은 Hash Table의 충돌 처리 비용에도 영향을 미친다. 해시 충돌 시 두 키가 실제로 같은지 문자열 비교를 해야 하는데, 이것이 O(n)이다. 알고리즘에서 대량 문자열 비교를 자주 해야 한다면 Rabin-Karp 같은 해시 기반 문자열 매칭 알고리즘을 사용해 비교 횟수를 줄이는 것을 고려할 수 있다.
