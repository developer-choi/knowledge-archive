# [UNVERIFIED] 서버 세션 기반 인증은 REST의 stateless 제약에 부합하는가?

## 도입

REST의 6가지 제약 중 하나가 "stateless" — 서버가 클라이언트 상태를 저장하지 않아야 한다는 것이다. 서버 세션 기반 인증은 이 제약과 충돌한다. 부합하지 않는다가 정답이다.

---

## 본문

**서버 세션 기반 인증의 구조**

```
클라이언트                    서버
─────────────────────────────────────────────────
1. POST /login {id, pw}  →
                         ← 200 OK + Set-Cookie: sessionId=abc123
2. GET /profile
   Cookie: sessionId=abc123  →
                         ← (세션 저장소에서 abc123 조회)
                         ← 200 OK + 프로필 데이터
```

서버는 `sessionId=abc123`과 매핑된 사용자 정보(userId, 권한 등)를 **메모리 또는 Redis 같은 세션 저장소에 보관**한다. `GET /profile` 요청이 들어올 때, 서버는 Cookie의 세션 ID를 가지고 저장소를 조회해야 비로소 "이 요청이 누구의 것인지" 알 수 있다.

**왜 REST stateless 위반인가?**

REST의 stateless 제약은 "각 요청은 그 자체만으로 처리 가능해야 한다"는 것이다. 즉, 서버가 이전 요청을 기억하지 않아도 현재 요청 하나만 보고 처리할 수 있어야 한다.

세션 기반 인증에서 `GET /profile` 요청은 "혼자서" 처리될 수 없다. 서버 세션 저장소에 `abc123 → userId: 1`이라는 상태가 보관되어 있어야 처리된다. 이 상태는 **서버 쪽에 존재하는 클라이언트 상태** — 정확히 REST가 금지하는 것이다.

```
stateless 위반 구조
─────────────────────────────────────────────────
요청: GET /profile
      Cookie: sessionId=abc123

서버가 해야 하는 일:
  1. abc123을 세션 저장소에서 조회
  2. 저장소에 userId: 1 있음 → 이 사람이 Alice
  3. Alice의 프로필 반환

  ↑ 서버 세션 저장소(상태)에 의존 = stateless 위반
```

**추가 문제: 수평 확장**

세션을 서버 메모리에 저장하면 스케일 아웃 시 문제가 생긴다. 요청이 서버 A로 갔다가 다음 요청은 서버 B로 가면, 서버 B에는 세션이 없다. 이를 해결하려면 Redis 같은 공유 세션 저장소를 두어야 한다 — 이 순간 인프라 의존성이 추가된다.

---

## 종합

서버 세션 기반 인증은 REST의 stateless 제약에 부합하지 않는다. 서버가 클라이언트의 인증 상태를 세션 저장소에 유지해야 하기 때문이다. 이것이 JWT 같은 토큰 기반 인증이 등장한 배경이다 — 인증 정보를 서버가 아닌 클라이언트(토큰 자체)에 담아 stateless를 달성하려는 시도다. 실무에서는 세션 기반을 쓰더라도 Redis로 세션 저장소를 공유하면 수평 확장 문제는 해결할 수 있지만, REST stateless 위반이라는 사실은 바뀌지 않는다.

# [UNVERIFIED] JWT가 stateless 인증으로 분류되는 이유는?

## 도입

JWT(JSON Web Token)가 stateless 인증으로 불리는 이유는 인증에 필요한 모든 정보가 토큰 자체에 담겨 있어서, 서버가 별도 저장소를 조회하지 않아도 요청을 검증할 수 있기 때문이다.

---

## 본문

**JWT 구조**

JWT는 세 부분이 점(.)으로 연결된 문자열이다.

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsInJvbGUiOiJ1c2VyIn0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
─────────────────── ─────────────────────────────────── ─────────────────────────────────────
     Header                    Payload                         Signature
```

```js
// 각 부분은 Base64URL 인코딩
const [header, payload, signature] = token.split('.')

// Payload 디코드
const data = JSON.parse(atob(payload))
// → { userId: 1, role: "user", exp: 1716000000 }
```

- **Header**: 서명 알고리즘 명시 (`{ "alg": "HS256" }`)
- **Payload**: 실제 데이터 — userId, 권한, 만료 시간(exp). 누구나 읽을 수 있지만 수정하면 서명이 깨진다.
- **Signature**: `HMAC_SHA256(header + "." + payload, secretKey)`. 서버만 아는 비밀 키로 만든 서명이다.

**왜 stateless인가?**

```
세션 기반 (stateful)
  요청: Authorization: Bearer {sessionId}
  서버: DB/Redis에서 sessionId 조회 → userId 확인

JWT 기반 (stateless)
  요청: Authorization: Bearer {jwt}
  서버: Signature 검증(비밀키로 재계산) → 일치하면 Payload 신뢰
  ↑ 저장소 조회 없음 — 토큰 자체가 신뢰의 근거
```

서버는 JWT를 받으면 비밀 키로 Signature를 재계산해서 토큰과 일치하는지 확인한다. 일치하면 Payload가 변조되지 않았다는 뜻이므로, Payload 안의 `userId`·`role`을 신뢰할 수 있다. 세션 저장소 조회가 전혀 필요 없다.

비유하면: 공증된 신분증을 제시하는 것과 같다. 경찰관은 신분증의 홀로그램(서명)만 확인하면 되고, 발급 기관 서버에 "이 신분증 진짜야?" 물어볼 필요가 없다.

---

## 종합

JWT의 stateless 특성은 서버 확장에 직접적인 이점을 준다. 서버 A에서 발급한 JWT를 서버 B, C, D 어디에서든 비밀 키만 공유되어 있으면 검증할 수 있다. 세션 저장소 없이도 수십 대 서버가 동일하게 동작한다. 이것이 마이크로서비스와 서버리스 환경에서 JWT가 선호되는 이유다. 단, stateless의 이점은 다음 질문에서 다룰 한계를 동반한다 — 토큰을 한 번 발급하면 만료 전에 무효화하기 어렵다는 것이다.

# [UNVERIFIED] JWT의 한계(폐기 어려움, 페이로드 노출, 사이즈)는 각각 무엇인가?

## 도입

JWT의 stateless 특성은 확장성의 장점이지만, 동시에 구조적 한계를 내포한다. "서버가 상태를 갖지 않는다"는 것이 결국 "서버가 제어권을 일부 포기한다"는 의미이기 때문이다. 세 가지 한계를 각각 살펴본다.

---

## 본문

**한계 1: 폐기 어려움 (Revocation Problem)**

JWT는 만료 시간(exp)이 되기 전까지 서버 입장에서 유효하다. 사용자가 로그아웃하거나, 권한이 바뀌거나, 계정이 탈취됐어도 토큰이 살아있는 동안은 서버가 막을 방법이 없다.

```js
// 사용자가 로그아웃 → 클라이언트에서 토큰 삭제
// 하지만 토큰 자체는 여전히 유효
// 누군가 그 토큰을 가로챘다면 exp 전까지 사용 가능

// 해결책: 블랙리스트 DB에 폐기 토큰 저장
// → 결국 저장소 조회가 생겨 stateless 이점 훼손
```

이를 해결하려면 "폐기 목록(블랙리스트)"을 서버 측에 유지해야 하는데, 이 순간 stateless의 장점이 희석된다.

실무적 대응: 액세스 토큰의 만료 시간을 짧게(15분~1시간) 설정하고, 리프레시 토큰으로 재발급하는 방식을 쓴다. 위험한 토큰은 블랙리스트에 등록하지만, 짧은 만료 덕분에 블랙리스트 크기가 관리 가능하다.

**한계 2: 페이로드 노출**

Payload는 Base64URL 인코딩이지 암호화가 아니다. 누구나 디코딩해서 내용을 읽을 수 있다.

```js
const payload = 'eyJ1c2VySWQiOjEsInJvbGUiOiJ1c2VyIn0'
console.log(atob(payload))
// → {"userId":1,"role":"user"}  ← 그냥 보임
```

Signature가 변조를 막을 뿐, 내용의 기밀성은 보장하지 않는다. 따라서 Payload에 비밀번호·카드번호 같은 민감 정보를 담으면 안 된다. 기밀이 필요하면 JWE(JSON Web Encryption)를 써야 하는데 이는 추가 복잡도다.

**한계 3: 사이즈**

세션 ID는 `abc123` 같은 짧은 문자열이지만, JWT는 Payload 크기에 비례해서 커진다.

```
세션 쿠키:  sessionId=abc123
JWT:        eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySWQiOjEsInJvbGUiOiJ1c2VyIiwiZXhwIjoxNjk...
            (보통 200~2000 bytes)
```

매 HTTP 요청마다 Authorization 헤더에 JWT가 실려 나간다. 권한 정보나 커스텀 클레임이 많을수록 토큰이 커지고, 이것이 대역폭 비용으로 이어진다. 특히 REST API를 자주 호출하는 모바일 앱에서 누적 영향이 있다.

```
JWT 한계 요약
─────────────────────────────────────────────────────────
한계              원인                   실무 대응
─────────────────────────────────────────────────────────
폐기 어려움       서버 상태 없음          짧은 만료 + 블랙리스트
페이로드 노출     Base64 인코딩(암호화 X) 민감 정보 제외, JWE
사이즈 증가       Payload가 토큰에 포함   Payload 최소화
─────────────────────────────────────────────────────────
```

---

## 종합

JWT의 세 가지 한계는 모두 "stateless를 위해 상태를 토큰에 옮겼다"는 근본 결정에서 비롯된다. 상태를 서버 대신 토큰에 넣었기 때문에 서버가 통제권을 잃고(폐기 어려움), 네트워크에 내용이 노출되며(페이로드 노출), 토큰이 커진다(사이즈). 이것이 "JWT가 모든 상황의 정답이 아니다"라는 업계 인식이 생긴 이유다. 보안 민감도가 높거나 즉각 권한 회수가 필요한 시스템(금융, 관리자 권한)에서는 세션 기반 인증이 여전히 유효하다. JWT는 확장성이 중요하고 만료 주기가 짧게 운영 가능한 환경에서 강점을 발휘한다.
