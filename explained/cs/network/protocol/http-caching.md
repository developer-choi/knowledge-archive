# [UNVERIFIED] HTTP 캐싱의 동작 흐름은 어떻게 되는가?

## 도입

브라우저에서 `fetch()`를 호출할 때마다 매번 서버까지 왕복하면 느리고 낭비다. HTTP 캐싱은 이미 받은 리소스를 로컬(브라우저 캐시, CDN 등)에 저장해두고 재사용하는 메커니즘이다. DevTools Network 탭에서 "from cache"로 표시되는 요청이 바로 캐시 hit이다.

---

## 본문

HTTP 캐싱의 기본 흐름은 다음 세 가지 질문을 순서대로 묻는 것이다.

**1단계 — 저장된 캐시가 있는가?**

브라우저가 동일한 URL에 요청을 보낼 때, 먼저 캐시 스토리지에 해당 리소스가 있는지 확인한다. 없으면 서버로 요청을 보내고, 서버 응답을 캐시에 저장한다.

**2단계 — 캐시가 신선(fresh)한가?**

`Cache-Control: max-age=3600`처럼 만료 기간이 설정된 경우, 저장 시각으로부터 그 기간이 지나지 않았으면 캐시를 그대로 쓴다. 서버에 아무것도 묻지 않는다. DevTools에서 Status 코드 없이 `(memory cache)` 또는 `(disk cache)`로 표시된다.

**3단계 — 조건부 요청(재검증)**

캐시가 만료됐으면 서버에 "이 리소스, 아직 같은 거야?"를 물어본다. 이 질문이 조건부 요청(conditional request)이다. 서버가 "변경 없음"이라고 답하면 `304 Not Modified`를 보내고, 브라우저는 기존 캐시를 그대로 쓴다. 변경됐으면 `200 OK`와 함께 새 리소스를 보낸다.

```
브라우저 fetch(url)
    │
    ▼
캐시 있는가? ──No──► 서버 요청 → 200 + 리소스 → 캐시 저장
    │ Yes
    ▼
fresh한가? ──Yes──► 캐시 바로 사용 (요청 없음)
    │ No (만료)
    ▼
조건부 요청 (If-None-Match / If-Modified-Since)
    │
    ├── 304 Not Modified → 기존 캐시 사용 (본문 전송 없음)
    └── 200 OK → 새 리소스 수신 + 캐시 갱신
```

---

## 종합

HTTP 캐싱의 핵심 이득은 두 가지다. 첫째, fresh 상태에서는 네트워크 왕복 자체가 없으므로 응답 시간이 0에 가깝다. 둘째, 만료 후에도 조건부 요청으로 304를 받으면 본문 전송이 없어 대역폭을 아낀다. `Cache-Control` 헤더가 "얼마나 신선하게 유지할지"를, ETag/Last-Modified가 "변경됐는지 판별하는 지문"을 담당한다. Service Worker 캐싱은 이 HTTP 캐시 레이어 위에 프로그래매틱하게 올라탄 것으로, `fetch` 이벤트를 가로채 캐시 전략을 직접 제어한다.

# [UNVERIFIED] Cache-Control의 주요 디렉티브(max-age, no-cache, no-store, public/private)는 각각 무엇을 지시하는가?

## 도입

`Cache-Control`은 HTTP/1.1에서 도입된 헤더로, 응답(또는 요청)이 캐시되는 방식을 제어한다. 여러 디렉티브를 쉼표로 조합할 수 있으며, 이름이 직관적이지 않은 것들이 있어 혼동하기 쉽다. 특히 `no-cache`와 `no-store`는 이름만 보면 비슷해 보이지만 의미가 전혀 다르다.

---

## 본문

**max-age=N**

리소스를 N초 동안 신선(fresh)하게 유지한다. 이 기간 내에는 서버에 묻지 않고 캐시를 바로 사용한다. `Cache-Control: max-age=3600`이면 1시간 동안 캐시가 유효하다.

```js
// fetch 응답 헤더에서 확인
response.headers.get('Cache-Control') // "max-age=3600"
```

**no-cache**

"캐시하지 말라"가 아니다. "캐시해도 되지만, 쓰기 전에 반드시 서버에 재검증하라"는 뜻이다. 즉 `max-age=0`과 거의 동일하게 동작하며, 매 요청마다 조건부 요청을 보낸다. 서버가 `304`를 보내면 캐시를 그대로 쓴다.

**no-store**

진짜 "캐시하지 말라"다. 응답을 어디에도 저장하지 않는다. 로그인 정보, 결제 내역처럼 민감한 데이터에 사용한다. DevTools에서 같은 URL을 두 번 요청해도 항상 서버까지 간다.

**public**

"공유 캐시(CDN, 프록시)에도 캐시해도 된다"는 뜻이다. 일반적으로 인증이 필요 없는 정적 리소스(이미지, JS, CSS)에 적합하다.

**private**

"브라우저(개인 캐시)에만 캐시하고, CDN 같은 공유 캐시에는 저장하지 말라"는 뜻이다. 사용자별로 다른 내용을 담은 응답(마이페이지 HTML, 개인화된 API 응답 등)에 적합하다.

```
no-store   — 아무데도 저장 X
no-cache   — 저장은 O, 재검증 필수
max-age=0  — 즉시 만료 (no-cache와 유사)
max-age=N  — N초 fresh 유지
public     — CDN/프록시까지 캐시 허용
private    — 브라우저에만 캐시
```

---

## 종합

프론트엔드에서 가장 많이 만나는 패턴은 `Cache-Control: public, max-age=31536000, immutable`(빌드된 JS/CSS — 해시 파일명이 바뀌므로 1년 캐시)과 `Cache-Control: no-cache`(HTML index 파일 — 항상 최신 확인 필요)의 조합이다. `no-cache`와 `no-store`를 혼동하면 민감 데이터가 CDN에 캐시되거나, 반대로 정적 자산이 매번 서버를 거쳐 성능이 저하되는 문제가 발생한다.

# [UNVERIFIED] ETag와 Last-Modified의 역할 및 차이는?

## 도입

캐시가 만료된 후 조건부 요청을 보낼 때, 서버에게 "이 리소스가 변경됐는지"를 판단할 기준이 필요하다. 두 가지 방법이 있다. 하나는 마지막 수정 시각(Last-Modified)을 비교하는 방식이고, 다른 하나는 리소스의 고유 식별자(ETag)를 비교하는 방식이다.

---

## 본문

**Last-Modified**

서버 응답에 포함되는 날짜 값이다. 리소스가 마지막으로 수정된 시각을 나타낸다.

```
Last-Modified: Wed, 21 May 2025 07:28:00 GMT
```

클라이언트는 이 값을 저장해두었다가, 재검증 시 `If-Modified-Since` 헤더로 다시 보낸다. 서버는 수정 시각을 비교해 변경이 없으면 `304 Not Modified`를 반환한다.

단점: 초 단위 정밀도만 있어, 1초 안에 여러 번 수정된 파일은 변경을 감지하지 못할 수 있다. 또한 파일 내용이 같아도 수정 시각만 바뀌면 변경으로 처리된다.

**ETag (Entity Tag)**

리소스 내용의 "지문"이다. 파일 내용의 해시값, 버전 번호 등 서버가 결정한 불투명한 문자열이다.

```
ETag: "3f80f-1b6-3e1cb03b"
```

클라이언트는 이 값을 저장해두었다가, 재검증 시 `If-None-Match` 헤더로 보낸다. 서버는 현재 ETag와 비교해 같으면 `304`, 다르면 `200 + 새 리소스`를 반환한다.

ETag는 Last-Modified의 두 가지 단점을 모두 해결한다. 내용 기반이므로 "파일이 실제로 바뀌었는가"를 정확히 판단한다.

```
요청                          응답
GET /api/data              ← 200 OK
                            ETag: "abc123"
                            Last-Modified: Mon, 01 Jan 2025 00:00:00 GMT

(캐시 만료 후)
GET /api/data              ← 조건부 요청
If-None-Match: "abc123"
If-Modified-Since: Mon, ...
                            ← 304 Not Modified (캐시 그대로)
```

---

## 종합

ETag가 더 정밀하므로 서버가 두 헤더를 모두 보내면 클라이언트는 ETag(`If-None-Match`)를 우선 사용한다. Last-Modified는 ETag를 지원하지 않는 구형 서버와의 호환성을 위해 병행 전송된다. Express나 Next.js 같은 서버 프레임워크는 정적 파일에 ETag를 자동으로 생성해준다.

---

# [UNVERIFIED] 조건부 요청(If-None-Match, If-Modified-Since)이 304 응답을 만드는 흐름은?

## 도입

캐시가 만료된 후 브라우저가 "이 리소스 아직 같아?"라고 서버에 묻는 것이 조건부 요청이다. 이때 사용하는 헤더가 `If-None-Match`(ETag 기반)와 `If-Modified-Since`(날짜 기반)다. 서버가 "변경 없음"이라고 답하면 `304 Not Modified`를 보내고, 본문(body)은 전송하지 않는다.

---

## 본문

**1단계 — 첫 번째 요청 (캐시 미스)**

서버가 리소스를 보내면서 ETag와 Last-Modified를 응답 헤더에 포함한다. 브라우저는 이 값을 캐시와 함께 저장한다.

```
← 200 OK
ETag: "v2-abc123"
Last-Modified: Mon, 01 Jan 2025 00:00:00 GMT
Cache-Control: max-age=3600
[리소스 본문]
```

**2단계 — 캐시 만료 후 재검증 요청**

`max-age` 경과 후 같은 URL에 요청을 보낼 때, 저장해둔 ETag와 날짜를 조건부 헤더로 포함한다.

```
GET /api/data HTTP/1.1
If-None-Match: "v2-abc123"
If-Modified-Since: Mon, 01 Jan 2025 00:00:00 GMT
```

**3단계 — 서버 응답**

서버는 현재 ETag와 클라이언트가 보낸 값을 비교한다.

- 같다면 → `304 Not Modified` (본문 없음, 헤더만 전송)
- 다르다면 → `200 OK` + 새 리소스

```
← 304 Not Modified
Cache-Control: max-age=3600
(본문 없음 — 네트워크 비용 절약)
```

브라우저는 304를 받으면 기존 캐시 본문을 그대로 사용하고, 만료 시각을 갱신한다.

```
첫 요청                  서버 응답
GET /img.png     →      200 OK + ETag: "abc" + 본문
                                       ↓ (3600초 경과)
재검증 요청              서버 응답
GET /img.png     →      304 Not Modified (본문 없음)
If-None-Match:
  "abc"
                                       ↓ (변경된 경우)
재검증 요청              서버 응답
GET /img.png     →      200 OK + ETag: "xyz" + 새 본문
If-None-Match:
  "abc"
```

---

## 종합

304 응답의 핵심 가치는 "본문 없이 캐시를 갱신한다"는 것이다. 대용량 이미지나 JS 번들을 매번 다시 받지 않아도 최신 상태를 유지할 수 있다. DevTools Network 탭에서 상태 코드 `304`가 보이면 조건부 요청이 작동한 것이다. `fetch()` 기반 API 호출에서는 브라우저가 자동으로 조건부 헤더를 붙여주므로, 서버가 ETag를 설정해두기만 하면 캐싱이 자동으로 동작한다.

# [UNVERIFIED] POST는 왜 기본적으로 캐싱되지 않는가?

## 도입

GET 요청은 "데이터를 가져온다"는 의도가 명확하고 서버 상태를 바꾸지 않으므로 캐싱이 자연스럽다. 반면 POST는 "데이터를 처리한다"는 의도이며, 같은 요청을 두 번 보내면 두 번 처리가 일어날 수 있다. 이 차이가 캐싱 여부의 근본적인 이유다.

---

## 본문

**POST는 서버 상태를 변경하는 메서드다**

`POST /orders`로 주문을 생성하는 요청을 캐싱하면 어떻게 될까? 동일한 요청을 다시 보낼 때 캐시가 "이전에 받은 응답이 있다"며 주문 생성 요청 자체를 서버에 보내지 않을 수 있다. 이것은 사용자가 기대하는 동작이 아니다.

**멱등성(idempotency)과 캐싱의 관계**

HTTP 명세는 캐싱 가능한 메서드를 GET, HEAD, POST로 정의하지만, POST 캐싱은 서버가 명시적으로 `Cache-Control` 또는 `Expires` 헤더를 응답에 포함할 때만 허용한다. 브라우저와 CDN은 기본적으로 POST 응답을 캐싱하지 않는다.

**실무에서의 의미**

```js
// GET — 브라우저가 자동으로 캐싱
fetch('/api/products') // 두 번째 호출 시 캐시 사용 가능

// POST — 캐싱 없음, 매번 서버까지 요청
fetch('/api/orders', { method: 'POST', body: JSON.stringify(order) })
```

서버 측에서 POST로 검색(쿼리 파라미터가 너무 길어 GET URL에 담기 어려운 경우)을 구현하는 경우, POST 응답에 명시적으로 `Cache-Control: max-age=60`을 붙이면 캐싱이 가능하다. 하지만 이는 표준적인 사용 패턴이 아니다.

---

## 종합

POST가 캐싱되지 않는 이유는 "같은 POST를 두 번 보내면 두 번 처리가 발생할 수 있다"는 비멱등성에 있다. 캐시 레이어가 POST를 가로채면 사용자의 의도와 다른 동작이 발생할 수 있다. 데이터를 읽기만 하는 작업이면 GET으로 설계해야 브라우저 캐시, CDN 캐시, Service Worker 캐시를 자연스럽게 활용할 수 있다.
