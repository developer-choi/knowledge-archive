# [UNVERIFIED] Connectionless Communication(비연결형 통신)이란 무엇인가?

## 도입

TCP가 연결을 먼저 확립하고 데이터를 보내는 "연결형"이라면, UDP는 연결 없이 바로 데이터를 보내는 "비연결형"이다. 확인 절차를 생략하여 빠르지만, 신뢰성은 보장하지 않는다.

---

## 본문

Connectionless Communication(비연결형 통신)은 사전 연결 확립 없이 데이터를 전송하는 방식이다.

**특징:**

- **확인 절차 없음**: 데이터를 보내기 전에 상대가 준비됐는지 확인하지 않는다. 그냥 보낸다.
- **서로 다른 경로**: 같은 목적지로 보내는 패킷들이 서로 다른 경로를 통해 전달될 수 있다. TCP와 달리 모든 패킷이 같은 경로(가상회선)를 쓰지 않는다.
- **패킷 누락 가능**: 중간에 일부 패킷이 손실되어도 재전송하지 않는다.

IP 자체도 비연결형 프로토콜이다. IP는 각 패킷을 독립적으로 최선(best-effort)으로 전달하며, 도착을 보장하지 않는다.

```
연결형 (TCP):
연결 확립 → 데이터 → ACK → 데이터 → ACK → 연결 종료
순서 보장, 재전송, 신뢰성 ↑, 속도 ↓

비연결형 (UDP):
데이터 → (도착 여부 모름)
순서 미보장, 재전송 없음, 신뢰성 ↓, 속도 ↑
```

---

## 종합

비연결형은 "효율성을 위해 정확성을 포기"하는 트레이드오프다. 이 특성이 적합한 경우: 실시간 스트리밍(음성/영상), 게임 상태 업데이트, DNS 쿼리(간단한 1회 요청-응답). 이런 경우 약간의 패킷 손실은 허용되고, 재전송으로 인한 지연이 더 나쁜 결과를 낸다.

# [UNVERIFIED] UDP는 TCP와 비교하여 어떤 점이 다른가?

## 도입

UDP(User Datagram Protocol)는 TCP의 반대편에 있는 전송 프로토콜이다. TCP가 신뢰성을 위해 복잡한 절차를 갖는다면, UDP는 그 절차를 모두 버리고 속도를 선택했다.

---

## 본문

**TCP vs UDP 핵심 비교:**

**연결 확립:**

TCP는 데이터를 보내기 전에 3-way handshake로 연결을 확립한다. UDP는 이 과정이 없다. 받는 쪽이 준비됐는지 확인하지 않고 바로 데이터를 보낸다.

```
TCP:
클라이언트 → SYN → 서버
클라이언트 ← SYN-ACK ← 서버
클라이언트 → ACK → 서버
[연결 확립 후 데이터 전송]

UDP:
클라이언트 → 데이터 → 서버 (바로 전송)
```

**데이터 전송 중 확인:**

TCP는 데이터를 보낼 때마다 ACK를 받아 확인한다. ACK가 오지 않으면 재전송한다. UDP는 이 절차가 없다. 보내고 나면 끝이다.

```
TCP:
데이터 → [ACK 기다림] → 다음 데이터

UDP:
데이터 → 데이터 → 데이터 (확인 없이 계속 전송)
```

**기타 차이:**

```
              TCP             UDP
연결 확립    3-way handshake   없음
확인(ACK)    있음               없음
순서 보장    있음               없음
재전송       있음               없음
헤더 크기    20 bytes (최소)    8 bytes
신뢰성       높음               낮음
속도         상대적으로 느림    빠름
용도         HTTP, FTP, 이메일  DNS, 스트리밍, 게임
```

---

## 종합

TCP와 UDP는 같은 전송 계층 프로토콜이지만 설계 철학이 다르다. TCP는 "반드시 정확하게", UDP는 "최대한 빠르게"가 목표다. HTTP/1.1과 HTTP/2가 TCP를 사용하는 이유는 웹 콘텐츠는 정확한 전달이 필수적이기 때문이다. HTTP/3는 QUIC(UDP 기반)을 사용하여 UDP의 속도를 가져오면서 자체적으로 신뢰성을 구현했다.

---

# [UNVERIFIED] UDP가 TCP보다 빠른 이유는 무엇인가?

## 도입

UDP가 TCP보다 빠른 것은 확인 절차를 생략하기 때문이다. 두 종류의 생략이 있다.

---

## 본문

UDP는 두 가지 절약으로 TCP보다 빠르다:

**1. 연결 확립 절차 생략 (3-way handshake 없음)**

TCP는 첫 데이터를 보내기 전에 3-way handshake가 필요하다. 서울-미국 서버 기준으로 RTT(왕복 시간)가 150ms라면, 핸드셰이크만으로 450ms(3번 왕복)가 걸린다. UDP는 이 과정 없이 바로 첫 데이터를 보낸다.

```
TCP:
t=0ms:   SYN 전송
t=150ms: SYN-ACK 수신
t=150ms: ACK 전송
t=300ms: 서버가 ACK 수신, 연결 확립
[이제 데이터 전송 시작]

UDP:
t=0ms: 데이터 전송 시작 (즉시)
```

**2. 데이터 전송 중 확인 절차 생략**

TCP는 매 Segment마다 ACK를 기다린다 (Window Size로 여러 개를 보내더라도). UDP는 보내고 확인 없이 계속 보낸다.

```
TCP (Window Size=1 가정):
→ 데이터 → [150ms 대기] ← ACK → 데이터 → [150ms 대기] ← ACK

UDP:
→ 데이터 → 데이터 → 데이터 → 데이터 (확인 없이 계속)
```

---

## 종합

UDP의 속도 이점은 지연 시간(latency) 측면에서 가장 크다. 특히 첫 번째 데이터를 빠르게 보내야 할 때, 연결 확립 과정을 생략하는 것이 결정적이다. HTTP/3가 QUIC(UDP 기반)을 선택한 이유 중 하나가 연결 확립 시간 단축이다 — QUIC은 암호화 핸드셰이크까지 포함해도 1-RTT 또는 0-RTT로 첫 데이터를 보낼 수 있다.

---

# [UNVERIFIED] 데이터그램(Datagram)이란 무엇인가?

## 도입

TCP에서 데이터 단위를 Segment라고 부르듯, UDP에서는 Datagram이라고 부른다. UDP 헤더가 붙은 데이터 덩어리다.

---

## 본문

Datagram은 UDP 헤더가 붙은 데이터다. 네트워크 계층별 데이터 단위:

```
응용 계층 (DNS, 게임 등):  메시지
     ↓ (UDP 헤더 추가)
전송 계층 (UDP):            데이터그램 (Datagram)
     ↓ (IP 헤더 추가)
네트워크 계층 (IP):          패킷 (Packet)
```

**UDP 헤더 크기:**

UDP 헤더는 8바이트로 매우 작다. TCP 헤더(최소 20바이트)보다 훨씬 작다. 오직 빠른 전송을 위해 최소한의 정보만 포함한다.

```
UDP 헤더 (8 bytes):
┌──────────────────────────────────────────┐
│ Source Port (16 bits) │ Dest Port (16 bits)│
├──────────────────────────────────────────┤
│   Length (16 bits)    │ Checksum (16 bits) │
└──────────────────────────────────────────┘
```

**TCP Segment vs UDP Datagram:**

```
TCP Segment 헤더: 20+ bytes
- Source Port, Dest Port
- Sequence Number (32 bits)
- Acknowledgement Number (32 bits)
- Flags (SYN, ACK, FIN, ...)
- Window Size
- Checksum
- Urgent Pointer
- Options

UDP Datagram 헤더: 8 bytes
- Source Port, Dest Port
- Length
- Checksum
```

TCP가 신뢰성을 위해 많은 정보를 헤더에 담는 반면, UDP는 포트 번호와 길이, 간단한 오류 검사만 포함한다.

---

## 종합

헤더 크기 차이(TCP 20+ bytes vs UDP 8 bytes)는 작아 보이지만, 수백만 개의 작은 패킷을 보내는 경우 누적 오버헤드가 된다. DNS 쿼리처럼 아주 짧은 요청-응답에서 UDP가 선호되는 이유 중 하나다. 또한 UDP 헤더가 작을수록 단일 패킷에 더 많은 실제 데이터를 담을 수 있다.

---

# [UNVERIFIED] UDP 헤더에는 어떤 정보가 담기는가?

## 도입

UDP 헤더는 최소한의 정보만 담는다. 4개의 필드, 총 8바이트다.

---

## 본문

**Source Port Number (출발지 포트)**

이 데이터그램을 보낸 애플리케이션의 포트 번호다. 응답이 필요한 경우 수신측이 이 포트로 보낸다. 응답이 필요 없으면 0으로 설정할 수 있다.

**Destination Port Number (목적지 포트)**

이 데이터그램을 받을 애플리케이션의 포트 번호다. DNS는 포트 53, QUIC(HTTP/3)은 포트 443을 사용한다.

**Length**

UDP 헤더 + 데이터의 전체 바이트 수다.

**Checksum**

전송 중 데이터 손상 여부를 확인하는 오류 검사 값이다. UDP는 오류를 수정하지 않고 감지만 한다. 손상된 데이터그램은 버린다.

```
UDP 헤더 구조:
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
│ Source Port    │ Destination Port │
+--------+--------+--------+--------+
│     Length     │    Checksum     │
+--------+--------+--------+--------+
│                데이터             │
+-----------------------------------+
```

**TCP 헤더와 비교:**

TCP에 있지만 UDP에 없는 필드: Sequence Number, Acknowledgement Number, Window Size, Flags(SYN/ACK/FIN). 이 필드들이 신뢰성(재전송, 순서 보장, 흐름 제어)을 담당하는 것들이다.

---

## 종합

UDP 헤더가 이렇게 단순한 이유는 신뢰성 관련 모든 것을 포기했기 때문이다. 포트 번호는 "어느 애플리케이션"으로 갈지 식별하는 최소한의 정보다. Checksum으로 손상 감지는 하지만 수정은 않는다 — 손상됐으면 그냥 버린다. QUIC(HTTP/3)은 이런 UDP 위에서 Sequence Number, Acknowledgement, 흐름 제어를 직접 구현하여 신뢰성을 다시 추가했다.

# [UNVERIFIED] LAN 환경에서 UDP는 어떻게 활용되는가?

## 도입

같은 로컬 네트워크(LAN) 안에서 UDP가 특히 유용하게 쓰이는 패턴이 있다. 브로드캐스트와의 조합이다.

---

## 본문

LAN에 있는 모든 컴퓨터에 데이터를 동시에 보낼 때 UDP를 사용한다. 이를 브로드캐스트라 한다.

**UDP 브로드캐스트 활용 예:**

**DHCP (IP 자동 할당)**

새 장치가 와이파이에 연결될 때 IP를 받기 위해 DHCP 서버를 찾아야 한다. 아직 IP가 없으므로 직접 DHCP 서버에 연결할 수 없다. 브로드캐스트로 "DHCP 서버 있으면 응답해줘"를 LAN 전체에 UDP로 보낸다.

```
새 장치 (IP 없음)
→ UDP 브로드캐스트 (목적지: 255.255.255.255)
→ "DHCP Discover: 나 IP 좀 줘"

DHCP 서버 (같은 LAN 내)
← UDP 응답: "IP 192.168.0.5 써"
```

**mDNS (멀티캐스트 DNS)**

`MacBook.local` 같은 로컬 호스트명 해석. DNS 서버 없이 LAN 내에서 장치 이름을 찾는다.

**ARP (MAC 주소 조회)**

IP를 알아도 직접 통신하려면 MAC 주소가 필요하다. "192.168.0.5를 가진 장치 있으면 MAC 주소 알려줘"를 브로드캐스트로 보낸다.

---

## 종합

LAN 내 브로드캐스트는 "대상을 모르는 상황에서 모두에게 묻기"의 패턴이다. DHCP, ARP, mDNS가 모두 이 패턴을 사용한다. 이 패턴에 TCP는 적합하지 않다 — 다음 질문에서 그 이유를 설명한다. UDP가 LAN 네트워크 초기화(IP 할당, MAC 조회)의 기반이라는 점에서, 네트워크가 동작하는 가장 기초 단계에 UDP가 있다.

# [UNVERIFIED] 브로드캐스트처럼 불특정 다수에게 데이터를 보낼 때 TCP가 적합하지 않은 이유는?

## 도입

TCP는 일대일(unicast) 통신을 위해 설계됐다. 브로드캐스트처럼 불특정 다수를 대상으로 하는 통신에는 구조적으로 맞지 않는다.

---

## 본문

TCP가 브로드캐스트에 적합하지 않은 두 가지 이유:

**1. 연결 먼저 확립 (3-way handshake)**

TCP는 데이터를 보내기 전에 반드시 대상과 1:1로 연결을 확립해야 한다. 브로드캐스트는 불특정 다수에게 보내는 것이므로, 누구와 연결을 맺어야 하는지 알 수 없다. LAN의 모든 장치와 각각 3-way handshake를 하는 것은 브로드캐스트의 목적(빠르게 모두에게 알리기)과 정반대다.

```
TCP 브로드캐스트를 시도한다면:
→ 255.255.255.255와 SYN 전송
→ (모든 장치가 SYN-ACK 응답?)
→ 어떤 연결을 수락해야 하는지?
→ 구조적으로 불가능
```

**2. 데이터 전송 중 확인 필요**

TCP는 각 수신자로부터 ACK를 받아야 다음 데이터를 보낼 수 있다. 수십, 수백 명에게 동시에 데이터를 보내면서 각각의 ACK를 기다리는 것은 극도로 비효율적이다.

```
UDP 브로드캐스트:
→ 데이터 한 번 전송
→ LAN의 모든 장치가 수신 (확인 없음)

TCP 브로드캐스트 (가정):
→ 장치 A에게 전송 → A의 ACK 기다림
→ 장치 B에게 전송 → B의 ACK 기다림
→ 장치 C에게 전송 → C의 ACK 기다림
→ ... (N개 장치면 N배 시간 필요)
```

**UDP 브로드캐스트의 적합성:**

```
UDP 브로드캐스트:
송신자 → 데이터그램 1개 → 브로드캐스트 주소
                          ↓
          [LAN의 모든 장치가 동시 수신]
```

연결 없음, ACK 없음, 한 번 전송으로 모든 장치 도달. 이것이 DHCP, ARP가 UDP를 선택한 이유다.

---

## 종합

TCP와 UDP의 적합성은 "신뢰성 vs 속도"뿐 아니라 "통신 패턴"에도 달려있다. TCP는 1:1 신뢰성 통신, UDP는 1:N 브로드캐스트/멀티캐스트에 적합하다. 개발자가 `fetch()`로 서버와 통신하는 것은 1:1 TCP, 게임 서버가 모든 플레이어에게 상태를 뿌리는 것은 UDP 멀티캐스트에 가깝다. WebRTC에서 미디어 스트림이 UDP를 쓰는 것도 1:1이지만 "약간의 손실이 재전송 지연보다 낫다"는 판단으로 UDP를 선택한 것이다.
