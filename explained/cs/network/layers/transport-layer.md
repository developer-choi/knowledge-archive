# [UNVERIFIED] Transport Layer의 역할은 무엇인가?

## 도입

Network Layer가 다른 네트워크의 목적지 호스트까지 패킷을 전달한다면, Transport Layer는 그 호스트 안의 어떤 프로세스(포트)에게 데이터를 전달할지를 담당한다. 또한 데이터가 정확하게 전달되는 것을 보장하는 역할도 한다.

---

## 본문

Transport Layer는 다음 세 가지 역할을 한다.

1. **Port Addressing**: 어떤 프로세스(포트)에 데이터를 전달할지 결정.
2. **Segmentation and Reassembly**: 데이터를 세그먼트로 분할하고, 수신 측에서 재조합.
3. **Accuracy (정확성 보장)**: 데이터가 빠짐없이, 순서대로 전달되도록 보장.

```
Transport Layer의 위치

L7: Application (HTTP 메시지 생성)
L4: Transport   (포트 주소 지정, 세그먼테이션, 신뢰성)
L3: Network     (IP 라우팅)
L2: Data Link   (MAC, 프레임)
L1: Physical    (신호 전송)
```

Transport Layer의 대표 프로토콜:
- **TCP**: 신뢰성 보장, 순서 보장, 흐름 제어. HTTP/HTTPS가 사용.
- **UDP**: 신뢰성 미보장, 빠름. DNS 쿼리, 동영상 스트리밍, 게임이 사용.

---

## 종합

Transport Layer는 Network Layer가 "어디까지"를 담당한 것에 더해 "어떤 프로세스에게"와 "정확하게"를 추가한다. `fetch()`가 HTTPS를 통해 보내는 요청은 Transport Layer에서 TCP 세그먼트로 분할되고, 포트 443으로 목적지가 지정되고, 손실 없이 전달됨을 TCP가 보장한다.

# [UNVERIFIED] Port Addressing이란 무엇인가?

## 도입

같은 컴퓨터에서 브라우저로 웹을 열고, Slack으로 메시지를 받고, 음악 스트리밍까지 동시에 되는 이유가 Port Addressing 덕분이다. 포트가 어떤 데이터가 어느 프로세스로 가야 하는지를 구분한다.

---

## 본문

Port Addressing은 프로세스 대 프로세스 전달에 책임을 진다.

Network Layer가 "이 컴퓨터"까지 패킷을 전달하면, Transport Layer는 포트 번호를 보고 "이 컴퓨터의 어떤 프로세스"에게 데이터를 줄지 결정한다.

```
Port Addressing 흐름

인터넷 → 내 PC (IP: 192.168.0.10)
           ↓ Transport Layer
  포트 443 → 브라우저 프로세스 (HTTPS)
  포트 3000 → Node.js 개발 서버
  포트 6379 → Redis 프로세스
```

`fetch("https://example.com")`이 보내는 TCP 세그먼트에는:
- **출발지 포트**: 브라우저가 임의로 선택한 임시 포트 (예: 54321)
- **목적지 포트**: 443 (HTTPS)

서버에서 응답이 오면:
- **출발지 포트**: 443
- **목적지 포트**: 54321 → 브라우저 프로세스로 전달

---

## 종합

포트 번호는 Transport Layer의 "주소"다. IP 주소가 "어느 건물(호스트)"이라면 포트는 "몇 호(프로세스)"다. 포트가 없으면 운영체제가 받은 데이터를 어느 앱에 줄지 알 수 없다.

# [UNVERIFIED] Segmentation and Reassembly는 어떤 순서로 동작하는가?

## 도입

HTTP 응답처럼 큰 데이터를 한 번에 보낼 수 없다. Transport Layer가 데이터를 작은 세그먼트로 자르고, 각 세그먼트에 번호를 붙여서 수신 측이 올바른 순서로 재조합할 수 있게 한다.

---

## 본문

Segmentation and Reassembly는 데이터가 처음 보낸 순서 그대로 도착하는 것을 보장한다.

동작 순서:

1. **수신**: Application Layer에서 내려온 데이터를 받는다.
2. **세그먼테이션**: 데이터를 MTU(Maximum Transmission Unit) 이하 크기로 잘게 자른다.
3. **번호 부착**: 각 세그먼트에 Sequence Number를 붙인다.
4. **포트 추가**: 출발지·목적지 포트 번호도 추가해서 Network Layer로 내린다.
5. **전송**: Network Layer → Data Link Layer → Physical Layer를 거쳐 전송.
6. **재조합**: 도착지 Transport Layer에서 Sequence Number로 원래 순서 그대로 재조합한다. 패킷들이 순서 없이 도착할 수 있기 때문에 번호로 정렬한다.
7. **손실 감지**: Sequence Number로 누락된 세그먼트를 감지하면 재전송을 요청한다.

```
Segmentation and Reassembly

[Application 데이터: 100KB]
         ↓ 분할
세그먼트1 (seq:1, 1KB)
세그먼트2 (seq:2, 1KB)
세그먼트3 (seq:3, 1KB)
...
         ↓ 전송 (순서가 뒤섞일 수 있음)
         ↓ 수신
seq:3 도착 → 보관
seq:1 도착 → 보관
seq:2 도착 → 보관
         ↓ Sequence Number로 정렬 후 재조합
[Application 데이터: 100KB 복원]
```

---

## 종합

Segmentation 없이는 대용량 데이터를 전송할 수 없다 — 네트워크 장비들은 한 번에 처리할 수 있는 패킷 크기에 한계가 있다(MTU). Reassembly 없이는 순서가 바뀌거나 손실된 데이터를 올바르게 복원할 수 없다. TCP의 신뢰성이 바로 이 메커니즘에서 나온다.

# [UNVERIFIED] Accuracy는 왜 Transport Layer의 책임인가?

## 도입

Physical Layer부터 Network Layer까지는 데이터를 목적지까지 "전달하는" 것에 집중하지만, 정확하게 빠짐없이 전달되는 것을 보장하지는 않는다. 이 정확성 보장이 Transport Layer의 역할이다.

---

## 본문

전송 계층 없이 Physical Layer, Datalink Layer, Network Layer까지만 있으면, 다른 네트워크의 목적지까지 데이터를 전송할 수는 있다. 그러나 데이터가 정확하게 전송되는 것까지는 보장할 수 없다.

Transport Layer(TCP)는 정확한 전송을 다음 방법으로 보장한다:
- **ACK (확인 응답)**: 수신 측이 받은 세그먼트에 대해 ACK를 보낸다.
- **재전송**: 송신 측이 타이머 내에 ACK를 받지 못하면 해당 세그먼트를 재전송한다.
- **순서 보장**: Sequence Number로 순서를 맞춘다.
- **오류 감지**: 체크섬으로 손상된 세그먼트를 감지하고 폐기 후 재전송 요청.

```
Transport Layer 없는 세계:
세그먼트 손실 → 그냥 사라짐, 수신 측은 모름
응용 프로그램이 받는 데이터: 불완전할 수 있음

Transport Layer(TCP) 있는 세계:
세그먼트 손실 → ACK 없음 감지 → 자동 재전송
응용 프로그램이 받는 데이터: 항상 완전하고 순서 정확
```

---

## 종합

Transport Layer의 정확성 보장은 상위 계층(Application Layer)이 네트워크를 신뢰할 수 있게 해주는 기반이다. `fetch()` 응답이 항상 완전한 JSON으로 오는 것은 TCP가 배경에서 손실된 패킷을 재전송하고 순서를 맞춰주기 때문이다. 이 보장이 필요 없는(빠름이 더 중요한) 상황에서는 UDP를 선택한다.

# [UNVERIFIED] Port Number와 PID의 차이는 무엇인가?

## 도입

포트 번호(Port Number)와 프로세스 ID(PID)는 둘 다 "무언가를 구분하는 번호"지만 다른 것을 구분한다. 이 차이를 이해하면 "왜 포트가 별도로 필요한가"가 명확해진다.

---

## 본문

- **PID**: 실행 중인 프로그램(프로세스)을 구분하기 위한 값. OS가 관리하는 프로세스 식별자다.
- **Port Number**: (통신 중인) 포트를 구분하기 위한 값. 프로세스가 네트워크와 통신하는 창구(소켓)를 식별한다.

세 가지 차이점:

1. **모든 프로세스가 네트워크와 통신하지는 않는다**: PID는 모든 실행 중인 프로세스에 있지만, 포트는 네트워크 소켓을 열 때만 존재한다.

2. **PID는 매번 다르다**: 프로세스를 재시작하면 PID가 바뀐다. 아무 값이나 가질 수 있다 (OS가 그때그때 빈 번호를 할당).

3. **포트는 매번 다르지 않을 수 있다**: Well-known port는 서비스와 약속된 번호다. HTTP는 항상 80, HTTPS는 항상 443 — 프로세스가 재시작해도 같은 포트를 쓴다.

```
PID vs Port Number

Node.js 서버 실행:
PID: 12345 (이번 실행에 임의로 배정됨)
Port: 3000 (코드에서 명시)

재시작 후:
PID: 12346 (바뀜)
Port: 3000 (그대로)
```

---

## 종합

PID는 OS 내부에서 프로세스를 관리하는 번호이고, Port는 네트워크 상에서 프로세스의 통신 창구를 식별하는 번호다. Port를 쓰는 이유는 PID가 매번 바뀌어 예측 불가능하기 때문이다 — 클라이언트가 "포트 3000으로 Node.js 서버에 연결"이라고 하면 PID가 달라져도 항상 같은 서비스에 도달한다.

---

# [UNVERIFIED] Port Number란 무엇이며, 한 컴퓨터에서 여러 서비스를 동시에 사용할 수 있는 이유는?

## 도입

같은 컴퓨터에서 브라우저와 메일 클라이언트와 Slack이 동시에 네트워크를 쓸 수 있는 이유가 포트 번호다. IP 주소는 컴퓨터까지의 주소이고, 포트는 그 안의 어느 프로세스인지를 지정한다.

---

## 본문

Port Number는 포트를 구분하기 위한 값이다. 포트는 프로세스가 네트워크와 통신하는 창구를 말한다.

한 컴퓨터에서 여러 서비스(예: 웹 브라우저, 메일 프로그램)를 동시에 사용할 수 있는 이유는 포트 번호가 있기 때문이다. 포트 번호가 있어야 도착한 데이터가 어느 프로세스로 전달되어야 하는지 구분할 수 있다.

```
포트 번호가 없다면?

서버에서 응답이 오면 OS가 "이걸 어느 앱에 줄지" 알 수 없다.
→ 한 컴퓨터에서 하나의 네트워크 애플리케이션만 실행 가능.

포트 번호가 있으면?

응답 패킷의 목적지 포트 = 54321 → 브라우저 창구
응답 패킷의 목적지 포트 = 49152 → Slack 창구
→ 수천 개 앱이 동시에 네트워크 사용 가능.
```

---

## 종합

포트 번호는 같은 호스트 내의 멀티플렉싱을 가능하게 한다. IP 주소가 편지 봉투의 "건물 주소"라면, 포트 번호는 "수신자(호실)"이다. 운영체제는 도착하는 패킷의 목적지 포트를 보고 해당 소켓을 열고 있는 프로세스에게 데이터를 전달한다.

# [UNVERIFIED] Port Number의 범위별 분류(well-known / registered / dynamic)는 어떻게 나뉘는가?

## 도입

포트 번호는 0~65535 범위가 있고, 그 범위를 세 구간으로 나눠 용도를 정해둔다. 어떤 포트가 "예약된" 포트이고 어떤 것을 자유롭게 써도 되는지 알면 서버 개발 시 포트 충돌을 피할 수 있다.

---

## 본문

포트 번호는 0~65535 (2^16 - 1)까지 65536개다.

세 구간으로 나뉜다:

| 범위 | 이름 | 용도 |
|---|---|---|
| 0 ~ 1023 | Well-known port | IANA가 주요 프로토콜에 예약. 관리자 권한 필요. |
| 1024 ~ 49151 | Registered port | 특정 서비스가 등록. 일반 사용자도 사용 가능. |
| 49152 ~ 65535 | Dynamic port | 임시 포트. 클라이언트 소켓이 자동 할당. |

대표적인 Well-known 포트:
- 22: SSH
- 25: SMTP (이메일 송신)
- 53: DNS
- 80: HTTP
- 443: HTTPS
- 3306: MySQL
- 5432: PostgreSQL

```
포트 번호 구조

0                    1024              49152           65535
│← well-known →│← registered port →│← dynamic port →│
│ (시스템 예약)  │ (등록된 서비스)    │ (임시 할당)      │

fetch("https://example.com") → 브라우저가 임시 포트(동적, 예: 54321) 사용
서버: 443 포트 (well-known, HTTPS)
```

---

## 종합

개발 서버를 3000번 포트에 띄우는 것이 관행인 이유는 1024~49151 범위(registered port)에 속해 관리자 권한 없이 쓸 수 있기 때문이다. 1024 이하 포트(well-known)는 root/관리자 권한이 필요해 개발 환경에서는 불편하다. 클라이언트(`fetch()`)가 연결할 때 자동으로 할당받는 포트가 49152~65535 범위의 동적 포트다.
