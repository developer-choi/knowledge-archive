# [UNVERIFIED] 데이터링크 계층(Datalink Layer)이란 무엇인가?

## 도입

Network Layer(L3)가 다른 네트워크까지 경로를 찾아준다면, Datalink Layer(L2)는 같은 네트워크 안의 다음 노드까지 실제 전달을 담당한다. MAC 주소가 이 계층의 식별자이고, 이더넷이 가장 흔한 구현 기술이다.

---

## 본문

데이터링크 계층은 노드 간 전달에 책임이 있는 계층이다. 같은 네트워크 안에서 데이터를 주고받기 위해 필요하며, MAC 주소를 통해 다음 노드가 어디에 있는지를 알 수 있다.

Network Layer가 "서울에서 뉴욕까지의 경로"를 결정한다면, Datalink Layer는 "이 라우터에서 다음 라우터까지 어떻게 건네주는가"를 처리한다. 즉, IP가 정한 최종 목적지까지의 여정에서 한 홉(hop) 안의 전달을 Datalink Layer가 담당한다.

```
계층 간 역할 분리

L3 (Network):   출발지 → 목적지 전체 경로 (IP 기반)
L2 (Datalink):  이 홉에서 다음 홉까지 (MAC 기반)

예: PC(서울) → 공유기 → ISP 라우터 → ... → 서버(뉴욕)

각 구간은 Datalink Layer가 처리:
PC → 공유기: src MAC: AA:AA... dst MAC: BB:BB...
공유기 → ISP: src MAC: BB:BB... dst MAC: CC:CC...  (MAC은 홉마다 교체)
```

---

## 종합

Datalink Layer는 IP 패킷을 물리 네트워크 위에서 한 홉씩 전달하는 계층이다. 스위치가 이 계층에서 동작하며 MAC 주소 테이블로 정확한 포트로 프레임을 보낸다. Network Layer가 없으면 다른 네트워크에 도달할 수 없고, Datalink Layer가 없으면 같은 네트워크 안에서도 정확한 장치에 닿지 못한다.

# [UNVERIFIED] 이더넷(Ethernet)이란 무엇인가?

## 도입

Datalink Layer에서 가장 널리 쓰이는 기술이 이더넷이다. 유선 LAN을 구축할 때 쓰는 케이블과 포트가 바로 이더넷이고, 여기서 사용하는 프로토콜도 이더넷 프로토콜이다.

---

## 본문

이더넷은 데이터링크 계층에서 가장 많이 사용되는 기술이다. LAN을 구축하기 위해 디바이스를 연결할 때 쓰는 프로토콜이다.

이더넷은 프레임 단위로 데이터를 전송한다. 프레임은 이더넷 헤더(출발지 MAC, 목적지 MAC, 타입) + 페이로드(IP 패킷 등) + 트레일러(FCS)로 구성된다.

```
이더넷의 위치

물리 인프라:    RJ45 케이블, 이더넷 스위치, NIC
프로토콜:       이더넷 프레임 (MAC 주소 기반 전달)
상위에서 보면:  IP 패킷을 운반하는 포장지 역할
```

와이파이(Wi-Fi)도 Datalink Layer에서 동작하지만 이더넷 대신 IEEE 802.11 프로토콜을 쓴다. 유선 이더넷과 무선 Wi-Fi는 같은 계층의 서로 다른 구현이다.

---

## 종합

이더넷은 현대 LAN의 표준이다. 사무실 PC가 랜 케이블로 꽂히면 이더넷 프레임으로 통신하고, 노트북이 와이파이로 연결하면 802.11 프레임으로 통신한다. 상위 계층(IP, TCP, HTTP)은 이 차이를 알 필요가 없다 — Datalink Layer가 추상화해준다.

# [UNVERIFIED] CSMA/CD란 무엇이며, 어떤 순서로 동작하는가?

## 도입

허브 환경에서는 여러 장치가 공유 케이블을 사용하기 때문에 동시에 신호를 보내면 충돌이 발생한다. CSMA/CD는 이 충돌을 최소화하기 위해 이더넷이 채택한 프로토콜이다.

---

## 본문

허브 환경에서는 여러 컴퓨터가 동시에 데이터를 보내면 충돌(collision)이 발생할 수밖에 없다. 충돌이 발생하는 원인은 데이터가 동시에 같은 케이블을 지나가기 때문이다. 그래서 이더넷은 데이터를 보내는 시점을 조율하는 방식(CSMA/CD)으로 동작한다.

동작 순서:

**CS (Carrier Sense) — 반송파 감지**
- 송신 측 컴퓨터가 케이블에 데이터가 전송되고 있는지 확인한다.
- 전송 중이면 잠시 기다렸다가 다시 확인한다.

**MA (Multiple Access) — 다중 접근**
- 케이블에 데이터가 흐르고 있지 않으면 데이터를 보내도 좋다는 규칙이다.
- 여러 장치가 케이블에 동시에 접근할 수 있다는 전제 하에 순서를 조율한다.

**CD (Collision Detection) — 충돌 감지**
- 전송 중 충돌이 발생했는지 감지한다.
- 충돌이 감지되면 JAM 신호를 보내 모든 장치에 알리고, 무작위 시간 후 재시도한다.

```
CSMA/CD 흐름

PC-A: 케이블 확인 (CS) → 비어있음 → 전송 시작
PC-B: 케이블 확인 (CS) → 비어있음 → 전송 시작  ← 거의 동시에
                ↓
        충돌 발생 (CD 감지)
        JAM 신호 송출
        모든 장치 전송 중단
        무작위 백오프 타이머 설정
        타이머 만료 후 재시도 (CS부터 다시)
```

---

## 종합

CSMA/CD는 충돌을 완전히 막는 게 아니라 충돌 발생 후 빠르게 복구하는 방식이다. 장치가 많아질수록 충돌 빈도가 높아져 효율이 떨어진다. 현재 스위치가 표준이 된 이유가 여기 있다 — 스위치는 포트마다 별도 collision domain을 만들어 충돌 자체를 근본적으로 제거한다.

---

---

# [UNVERIFIED] 현재 이더넷에서 CSMA/CD가 사용되지 않는 이유는?

## 도입

CSMA/CD는 허브 환경의 산물이다. 스위치가 표준이 된 지금은 CSMA/CD가 더 이상 필요하지 않다.

---

## 본문

효율이 좋지 않기 때문에 지금은 CSMA/CD를 사용하지 않는다. 그 대신 스위치를 사용한다.

스위치는 각 포트가 독립 collision domain이므로 충돌 자체가 발생하지 않는다. 또한 Full Duplex(전이중) 통신이 가능해 송신과 수신이 동시에 일어날 수 있어 CSMA/CD가 필요 없다.

장비 발전 순서: **리피터 → 허브 → 스위치**

- 리피터: 신호 증폭만, 1:1 통신
- 허브: 신호 증폭 + 다중 장치, Half Duplex, 충돌 빈번
- 스위치: MAC 학습, 정확한 포트 전달, Full Duplex, 충돌 없음

---

## 종합

CSMA/CD가 역사의 뒤안길로 사라진 것은 스위치라는 더 나은 대안이 등장했기 때문이다. 스위치 환경에서는 각 포트가 독립 점대점 링크처럼 동작해 충돌이 원천적으로 없다. 현재 이더넷 표준은 여전히 CSMA/CD를 명세하지만, 실제로 쓰이는 환경은 거의 없다.

# [UNVERIFIED] 프레임(Frame)이란 무엇이며, 어떻게 구성되는가?

## 도입

Datalink Layer는 IP 패킷을 그냥 전선에 올리는 게 아니라 "프레임"이라는 형식으로 감싸서 전달한다. 프레임은 이더넷 헤더와 트레일러로 패킷을 감싼 구조다.

---

## 본문

데이터에 이더넷 헤더와 트레일러를 추가한 것을 프레임(Frame)이라고 부른다.

```
이더넷 프레임 구조

┌────────────────┬────────────────────────────────────┬──────┐
│ 이더넷 헤더     │         페이로드 (IP 패킷)            │ FCS  │
│                │                                    │ 트레일러 │
│ 목적지 MAC (6B) │ IP 헤더 + TCP 헤더 + 데이터          │ 4B   │
│ 출발지 MAC (6B) │                                    │      │
│ 이더넷 타입 (2B) │                                    │      │
└────────────────┴────────────────────────────────────┴──────┘
```

헤더 앞에는 프리앰블(8바이트)이 붙어 수신 측이 프레임 시작을 동기화하도록 한다.

---

## 종합

프레임은 IP 패킷을 한 홉 전달하기 위한 포장이다. 다음 홉에 도달하면 이더넷 프레임이 벗겨지고(decapsulation), IP 패킷이 추출되어 라우팅 결정에 따라 다시 새 이더넷 프레임으로 포장(encapsulation)된다. 목적지에 도달할 때까지 이 포장-벗기기-재포장 과정이 홉마다 반복된다.

---

# [UNVERIFIED] 이더넷 헤더에는 어떤 정보가 담기는가?

## 도입

이더넷 헤더는 Datalink Layer가 "이 프레임을 어디서 누구에게 보내는가"를 기록하는 곳이다. 세 가지 필드로 구성된다.

---

## 본문

이더넷 헤더는 다음 세 가지로 구성된다.

- **목적지 MAC 주소 (6바이트)**: 이 프레임이 도달해야 할 다음 노드의 MAC 주소. 브로드캐스트이면 `FF:FF:FF:FF:FF:FF`.
- **출발지 MAC 주소 (6바이트)**: 이 프레임을 보내는 NIC의 MAC 주소. 스위치가 이 값을 보고 MAC 주소 테이블을 자동 학습한다.
- **이더넷 타입 (2바이트)**: 상위 계층(Network Layer)의 프로토콜 종류를 나타낸다. `0x0800`이면 IPv4, `0x86DD`이면 IPv6, `0x0806`이면 ARP다.

```
이더넷 헤더 예시

목적지 MAC: AA:BB:CC:DD:EE:FF (다음 홉 라우터)
출발지 MAC: 11:22:33:44:55:66 (내 NIC)
타입:       0x0800            (IPv4 패킷 포함)
```

---

## 종합

이더넷 헤더의 MAC 주소는 홉마다 교체된다 — 목적지 MAC은 다음 홉의 MAC이고, 출발지 MAC은 현재 장치의 MAC이다. IP 헤더의 IP 주소(최종 출발지·목적지)는 바뀌지 않는 것과 대조적이다. 이더넷 타입 필드 덕분에 수신 측이 페이로드를 올바른 프로토콜 스택으로 넘길 수 있다.

---

# [UNVERIFIED] 트레일러(FCS, Frame Check Sequence)의 역할은?

## 도입

프레임이 전송되는 과정에서 전기 노이즈 등으로 비트가 뒤집힐 수 있다. FCS는 이 오류를 수신 측에서 감지하기 위한 체크섬이다.

---

## 본문

트레일러(FCS, Frame Check Sequence)는 데이터 전송 도중 오류가 발생했는지 확인하는 용도다. CRC(Cyclic Redundancy Check) 값이 들어 있다.

동작 방식:
1. 송신 측: 헤더 + 페이로드에 대한 CRC 값을 계산해 FCS에 붙인다.
2. 수신 측: 받은 프레임의 헤더 + 페이로드로 CRC를 다시 계산한다.
3. 계산된 CRC와 FCS의 값이 일치하면 정상, 다르면 오류 → 프레임 폐기.

```
FCS 동작

송신: [헤더 + 페이로드] → CRC 계산 → 결과값을 FCS로 붙임
전송: [헤더 + 페이로드 + FCS]
수신: [헤더 + 페이로드]의 CRC 재계산
비교: 재계산값 == FCS값 → OK / != → 프레임 폐기
```

---

## 종합

FCS는 오류를 "감지"하는 도구지 "수정"하는 도구가 아니다. 오류가 감지된 프레임은 폐기되고, 상위 계층(TCP 등)이 재전송을 요청한다. Datalink Layer는 전달 보장이 아니라 최선 전달(best effort)이다 — 신뢰성(재전송 보장)은 Transport Layer(TCP)의 역할이다.

# [UNVERIFIED] 허브 환경에서 특정 컴퓨터로 데이터를 보내면 다른 컴퓨터들은 어떻게 동작하는가?

## 도입

허브는 들어온 신호를 모든 포트에 복사해 뿌리는 장치다. 따라서 목적지가 아닌 컴퓨터도 프레임을 받게 되는데, 이 상황에서 각 컴퓨터의 NIC가 어떻게 처리하는지를 다룬다.

---

## 본문

허브에 연결된 PC-A가 PC-C로 데이터를 전송한다고 가정하면:

1. PC-A가 이더넷 프레임을 전송한다. 목적지 MAC: PC-C의 MAC.
2. 허브는 연결된 모든 포트(PC-B, PC-C, PC-D, PC-E)에 프레임을 복사해 전달한다.
3. PC-B, PC-D, PC-E의 NIC: 목적지 MAC 주소와 자신의 MAC 주소를 비교 → 불일치 → 프레임 폐기.
4. PC-C의 NIC: 목적지 MAC 주소가 자신과 일치 → 프레임 수신, 상위 계층으로 전달.

```
허브 환경 데이터 전달

PC-A (전송) → [허브] → PC-B (받음, 파기)
                    → PC-C (받음, 처리) ← 목적지
                    → PC-D (받음, 파기)
                    → PC-E (받음, 파기)
```

이 동작의 보안 문제: PC-B, D, E의 NIC를 promiscuous mode(무차별 모드)로 설정하면 목적지가 다른 프레임도 파기하지 않고 상위 계층으로 올린다. 이것이 허브 환경에서의 도청(packet sniffing)이다.

---

## 종합

허브 환경에서 "필터링"은 NIC 레벨에서 일어난다. NIC가 정상 모드이면 자신의 MAC이 아닌 프레임은 자동으로 버린다. 스위치는 이 불필요한 전달 자체를 없앤다 — 목적지 포트에만 프레임이 가므로 다른 장치가 아예 받지 않는다.

---

# [UNVERIFIED] 스위치의 MAC 주소 테이블이 비어 있을 때 발생하는 flooding이란 무엇인가?

## 도입

스위치는 MAC 주소 테이블을 기반으로 정확한 포트에만 프레임을 전달한다. 그런데 테이블이 비어 있거나 목적지 MAC을 모를 때는 어떻게 동작하는가?

---

## 본문

스위치에는 MAC address table이 있고, 포트 번호와 MAC 주소가 쌍으로 저장된다. 이 테이블 덕분에 스위치는 특정 컴퓨터에게만 데이터를 전송할 수 있다.

하지만 스위치 전원을 처음 켰을 때는 MAC address table이 비어 있다. 이 상태에서 테이블에 없는 MAC 주소로 데이터를 전송하려고 하면, 스위치는 연결된 다른 모든 포트로 프레임을 전달한다. 이것을 **flooding**이라고 부른다.

```
스위치 flooding 과정

초기 상태: MAC 주소 테이블 비어 있음

PC-A (MAC: 11:11:11) → 스위치 → "22:22:22 어디 있어?"
스위치: 테이블에 없음 → 모든 포트에 flooding

동시에 스위치는 PC-A의 MAC(11:11:11)을 포트1에 기록
PC-C (MAC: 22:22:22)가 응답하면:
스위치: PC-C의 MAC(22:22:22)을 해당 포트에 기록

이후 22:22:22로의 프레임은 해당 포트에만 전달 (flooding 없음)
```

flooding은 미지의 목적지 MAC에 대한 일시적 동작이고, 스위치가 MAC을 학습하면 이후부터는 정확한 포트로만 전달한다.

---

## 종합

flooding은 스위치의 자동 학습 메커니즘의 부산물이다. 처음에는 모르기 때문에 허브처럼 동작하지만, 트래픽이 오가면서 빠르게 MAC 테이블이 채워진다. 정상 운영 중에는 flooding이 거의 없다. 단, MAC flooding 공격(가짜 MAC을 대량으로 흘려 테이블을 꽉 채우면 스위치가 계속 flooding하게 만들 수 있음)은 보안 취약점이 된다.

# [UNVERIFIED] Full Duplex와 Half Duplex의 차이는?

## 도입

통신 회선에서 송신과 수신을 동시에 할 수 있는가 없는가에 따라 Full Duplex와 Half Duplex로 나뉜다. 이 차이가 이더넷 환경의 효율과 충돌 발생 여부에 직접 영향을 준다.

---

## 본문

**Full Duplex**: 송신과 수신이 동시에 일어나는 통신 방식이다. 전화 통화처럼 양측이 동시에 말할 수 있다.

**Half Duplex**: 하나의 회선으로 송신과 수신을 번갈아 가며 하는 방식이다. 무전기처럼 한쪽이 말할 때 다른 쪽은 들어야 한다.

```
Full Duplex vs Half Duplex

Full Duplex:    ──────→  (송신)
                ←──────  (수신)
동시에 가능. 충돌 없음.

Half Duplex:    ──────→  (송신 중)
              (수신 대기)
순서가 필요. 충돌 발생 가능.
```

---

## 종합

현대 스위치 환경에서는 모두 Full Duplex다. 스위치의 각 포트가 독립 점대점 링크이고, 송신·수신 선로가 분리되어 있어 동시 통신이 가능하다. CSMA/CD가 필요 없는 이유도 Full Duplex에서 충돌이 원천 불가능하기 때문이다.

---

---

# [UNVERIFIED] 허브가 Half Duplex를 사용할 수밖에 없는 이유는?

## 도입

허브는 공유 매체(shared medium) 방식이다. 모든 장치가 같은 선로를 공유하므로 동시에 신호를 보내면 충돌이 난다. 이것이 Half Duplex를 강제하는 이유다.

---

## 본문

허브는 송신과 수신을 동일한 회선으로 진행하기 때문에, 동시에 데이터를 보내면 충돌이 발생한다. 그래서 송신과 수신을 번갈아 가며 진행할 수밖에 없고, 결과적으로 효율이 떨어진다.

충돌이 발생하면 CSMA/CD가 처리하지만, 장치 수가 많을수록 충돌 빈도가 올라가 실효 처리량(goodput)이 크게 낮아진다.

---

## 종합

허브가 Half Duplex인 것은 공유 선로 구조의 필연적 결과다. 단일 collision domain에 여러 장치가 붙어 있으면 "하나가 말할 때 나머지는 듣는다"는 규칙이 없으면 혼선이 생긴다. 스위치는 이 공유 선로 문제를 포트 별 독립 링크로 해결했다.

---

---

# [UNVERIFIED] 스위치가 Full Duplex 통신이 가능한 이유는?

## 도입

스위치가 Full Duplex로 동작할 수 있는 물리적 이유는 송신과 수신에 별도 선로를 사용하기 때문이다.

---

## 본문

스위치는 송신과 수신하는 선로가 별도이기 때문에 Full Duplex로 통신할 수 있다.

이더넷 케이블(Cat5e 이상)은 여러 쌍의 와이어로 구성되어 있다. 송신에 사용되는 쌍과 수신에 사용되는 쌍이 물리적으로 분리되어 있어, 동시에 양방향 신호를 실어 나를 수 있다.

```
이더넷 케이블 (RJ45, 4쌍 8핀)

핀 1, 2: 송신 (TX+, TX-)
핀 3, 6: 수신 (RX+, RX-)
핀 4, 5, 7, 8: 나머지 (기가비트에서는 4쌍 모두 사용)

→ TX와 RX가 물리적으로 분리 → 동시 사용 가능 → Full Duplex
```

---

## 종합

Full Duplex는 선로 분리라는 물리적 설계 덕분에 가능하다. 허브는 모든 장치가 같은 선로를 공유해 Half Duplex가 강제되지만, 스위치는 포트별 독립 링크로 Full Duplex를 기본으로 제공한다. 이론상 100Mbps 링크에서 Full Duplex는 송수신 각 100Mbps로 합계 200Mbps 처리량이 가능하다.

# [UNVERIFIED] 데이터링크 계층의 sublayer인 LLC와 MAC은 각각 어떤 역할을 하는가?

## 도입

IEEE 802 표준은 Datalink Layer를 두 개의 서브레이어로 나눈다 — 상위의 LLC와 하위의 MAC. 각각 흐름 제어·오류 제어와 매체 접근 제어를 담당한다.

---

## 본문

**LLC sublayer (Logical Link Control)**
- 흐름 제어와 오류 제어를 담당한다.
- 상위 계층(Network Layer)과 하위 MAC sublayer 사이의 인터페이스 역할을 한다.
- 여러 Network Layer 프로토콜(IPv4, IPv6 등)을 동일한 MAC sublayer 위에서 다룰 수 있게 한다.

**MAC sublayer (Medium Access Control)**
- CSMA/CD 같은 매체 접근 방법에 대한 동작을 담당한다.
- MAC 주소를 사용해 프레임의 출발지·목적지를 식별한다.
- 실제 물리 매체(케이블, 와이파이)에 접근하는 규칙을 정의한다.

```
Datalink Layer 내부 구조

7. Application
   ...
3. Network (IP)
   ─────────────────
2. Data Link
   ├── LLC sublayer  흐름 제어, 오류 제어, 다중 프로토콜 지원
   └── MAC sublayer  매체 접근 제어 (CSMA/CD), MAC 주소 처리
   ─────────────────
1. Physical
```

---

## 종합

LLC가 없으면 Network Layer가 Datalink Layer에 어떻게 요청을 전달해야 하는지 표준화된 방법이 없고, MAC이 없으면 공유 매체에서 충돌을 조율할 방법이 없다. 이 두 서브레이어의 분리 덕분에 이더넷 MAC 위에서 IPv4·IPv6을 모두 운반할 수 있다.
