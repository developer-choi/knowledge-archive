# [UNVERIFIED] HDD란 무엇이며, 어떤 특성을 가지는가?

## 도입

보조기억장치는 전원이 꺼져도 데이터가 유지되는 비휘발성 저장장치다. 그 중 HDD는 회전하는 자기 원형판에 데이터를 자기적으로 기록하는 방식을 사용한다. Node.js에서 `fs.readFile()`로 파일을 읽을 때 실제로 데이터를 꺼내오는 곳이 HDD나 SSD 같은 보조기억장치다.

---

## 본문

HDD(Hard Disk Drive)는 플래터(platter)라 불리는 원형 자기 판이 고속으로 회전하면서 동작한다. 헤드(head)라는 읽기/쓰기 부품이 플래터 위를 지나가며 데이터를 자기적으로 기록하거나 읽어낸다. 데이터는 플래터의 동심원 트랙(track) 위에 저장된다.

주요 특성:

- **비휘발성**: 전원이 꺼져도 자기 방식으로 기록된 데이터가 유지된다. RAM이 전원 차단 시 데이터를 잃는 것과 반대다.
- **순차 접근(Sequential Access)**: 헤드가 물리적으로 움직여야 하기 때문에, 연속된 위치의 데이터를 읽을 때는 빠르지만 떨어진 위치로 이동할 때는 탐색 시간(seek time)이 발생한다.

```
HDD 구조
┌─────────────────────────┐
│  플래터 (원형 자기 판)    │
│  ┌─────────────────┐   │
│  │ 트랙 (동심원)    │   │
│  │  • 섹터로 분할   │   │
│  └─────────────────┘   │
│  헤드: 플래터 위를 이동  │
└─────────────────────────┘
```

---

## 종합

HDD는 비휘발성과 대용량이라는 장점 덕분에 수십 년간 기본 보조기억장치로 사용됐다. 물리적인 회전 메커니즘으로 인해 임의 접근(random access)이 SSD보다 느리다는 단점이 있다. Node.js 서버가 파일을 읽을 때 지연이 생기는 이유 중 하나가 바로 이 물리적 탐색 시간이다. 현대에는 SSD가 HDD를 빠르게 대체하고 있지만, 대용량 데이터 저장 비용이 중요한 환경에서는 HDD가 여전히 사용된다.

# [UNVERIFIED] RAID란 무엇인가?

## 도입

디스크 하나는 언제든 고장날 수 있다. RAID는 여러 개의 물리 디스크를 묶어 하나의 논리적 저장소처럼 다루는 기술이다. 단순히 용량을 합치는 것을 넘어, 고장에도 데이터를 지키거나 성능을 높이는 목적으로 사용된다.

---

## 본문

RAID는 **Redundant Array of Independent Disks**의 약자다. 단어를 분해하면:

- **Redundant**: 중복. 같은 데이터를 여러 디스크에 나눠 담아 하나가 죽어도 복구할 수 있게 한다.
- **Array**: 배열. 단일 디스크가 아니라 여러 개를 묶는다.
- **Independent Disks**: 각 디스크는 물리적으로 독립된 장치다.

OS나 애플리케이션 입장에서는 RAID로 묶인 디스크 배열이 마치 하나의 큰 디스크처럼 보인다. 이것이 스토리지 가상화(storage virtualization)의 핵심이다.

---

## 종합

RAID가 없다면 디스크 하나가 고장났을 때 그 위의 데이터는 모두 사라진다. RAID는 이 단일 장애점(single point of failure) 문제를 해결한다. 데이터센터에서 수천 대의 서버가 돌아가는 환경에서 디스크 고장은 "언제 발생할까"가 아니라 "얼마나 자주 발생할까"의 문제다. RAID는 그 필연적인 고장에 대비하는 첫 번째 방어선이다.

---

---

# [UNVERIFIED] RAID의 궁극적인 목적은 무엇인가?

## 도입

RAID를 처음 설계할 때의 목표와 현재 실무에서 RAID를 사용하는 목적은 다르다. RAID가 처음 등장했을 때는 작고 느린 디스크들을 합쳐 대용량을 만드는 것이 주목적이었지만, 저장장치 기술이 발전하면서 주목적이 바뀌었다.

---

## 본문

RAID가 제공하는 가치는 네 가지다:

- **Capacity(용량)**: 여러 디스크를 묶어 더 큰 논리 볼륨을 만든다.
- **Reliability(신뢰성)**: 데이터를 중복·분산 저장해 일부 디스크가 고장나도 데이터를 유지한다.
- **Availability(가용성)**: 고장 중에도 시스템이 계속 동작할 수 있게 한다.
- **Performance(성능)**: 여러 디스크에 병렬로 읽고 써서 I/O 처리량을 높인다.

현대 RAID의 궁극 목적은 **Fault tolerant(장애 허용)** 즉, 신뢰성과 가용성이다. 옛날에는 디스크가 느리고 작아서 여러 개를 합치는 것 자체가 필요했지만, 현재 SSD 한 장만으로도 충분한 용량과 성능을 얻을 수 있기 때문에 RAID를 구성하는 주된 이유는 "하나가 죽어도 서비스가 계속 돌아가게"이다.

---

## 종합

Node.js 서버가 파일 시스템에 쓰는 데이터는 RAID가 없다면 디스크 고장 한 번으로 모두 날아갈 수 있다. RAID Fault Tolerance는 그 순간에도 서비스를 계속 제공할 수 있게 해준다. 용량이 넉넉하고 SSD처럼 빠른 저장장치가 보편화된 현재, RAID의 핵심 가치는 데이터 중복 저장을 통한 신뢰성 확보에 집중되어 있다.

---

# [UNVERIFIED] Fault tolerant란 무엇인가?

## 도입

시스템은 반드시 어느 순간 고장난다. Fault tolerant는 그 고장이 발생했을 때도 전체 시스템이 멈추지 않고 계속 동작할 수 있는 능력이다. 고장을 "막는" 것이 아니라 고장이 발생해도 "버티는" 것이 핵심이다.

---

## 본문

Fault tolerant 시스템은 구성 요소 일부에 장애가 생겨도 전체 기능을 유지한다. 이를 달성하는 전형적인 방법이 **중복(redundancy)**이다 — 같은 역할을 하는 구성 요소를 여러 개 두면 하나가 죽어도 나머지가 이어받는다.

디스크 RAID에서의 Fault Tolerance:

```
디스크 A ─┐
           ├─→ RAID 컨트롤러 ─→ OS(단일 볼륨으로 보임)
디스크 B ─┘

디스크 A 고장 → RAID 컨트롤러가 디스크 B로 계속 서비스
```

---

## 종합

Fault tolerant가 없으면 단일 부품 고장이 곧 서비스 중단으로 이어진다. Chrome 브라우저가 탭 하나가 죽어도 다른 탭이 살아있는 것처럼, Fault tolerant 설계는 "부분 실패가 전체 실패로 번지지 않게" 격리하는 것이다. 대규모 서비스에서는 디스크 수준을 넘어 서버·데이터센터 단위까지 Fault tolerance를 설계한다.

---

# [UNVERIFIED] RAID 구성 방식은 어떻게 되는가?

## 도입

RAID는 하나의 방식이 아니라 여러 레벨로 나뉜다. 레벨마다 성능·용량·신뢰성의 트레이드오프가 다르기 때문에, 상황에 맞는 레벨을 선택해야 한다.

---

## 본문

RAID 구성 방식은 크게 세 가지로 분류된다:

- **Standard levels**: RAID 0, RAID 1, RAID 2, RAID 3, RAID 4, RAID 5, RAID 6. 각 번호마다 스트라이핑·미러링·패리티 조합이 다르다.
- **Nested (hybrid) levels**: 표준 레벨을 중첩 조합한다. RAID 01(RAID 0+1), RAID 10(RAID 1+0), RAID 03 등.
- **Non-standard levels**: 특정 벤더나 용도에 맞춰 설계된 레벨. RAID-DP, RAID-tE, Intel Matrix RAID 등.

```
RAID 0 — 스트라이핑(성능 최대, 신뢰성 없음)
디스크A [1][3][5]
디스크B [2][4][6]

RAID 1 — 미러링(신뢰성 최대, 용량 50%)
디스크A [1][2][3]
디스크B [1][2][3] (동일 복사)

RAID 5 — 스트라이핑+패리티(균형)
디스크A [1][2][P]
디스크B [3][P][4]
디스크C [P][5][6]
```

---

## 종합

RAID 레벨 선택은 결국 용량·성능·신뢰성 중 무엇을 우선하느냐의 문제다. 개인용 NAS에서 데이터 백업 목적이라면 RAID 1(미러링)이 단순하고 확실하다. 고성능이 필요한 데이터베이스 서버라면 RAID 10(미러링+스트라이핑)이 흔히 사용된다. 어떤 레벨을 선택하든 RAID는 백업의 대체가 아님을 주의해야 한다 — RAID는 고장을 허용하는 것이지, 실수로 삭제한 파일을 복구해주지는 않는다.

---

# [UNVERIFIED] 네트워크 저장장치란 무엇인가?

## 도입

저장장치가 반드시 컴퓨터 내부에 꽂혀 있어야 할 필요는 없다. 네트워크가 충분히 빠르다면, 멀리 있는 저장장치를 마치 로컬 디스크처럼 사용할 수 있다.

---

## 본문

네트워크 저장장치는 네트워크를 통해 클라이언트들이 데이터를 읽고 쓸 수 있는 저장장치다. 단일 컴퓨터에 연결된 로컬 디스크와 달리, 여러 클라이언트가 동시에 같은 저장소에 접근할 수 있다. 네트워크 속도(기가비트 이더넷, 10GbE 등)가 충분히 빨라지면서 실용적인 고속 데이터 전송이 가능해졌기 때문에 등장했다.

비유하자면, 여러 Node.js 서버 인스턴스가 각자 로컬 디스크를 갖는 대신 공용 네트워크 드라이브에 파일을 올리고 내리는 구조다. 이렇게 하면 어떤 서버 인스턴스가 요청을 처리해도 같은 파일을 볼 수 있다.

---

## 종합

네트워크 저장장치가 없으면 수평 확장(scale out)된 서버 환경에서 파일 공유가 어렵다. 예를 들어 여러 서버에서 사용자 업로드 파일을 읽어야 할 때, 각 서버가 로컬 디스크만 가진다면 어느 서버에 파일이 있는지 추적해야 하는 복잡성이 생긴다. NAS 같은 네트워크 저장장치는 이 문제를 중앙 저장소로 단순화한다.

# [UNVERIFIED] 디스크 인터페이스란 무엇이며, 어떤 종류가 있는가?

## 도입

CPU가 저장장치에 데이터를 읽고 쓰려면 둘 사이를 연결하는 물리적·전기적 규약이 필요하다. 이것이 디스크 인터페이스다. USB로 외장 드라이브를 꽂는 것도, 내부 SSD가 M.2 슬롯에 꽂히는 것도 모두 인터페이스의 차이다.

---

## 본문

디스크 인터페이스는 컴퓨터(마더보드)와 저장장치 사이의 연결 규격이다. 주요 종류:

- **SATA(Serial ATA)**: 현재 가장 보편적인 소비자용 인터페이스. HDD·SSD 모두 지원. 최대 6Gbps.
- **SCSI(Small Computer System Interface)**: 서버·기업용 환경에서 사용하던 고성능 인터페이스. 현재는 SAS(Serial Attached SCSI)로 발전.
- **NVMe(Non-Volatile Memory Express)**: SSD 전용 고속 인터페이스. PCIe 버스를 직접 사용해 SATA보다 4~6배 빠르다.

---

## 종합

인터페이스가 병목이 되면 저장장치 자체가 빠르더라도 속도를 다 낼 수 없다. NVMe SSD를 SATA 포트에 꽂으면 NVMe의 속도를 사용할 수 없는 것처럼, 인터페이스 선택은 저장장치만큼 중요하다. 프론트엔드 개발자 입장에서는 개발 장비의 SSD 인터페이스가 빌드 속도나 `node_modules` 설치 속도에 직접적인 영향을 준다.

---

# [UNVERIFIED] DAS, NAS, SAN의 스토리지 아키텍처를 설명하라.

## 도입

저장장치를 어떻게 서버에 연결하느냐에 따라 세 가지 아키텍처로 나뉜다. DAS·NAS·SAN은 각각 성능·공유 방식·파일 시스템 위치에서 다르게 동작한다.

---

## 본문

**DAS (Direct Attached Storage)**

서버에 저장장치를 직접 연결하는 방식이다. 여러분이 노트북에 외장 SSD를 USB로 연결하면 DAS다. 서버와 저장장치가 물리적으로 가까워야 하고, 속도는 빠르지만 연결 수에 한계가 있다. 서버 A의 DAS에 저장된 파일은 서버 B가 직접 접근할 수 없다.

**NAS (Network Attached Storage)**

저장장치를 네트워크에 연결해 여러 클라이언트가 공유한다. NAS 자체가 파일 시스템을 소유한다는 점이 핵심이다. 서버·클라이언트는 파일 시스템 없이 NAS에게 "이 파일 줘"라고 요청만 한다. 가정용 NAS(Synology 등)가 대표적이다.

**SAN (Storage Area Network)**

네트워크를 사용하지만 NAS와 다르게 파일 시스템은 각 서버가 가진다. SAN은 논리 볼륨(raw block)만 서버에게 제공하고, 서버가 그 위에 파일 시스템을 만들어 사용한다. 볼륨 공유가 안 되기 때문에 한 번에 하나의 서버가 볼륨을 점유한다.

```
파일 시스템 위치 비교

DAS:  [서버 ← 직접연결 ← 저장장치]
      서버가 파일 시스템 소유

NAS:  [서버] ─ 네트워크 ─ [NAS]
      NAS가 파일 시스템 소유, 서버는 파일 요청만

SAN:  [서버] ─ 전용 네트워크 ─ [SAN(raw block)]
      서버가 파일 시스템 소유, 공유 불가
```

---

## 종합

세 아키텍처의 차이는 결국 "파일 시스템을 누가 가지느냐"와 "몇 개의 서버가 동시에 접근할 수 있느냐"로 요약된다. 여러 서버가 같은 파일에 접근해야 한다면 NAS가 단순하다. 고성능이 필요하고 공유가 필요 없다면 DAS가 빠르다. 대규모 데이터센터에서 블록 단위 고성능 I/O가 필요하다면 SAN을 선택한다. AWS S3 같은 오브젝트 스토리지는 이 세 가지와 또 다른 범주지만, 웹 개발에서 가장 자주 마주치는 형태다.
