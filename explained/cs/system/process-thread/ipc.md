# [UNVERIFIED] 프로세스간 통신이 뭔가요?

## 도입

프로세스는 기본적으로 자신의 메모리 공간 안에서만 동작하며, 다른 프로세스의 메모리에 접근할 수 없다. 그런데 여러 프로세스가 협력해야 하는 상황이 생긴다. 이때 프로세스끼리 데이터를 주고받을 수 있는 방법이 IPC다.

---

## 본문

IPC(Inter-Process Communication, 프로세스 간 통신)는 별도의 메모리 공간에서 실행 중인 프로세스들이 데이터를 교환하는 기법의 총칭이다.

프로세스 협력이 필요한 이유:

- **계산 가속화**: 작업을 여러 서브태스크로 나눠 병렬 실행한다. 멀티코어 CPU에서 효과적이다.
- **편의성**: 음악 플레이어와 문서 편집기가 동시에 실행되는 것처럼, 독립된 기능을 병렬로 유지한다.

IPC의 두 가지 주요 모델:

- **공유 메모리 방식(Shared Memory Model)**: 두 프로세스가 공통으로 접근하는 메모리 영역을 구축한다.
- **메시지 전달 방식(Message Passing Model)**: OS가 제공하는 채널을 통해 메시지를 주고받는다.

```
프로세스 A ─── IPC ─── 프로세스 B
(독립 메모리)         (독립 메모리)
        ↑
공유 메모리 또는 메시지 채널
```

**Chrome 비유:** Chrome 브라우저는 메인 프로세스와 각 탭 프로세스가 분리되어 있다. 탭이 죽어도 메인 프로세스는 살아있다. 이 탭 프로세스들 사이에 데이터를 주고받을 때 IPC(Chromium은 Mojo라는 내부 IPC 시스템 사용)가 사용된다.

---

## 종합

Node.js에서 `child_process.fork()`로 자식 프로세스를 만들면 부모-자식 사이에 IPC 채널이 자동으로 열린다. `process.send(data)`와 `process.on('message', fn)`이 이 IPC를 사용하는 것이다. IPC가 없으면 프로세스들은 완전히 격리되어 협력이 불가능하다.

# [UNVERIFIED] 생산자, 소비자 문제가 뭔가요? (Producer–consumer problem)

## 도입

두 프로세스가 데이터를 주고받을 때 생산 속도와 소비 속도가 다르면 문제가 생긴다. 이것이 생산자-소비자 문제다. IPC 설계에서 반드시 마주치는 고전적 문제다.

---

## 본문

생산자(Producer) 프로세스는 데이터를 만들고, 소비자(Consumer) 프로세스는 그 데이터를 소비한다.

문제의 핵심: 생산 속도 ≠ 소비 속도

- 생산자가 너무 빠르면 → 소비자가 처리하지 못한 데이터가 쌓인다
- 소비자가 너무 빠르면 → 소비자가 데이터가 올 때까지 기다린다

**해결 방법: 버퍼 사용**

두 프로세스 사이에 임시 저장 공간(버퍼)을 두어 속도 차이를 흡수한다.

- **무한 버퍼**: 생산자가 무제한 생산 가능. 소비자는 데이터가 없으면 대기. 현실에서 메모리가 무한할 수 없어서 실용적이지 않다.
- **유한 버퍼**: 크기가 고정된 버퍼. 버퍼가 비면 소비자 대기, 버퍼가 가득 차면 생산자 대기. 실용적이지만 양쪽 대기라는 트레이드오프가 있다.

**JS 비유:** Node.js의 `stream.pipe()`가 생산자-소비자 패턴이다. `readable.pipe(writable)`에서 readable(생산자)이 writable(소비자)보다 빠르면 Node.js가 내부 버퍼로 흡수하고, 버퍼가 가득 차면 backpressure를 통해 readable을 일시 정지시킨다. 유한 버퍼 패턴의 실제 구현이다.

---

## 종합

생산자-소비자 문제는 메시지 큐(Redis, RabbitMQ, Kafka), 스트림, 이벤트 큐 등 거의 모든 비동기 데이터 파이프라인의 기반이다. 이 문제를 해결하지 않으면 생산자가 소비자를 압도하거나(OOM), 소비자가 빈 큐를 계속 폴링하는(CPU 낭비) 문제가 발생한다.

# [UNVERIFIED] 공유메모리 방식(Shared Memory Model)에 대해 설명해주세요.

## 도입

IPC의 두 모델 중 첫 번째가 공유 메모리 방식이다. 두 프로세스가 같은 메모리 영역에 직접 접근하여 데이터를 교환한다. 빠르지만 직접 동기화를 책임져야 한다.

---

## 본문

공유 메모리(Shared Memory)는 협력하는 두 프로세스가 공통으로 접근할 수 있는 메모리 영역을 구축하는 IPC 방식이다.

핵심 특징:

- **OS 제약 해제**: 원칙적으로 프로세스는 다른 프로세스의 메모리에 접근할 수 없다. 공유 메모리는 이 제약을 명시적으로 해제한 영역이다.
- **프로세스가 직접 구축**: OS가 자동으로 만들어주지 않는다. 통신하려는 프로세스들이 직접 공유 메모리 영역을 요청하고 구축해야 한다.
- **동기화 책임**: 같은 위치에 동시에 쓰기를 하면 데이터가 오염된다. 이를 방지하는 동기화(뮤텍스, 세마포어 등)를 프로그래머가 구현해야 한다.

**JS 비유:** 브라우저의 `SharedArrayBuffer`가 공유 메모리 IPC의 정확한 예시다. 메인 스레드와 Web Worker가 같은 `SharedArrayBuffer`를 공유하여 복사 없이 데이터를 주고받는다. `Atomics`를 이용해 동시 쓰기 충돌을 방지한다. 프로세스 간이 아니라 스레드 간이지만 개념이 동일하다.

```
프로세스 A ──읽기/쓰기──→ 공유 메모리 영역 ←──읽기/쓰기── 프로세스 B
                         (OS가 매핑)
               ↑ 동기화 필요 (mutex, semaphore 등)
```

---

## 종합

공유 메모리 방식은 메시지 전달 방식보다 빠르다. OS 시스템 콜을 거치지 않고 메모리를 직접 읽고 쓰기 때문이다. 하지만 동시 접근 문제를 프로그래머가 직접 해결해야 한다는 복잡성이 있다. 높은 성능이 필요한 프로세스 간 대용량 데이터 공유에 주로 사용된다.

# [UNVERIFIED] 메시지전달 방식(Message Passing Model)에 대해 설명해주세요.

## 도입

IPC의 두 번째 모델인 메시지 전달 방식은 OS가 통신 채널을 제공한다. 프로세스들이 메모리를 직접 공유하지 않고, 메시지를 보내고 받는 방식으로 통신한다.

---

## 본문

메시지 전달(Message Passing)은 프로세스가 OS가 제공하는 채널(링크)을 통해 메시지를 주고받는 IPC 방식이다.

핵심 특징:

- **OS가 통신 수단 제공**: 공유 메모리처럼 직접 주소 공간을 공유하지 않아도 된다.
- **동기화 내장**: OS가 메시지 큐를 관리하므로 동시 접근 충돌 걱정이 없다.
- **분산 환경에 유리**: 네트워크로 연결된 서로 다른 컴퓨터 간에도 사용 가능하다.

OS가 제공하는 기본 연산: `Send(message)`, `Receive(message)`

메시지 길이:

- **고정 길이**: 시스템 구현이 단순하지만, 프로그래머가 고정 크기에 맞춰 데이터를 자르거나 패딩해야 한다.
- **가변 길이**: 시스템 구현이 복잡하지만, 프로그래머가 크기에 제약 없이 전송할 수 있다.

구현 방식의 종류:

1. 직접 통신 / 간접 통신
2. Blocking / Non-Blocking
3. 버퍼링 방식

**JS 비유:** `Service Worker`와 메인 스레드 사이의 `postMessage()` / `addEventListener('message', fn)`이 메시지 전달 방식 IPC다. `postMessage(data)`가 Send, `onmessage = fn`이 Receive다.

---

## 종합

Node.js `child_process.fork()`가 만드는 IPC 채널도 메시지 전달 방식이다. `process.send({ type: 'result', value: 42 })`로 메시지를 보내고 `process.on('message', fn)`으로 받는다. OS가 채널 관리를 담당하기 때문에 동기화 코드를 직접 짤 필요가 없다.

# [UNVERIFIED] 메시지 전달방식 중 직접 통신에 대해 설명해주세요.

## 도입

메시지 전달 방식에는 중간자 없이 두 프로세스가 직접 연결하는 방식과, 메일함 같은 중간자를 통하는 방식이 있다. 전자가 직접 통신이다.

---

## 본문

직접 통신(Direct Communication)은 두 프로세스가 중간에 아무것도 거치지 않고 직접 1:1로 메시지를 주고받는 방식이다.

대칭성과 비대칭성:

- **대칭(Symmetric)**: 송신 프로세스가 수신 프로세스를 알고, 수신 프로세스도 송신 프로세스를 안다. `Send(P2, msg)` / `Receive(P1, msg)` 형태로 양쪽 모두 상대방을 지정한다.
- **비대칭(Asymmetric)**: 송신 프로세스만 수신 프로세스를 알고, 수신 프로세스는 보낸 프로세스를 미리 모른다. `Send(P2, msg)` / `Receive(any, msg)` — 수신자가 누구에게서 받을지 지정하지 않는다.

```
직접 통신 (대칭)
프로세스 A ──→ 메시지 ──→ 프로세스 B
(B를 지정)             (A를 지정)

직접 통신 (비대칭)
프로세스 A ──→ 메시지 ──→ 프로세스 B
(B를 지정)             (누구든 받음)
```

---

## 종합

직접 통신은 단순하지만 유연하지 않다. 연결이 두 프로세스 사이에 하드코딩되기 때문에, 통신 상대를 바꾸려면 코드를 수정해야 한다. 또한 한 프로세스가 동시에 여러 프로세스와 통신하려면 링크를 여러 개 관리해야 한다. 이 한계를 해결하는 것이 간접 통신의 역할이다.

# [UNVERIFIED] 메시지 전달방식 중 간접 통신에 대해 설명해주세요.

## 도입

직접 통신이 두 프로세스가 1:1로 연결된다면, 간접 통신은 메일함(mailbox)이라는 중간자를 통해 여러 프로세스가 유연하게 통신할 수 있게 한다.

---

## 본문

간접 통신(Indirect Communication)은 프로세스들이 메일함(mailbox 또는 port)을 통해 메시지를 주고받는 방식이다.

핵심 특징:

- **메일함(Mailbox)**: 고유 ID를 가진 메시지 저장소. 프로세스는 메일함 ID를 알면 메시지를 넣거나 꺼낼 수 있다.
- **연결 다중화**: 두 프로세스가 여러 메일함을 통해 여러 독립적인 채널로 통신할 수 있다.
- **M:N 관계**: 하나의 메일함에 여러 프로세스가 연결될 수 있다.

성질:

1. 메일함이 있어야 프로세스들 사이에 연결이 구축된다.
2. 하나의 연결에 두 개 이상의 프로세스가 참여할 수 있다.
3. 각 프로세스 쌍에는 서로 다른 여러 연결(다른 메일함을 통한)이 존재할 수 있다.

**JS 비유:** Node.js의 `EventEmitter`가 간접 통신과 유사하다. `emitter.emit('event', data)`가 메일함에 메시지를 넣는 것이고, `emitter.on('event', fn)`이 메일함에서 꺼내는 것이다. 송신자와 수신자가 서로를 직접 알 필요 없이 이벤트 이름(메일함 ID)만 공유한다.

---

## 종합

간접 통신은 프로세스 간 결합도를 낮춘다. 송신 프로세스는 메일함에만 메시지를 넣고, 어떤 프로세스가 꺼내는지 알 필요가 없다. 마이크로서비스 아키텍처에서 메시지 큐(Kafka, RabbitMQ)를 사용하는 것이 간접 통신의 실무 패턴이다. 생산자가 메시지를 큐에 넣고, 소비자들이 큐에서 꺼낸다.

# [UNVERIFIED] 간접 통신 과정에서 송신자가 보낸 메시지를 받을 때 수신자 프로세스들이 서로 충돌하는 경우 어떻게 해결할 수 있을까요?

## 도입

하나의 메일함에 여러 수신 프로세스가 연결되어 있을 때, 메시지 하나를 누가 가져가야 하는지를 두고 충돌이 발생할 수 있다. 이 레이스 컨디션을 어떻게 해결하는가?

---

## 본문

시나리오: P1이 메시지를 보내고, P2와 P3가 모두 수신자로 연결되어 있다. P1이 메시지를 하나 보내면 P2와 P3 중 누가 받는가? 둘 다 받으려 하면 충돌이 발생한다.

세 가지 해결 방법:

**방법 1**: 하나의 링크는 최대 두 개의 프로세스만 공유하도록 허용한다. P1-P2 링크와 P1-P3 링크를 분리하여 충돌 상황 자체를 방지한다.

**방법 2**: 한 번에 한 프로세스만 `Receive` 연산을 실행하도록 제한한다. 뮤텍스나 세마포어로 `Receive` 접근을 직렬화한다.

**방법 3**: 송신자(P1)가 메시지를 보낼 때 수신자를 미리 선택한다. "이 메시지는 P2에게만"이라고 지정한다.

```
충돌 상황:
P1 → [메일함] ← P2, P3 (동시 수신 시도)

방법 1: 메일함을 P2 전용, P3 전용으로 분리
방법 2: P2, P3 중 하나만 Receive 가능하도록 잠금
방법 3: P1이 P2 또는 P3를 직접 지정
```

---

## 종합

Kafka의 Consumer Group이 방법 1과 유사하다. 각 파티션(링크)은 하나의 Consumer에게만 할당되어 충돌이 없다. 방법 2는 전통적인 큐(Queue)처럼 FIFO로 하나씩 꺼내는 방식이다. 분산 시스템에서 메시지 중복 수신 방지가 중요한 이유가 바로 이 충돌 문제 때문이다.

# [UNVERIFIED] 메시지전달 방식 중 Blocking, Non-Blocking 방식에 대해 설명해주세요.

## 도입

메시지를 보내고 받을 때, 상대방이 준비될 때까지 기다릴지(Blocking) 말지(Non-Blocking)를 선택할 수 있다. 이 선택이 프로그램의 동작 방식에 큰 차이를 만든다.

---

## 본문

**Blocking 모드 (동기)**

- **Blocking Send**: 송신자가 메시지를 보낸 후, 수신자가 메시지를 받을 때까지 대기한다.
- **Blocking Receive**: 수신자가 이용 가능한 메시지가 있을 때까지 대기한다.

두 쪽 모두 블로킹이면 "랑데뷰(Rendezvous)"가 발생한다 — 송신자와 수신자가 정확히 같은 시점에 만날 때까지 둘 다 기다린다.

**Non-Blocking 모드 (비동기)**

- **Non-Blocking Send**: 송신자가 메시지를 보낸 뒤 바로 다른 작업으로 넘어간다. 수신 여부를 기다리지 않는다.
- **Non-Blocking Receive**: 수신자가 메시지를 확인할 때 메시지가 있으면 받고, 없으면 `null`을 받는다. 블로킹 없이 즉시 반환한다.

```
Blocking:
  Send → [대기] → 수신 확인 → 다음 작업
  Receive → [대기] → 메시지 수신 → 처리

Non-Blocking:
  Send → 즉시 다음 작업
  Receive → 메시지 있으면 받음, 없으면 null → 다음 작업
```

**JS 비유:** `await fetch(url)` (Blocking Receive 유사, 응답 올 때까지 대기) vs `fetch(url).then(fn)` (Non-Blocking, 보내고 바로 다음 코드 실행).

---

## 종합

Node.js는 Non-Blocking I/O를 기본으로 사용한다. `fs.readFile()`은 Non-Blocking Send처럼 OS에 요청만 하고 바로 다음 코드로 넘어간다. 완료되면 콜백/Promise로 받는다. Blocking 방식이라면 파일을 읽는 동안 다른 요청을 전혀 처리할 수 없어서 Node.js의 높은 동시성이 불가능해진다.

# [UNVERIFIED] 메시지전달 방식 중 버퍼링 방식에 대해 설명해주세요.

## 도입

메시지 전달 방식에서 메시지가 이동하는 동안 임시로 보관되는 큐(Queue)가 있다. 이 큐의 용량을 어떻게 설정하느냐에 따라 송신자의 대기 동작이 달라진다.

---

## 본문

통신 프로세스 간 교환되는 메시지는 임시 큐(Queue)에 존재한다. 큐 구현 방식 3가지:

**1. 용량 0 (Zero Capacity)**

큐에 메시지를 저장할 공간이 없다. 송신자는 수신자가 메시지를 직접 받을 때까지 기다려야 한다. Blocking Send와 동일한 효과다.

**2. 제한된 용량 (Bounded Capacity)**

큐가 N개의 메시지를 저장할 수 있다. 큐가 가득 찰 때까지 송신자는 대기 없이 전송한다. 큐가 가득 차면 공간이 생길 때까지 송신자가 대기한다.

**3. 무한 용량 (Unbounded Capacity)**

큐 크기에 제한이 없다. 송신자는 절대 대기하지 않는다. 하지만 메시지가 무한정 쌓일 수 있어 메모리 문제가 발생할 수 있다.

```
큐 용량에 따른 송신자 동작
─────────────────────────────────────────────
용량 0       수신자 준비될 때까지 무조건 대기
제한된 용량  큐 가득 찰 때만 대기 (보통 가장 실용적)
무한 용량    대기 없음 (메모리 무한 소비 위험)
─────────────────────────────────────────────
```

**JS 비유:** Node.js의 `stream.write()`가 제한된 용량 버퍼링이다. 내부 버퍼(highWaterMark)가 가득 차면 `write()`가 `false`를 반환하고(backpressure), 소비자가 데이터를 처리한 뒤 `drain` 이벤트를 발생시키면 송신을 재개한다.

---

## 종합

제한된 용량 버퍼링이 실무에서 가장 많이 사용된다. Kafka의 파티션 오프셋, Node.js 스트림의 highWaterMark, Redis의 메시지 큐 최대 길이가 모두 제한된 용량 개념이다. 무한 용량을 사용하면 서버 OOM(Out of Memory)이 발생하고, 용량 0이면 처리량이 급격히 줄어든다. 적절한 버퍼 크기 설정이 시스템 튜닝의 핵심이다.

# [UNVERIFIED] 공유메모리 방식과 메시지전달 방식의 장단점이 어떻게 되나요?

## 도입

두 IPC 모델은 각각 다른 트레이드오프를 가진다. 무엇을 우선시하느냐 — 구현 단순성, 처리 속도, 분산 환경 지원 — 에 따라 어떤 모델을 선택할지 달라진다.

---

## 본문

세 가지 기준으로 비교:

**1. 구현 난이도: 공유 메모리 > 메시지 전달 (공유 메모리가 어렵다)**

- 공유 메모리는 같은 위치에 동시 쓰기를 방지하는 동기화 코드(뮤텍스, 세마포어)를 직접 구현해야 한다. 잘못 구현하면 데이터 경합(race condition), 데드락이 발생한다.
- 메시지 전달은 OS가 큐를 관리하므로 충돌 회피를 직접 구현할 필요가 없다.

**2. 속도: 공유 메모리 > 메시지 전달 (공유 메모리가 빠르다)**

- 메시지 전달은 매번 시스템 콜을 거친다. 메시지 큐는 커널 위에 있어서 데이터가 커널을 통과한다.
- 공유 메모리는 일단 설정되면 시스템 콜 없이 메모리를 직접 읽고 쓰므로 훨씬 빠르다.

**3. 분산 환경: 공유 메모리 < 메시지 전달 (메시지 전달이 낫다)**

- 공유 메모리는 여러 서버가 물리적으로 같은 메모리를 공유하기 어렵다. 네트워크를 통한 공유 메모리는 동기화 비용이 급증한다.
- 메시지 전달은 네트워크를 링크로 사용하면 되므로 다른 서버의 프로세스와도 같은 방식으로 통신할 수 있다.

```
비교 요약
────────────────────────────────────────────
              공유 메모리    메시지 전달
구현 난이도   어렵          쉬움
속도          빠름          느림(시스템 콜)
분산 환경     어렵          용이
────────────────────────────────────────────
```

---

## 종합

단일 서버에서 고성능이 필요한 프로세스 간 통신이라면 공유 메모리(`SharedArrayBuffer`, POSIX shm)가 적합하다. 마이크로서비스처럼 여러 서버에 걸쳐 통신해야 한다면 메시지 전달 방식(Kafka, RabbitMQ, gRPC)이 표준이 됐다. Node.js `cluster` 모듈이 내부적으로 Worker 간 IPC에 메시지 전달을 사용하는 것은 단순성과 안전성을 우선한 선택이다.
