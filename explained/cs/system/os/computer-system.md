# [UNVERIFIED] 시스템 소프트웨어란 무엇인가?

## 도입

소프트웨어는 하드웨어와 사람을 연결하는 계층이다. 그런데 모든 소프트웨어가 같은 역할을 하는 것은 아니다. "컴퓨터 자체가 작동하게 하는" 소프트웨어와 "사람이 특정 작업을 하기 위해 쓰는" 소프트웨어로 나뉜다. 전자가 시스템 소프트웨어다.

---

## 본문

시스템 소프트웨어는 컴퓨터 하드웨어를 관리하고, 그 위에서 다른 소프트웨어가 실행될 수 있는 환경을 만드는 소프트웨어다.

예:

- **운영체제(OS)**: Windows, macOS, Linux. CPU·메모리·파일 시스템을 관리하고 프로세스를 스케줄링한다.
- **부트로더(Bootloader)**: 컴퓨터를 켰을 때 OS를 디스크에서 RAM으로 로드하는 소프트웨어.
- **셸(Shell)**: 사용자가 OS에 명령을 내리는 인터페이스. `bash`, `zsh`, PowerShell 등.

---

## 종합

시스템 소프트웨어 없이는 Chrome 같은 응용 소프트웨어가 실행될 수 없다. OS가 CPU 시간을 배분하고, 메모리 공간을 할당하고, 파일 시스템 접근을 중재하기 때문에 애플리케이션이 하드웨어 세부사항을 몰라도 실행될 수 있다.

---

---

# [UNVERIFIED] 응용 소프트웨어란 무엇인가?

## 도입

시스템 소프트웨어가 컴퓨터 자체를 동작시키는 계층이라면, 응용 소프트웨어는 그 위에서 사람이 특정 목적을 달성하기 위해 사용하는 소프트웨어다.

---

## 본문

응용 소프트웨어는 컴퓨터 하드웨어의 직접 관리와 무관하며, 사용자가 특정 작업을 수행하기 위해 사용하는 프로그램이다.

예:

- Chrome 브라우저: 웹 페이지를 렌더링하기 위한 응용 소프트웨어
- VS Code: 코드 편집을 위한 응용 소프트웨어
- Excel: 스프레드시트 작업을 위한 응용 소프트웨어

응용 소프트웨어는 OS가 제공하는 API(시스템 콜)를 통해 하드웨어에 간접 접근한다. 직접 CPU에 명령을 내리거나 메모리 주소를 지정하지 않는다.

---

## 종합

Node.js 애플리케이션도 응용 소프트웨어다. `fs.readFile()`이 파일을 읽는 것처럼 보이지만, 실제로는 OS의 시스템 콜을 통해 OS가 대신 파일을 읽어주는 것이다. 응용 소프트웨어와 하드웨어 사이에는 항상 OS(시스템 소프트웨어)가 중간자로 있다.

---

---

# [UNVERIFIED] 컴퓨터의 하드웨어 구성 요소는 무엇인가?

## 도입

소프트웨어가 실행되는 물리적 기반이 하드웨어다. 컴퓨터의 하드웨어 구성 요소를 파악하면 소프트웨어가 어떤 자원을 사용하는지 이해할 수 있다.

---

## 본문

컴퓨터 하드웨어 구성 요소:

**외부에서 보이는 것:**
- 디스플레이: 출력 장치
- 키보드, 마우스: 입력 장치

**본체 내부:**
- CPU: 연산과 제어를 담당하는 핵심 부품
- 주기억장치(RAM): 실행 중인 프로그램과 데이터를 임시 저장
- 보조기억장치(HDD, SSD): 파일과 프로그램을 영구 저장
- Power Supply(PSU): 전력 공급

---

## 종합

Node.js 프로세스 하나를 실행하면 CPU(연산), RAM(메모리), SSD(파일 읽기), 디스플레이(터미널 출력), 키보드(입력)가 모두 협력한다. 이 하드웨어들이 시스템 버스를 통해 연결되고, OS가 그 위에서 각 자원을 중재한다.

---

---

# [UNVERIFIED] 컴퓨터의 가장 중요한 3요소는 무엇이며, 컴퓨터의 기본 구조는 어떻게 되는가?

## 도입

수많은 하드웨어 부품 중에서 컴퓨터 동작의 핵심을 이루는 세 요소가 있다. 이 셋이 시스템 버스로 연결되어 컴퓨터의 기본 구조를 형성한다.

---

## 본문

컴퓨터의 3대 핵심 요소:

- **CPU**: 프로그램의 명령어를 해석하고 실행한다. 연산과 제어를 담당한다.
- **메모리(주기억장치)**: 실행 중인 프로그램과 데이터를 임시 저장한다. CPU가 직접 읽고 쓰는 공간이다.
- **I/O Device**: 사용자와 컴퓨터, 또는 컴퓨터와 외부 세계가 데이터를 주고받는 장치. 키보드·마우스(입력), 모니터·프린터(출력), 네트워크 카드(입출력 겸용).

이 세 요소는 **시스템 버스**로 연결된다. 버스는 데이터가 이동하는 공통 경로로, 세 종류로 나뉜다 — 데이터 버스, 주소 버스, 제어 버스.

```
컴퓨터 기본 구조
┌───────┐    시스템 버스     ┌────────┐
│  CPU  │ ←────────────────→ │ 메모리  │
└───────┘                    └────────┘
    ↕ (시스템 버스)
┌──────────┐
│ I/O Device│ (키보드, 모니터, 디스크 등)
└──────────┘
```

---

## 종합

Chrome 브라우저에서 URL을 입력하는 순간 키보드(I/O Device)가 신호를 보내고, CPU가 이를 처리해 RAM에 데이터를 올리고, 네트워크 카드(I/O Device)를 통해 외부 서버에 요청하고, 응답 데이터가 다시 RAM에 올라온 뒤 GPU·모니터(I/O Device)를 통해 화면에 출력된다. 이 모든 과정이 시스템 버스로 연결된 세 요소의 협업이다.

---

# [UNVERIFIED] CPU란 무엇인가?

## 도입

CPU는 컴퓨터의 두뇌다. 모든 연산과 제어가 CPU를 통해 이루어진다. 프로그램의 명령어를 읽어서 실행하는 것이 CPU의 본질적인 역할이다.

---

## 본문

CPU(Central Processing Unit)는 프로그램의 실행과 데이터 처리를 담당하는 부품이다. 주로 입출력 장치로부터 데이터를 받아서 처리한 후 결과를 기억장치에 저장하는 일을 한다.

CPU 내부는 크게 두 부분으로 구성된다:

- **ALU(Arithmetic Logic Unit)**: 산술 연산(덧셈·뺄셈 등)과 논리 연산(AND·OR·NOT 등)을 실행하는 회로.
- **CU(Control Unit)**: 메모리에서 명령어를 가져와(fetch) 해독하고(decode) ALU에 실행 신호를 보내는 제어 장치.

---

## 종합

`1 + 2`라는 JS 표현식을 V8이 실행할 때, 실제로는 CPU의 ALU가 두 정수를 더하는 연산을 수행한다. V8은 JS 코드를 기계어로 번역하고, 그 기계어를 CPU가 실행한다. CPU 속도(GHz)는 초당 몇 번의 연산 사이클을 수행하는지를 나타내며, CPU 성능이 곧 단순 연산 처리 속도의 상한선이다.

---

---

# [UNVERIFIED] I/O Device란 무엇인가?

## 도입

CPU와 메모리가 컴퓨터 내부에서 데이터를 처리하는 장치라면, I/O Device는 컴퓨터와 외부 세계(사람, 네트워크, 다른 장치)를 연결하는 장치다.

---

## 본문

I/O Device(Input/Output Device)는 사용자와 컴퓨터의 상호작용, 또는 컴퓨터와 외부 환경 간의 데이터 교환을 위한 장치다.

- **Input(입력) 장치**: 사용자 또는 외부에서 컴퓨터로 데이터를 보내는 장치. 키보드, 마우스, 터치스크린, 마이크, 카메라, 네트워크 카드.
- **Output(출력) 장치**: 컴퓨터에서 사용자 또는 외부로 데이터를 보내는 장치. 모니터, 프린터, 스피커, 네트워크 카드.

하나의 장치가 입력과 출력을 모두 담당하기도 한다. 네트워크 카드는 데이터를 받기도(입력) 하고 보내기도(출력) 한다.

---

## 종합

`console.log("hello")`가 터미널에 출력되는 것은 Node.js → OS → 모니터(출력 I/O Device) 경로를 거친다. `fs.readFile()`은 SSD나 HDD(입력 I/O Device로 작동)에서 데이터를 읽는다. I/O Device 없이는 프로그램의 결과를 사람이 볼 수 없고, 사람의 입력이 프로그램에 전달될 수 없다.

---

---

# [UNVERIFIED] 컴퓨터가 다루는 정보의 종류에는 무엇이 있는가?

## 도입

컴퓨터가 처리하는 모든 정보는 0과 1의 이진수지만, 그 정보가 무엇을 나타내느냐에 따라 두 가지로 분류할 수 있다.

---

## 본문

컴퓨터가 다루는 정보의 종류:

- **Program Code**: 컴퓨터에게 무엇을 해야 하는지 알려주는 명령어들의 집합. 고급언어 코드, 어셈블리 코드, 기계어 코드로 나뉜다.
- **데이터(이진수로 표현된)**: 처리의 대상이 되는 값. 숫자, 문자, 이미지 등 모든 것이 궁극적으로 0과 1로 표현된다.

---

## 종합

`const x = 42`에서 `42`는 데이터고, `const x =`라는 선언 자체는 Program Code다. CPU가 이 코드를 실행할 때 메모리에서 코드도 읽고 데이터도 읽는데, 내부적으로는 모두 이진수다. 코드와 데이터를 같은 메모리에 올려 저장하는 것이 폰노이만 아키텍처의 특징이다.

---

---

# [UNVERIFIED] Program Code의 종류는 무엇인가?

## 도입

소스 코드를 작성하면 그대로 CPU가 실행하는 것이 아니다. 사람이 읽기 쉬운 고급언어에서 CPU가 읽을 수 있는 기계어까지 여러 단계의 변환이 있다. 이 과정에 등장하는 세 가지 코드 형태가 있다.

---

## 본문

Program Code의 세 종류:

- **고급언어 코드(High-level Language)**: JavaScript, Python, C 같은 언어로 작성된 코드. 사람이 이해하기 쉽지만 CPU는 직접 실행할 수 없다.
- **어셈블리 코드(Assembly)**: 기계어와 1:1 대응되는 기호화된 언어. 고급언어와 기계어의 중간. `MOV AX, 1` 같은 형식이다.
- **기계어 코드(Machine Code)**: CPU가 직접 실행하는 이진수(0과 1) 코드. CPU 아키텍처(x86, ARM 등)마다 명령어 집합이 다르다.

변환 경로:

```
고급언어(JS) → [컴파일러/인터프리터] → 어셈블리 → [어셈블러] → 기계어
```

V8의 경우: JS → 파싱 → AST → 바이트코드 → JIT 컴파일 → 기계어 순서로 변환된다.

---

## 종합

`node app.js`를 실행할 때 V8은 `app.js`를 기계어로 변환하여 CPU에게 전달한다. 이 과정이 없으면 CPU는 JS 코드를 이해할 수 없다. ARM 기반 M1 맥과 Intel x86 기반 PC에서 같은 JS가 실행되는 것은 V8이 각 아키텍처에 맞는 기계어로 따로 컴파일하기 때문이다.

---

---

# [UNVERIFIED] 컴파일러란 무엇인가?

## 도입

고급언어로 작성된 코드를 CPU가 실행할 수 있는 형태로 변환하는 소프트웨어가 컴파일러다. 한 번 변환하면 변환된 파일을 바로 실행할 수 있다는 것이 인터프리터와 구별되는 특징이다.

---

## 본문

컴파일러(Compiler)는 고급언어로 작성된 소스코드 전체를 기계어 또는 어셈블리어로 변환하는 소프트웨어다. 컴파일이 완료되면 실행 파일이 생성되고, 이후에는 원본 소스코드 없이 실행 파일만으로 프로그램을 실행할 수 있다.

컴파일러는 크게 두 종류로 분류된다:

- **네이티브 컴파일러(Native Compiler)**: 컴파일러가 실행되는 컴퓨터와 컴파일 결과물이 실행되는 컴퓨터가 같다. 내 맥북에서 컴파일하고 내 맥북에서 실행.
- **크로스 컴파일러(Cross Compiler)**: 컴파일러가 실행되는 플랫폼과 결과물이 실행될 플랫폼이 다르다. Windows에서 안드로이드 앱을 빌드할 때, 또는 고성능 개발 PC에서 임베디드 기기용 코드를 컴파일할 때 사용한다.

---

## 종합

TypeScript 컴파일러(`tsc`)는 `.ts` 파일을 `.js`로 변환한다. 이것도 넓은 의미에서 컴파일러다. 하지만 여기서는 기계어로 변환하는 것이 아니라 JS라는 다른 고급언어로 변환한다는 점이 다르다. 이를 트랜스파일(transpile)이라고도 부른다. 크로스 컴파일의 개념은 React Native 앱 빌드에서도 등장한다 — Mac에서 빌드하여 iOS·Android 양쪽에서 실행되는 바이너리를 만드는 것이 크로스 컴파일의 일종이다.

---

# [UNVERIFIED] 고급언어로 작성된 프로그램을 실행하는 방법은 무엇인가?

## 도입

고급언어로 작성된 코드를 CPU가 실행하려면 기계어로 변환해야 한다. 이 변환을 언제, 어떻게 하느냐에 따라 두 가지 방법이 있다.

---

## 본문

- **컴파일러 방식**: 실행 전에 소스코드 전체를 기계어로 미리 변환한다. 변환된 실행 파일을 반복 실행할 수 있다. C, Rust, Go가 대표적.
- **인터프리터 방식**: 실행 시점에 코드를 한 줄씩 읽고 바로 실행한다. 사전 컴파일 없이 소스코드를 즉시 실행할 수 있다. Python(CPython), 초기 JS 엔진이 대표적.

현대 JS 엔진(V8)은 두 방식을 혼합한다 — 처음에는 인터프리터로 실행하다가, 자주 실행되는 코드는 JIT(Just-In-Time) 컴파일하여 기계어로 변환해 성능을 높인다.

---

## 종합

`node app.js`를 처음 실행할 때 V8이 인터프리터로 파싱·실행하고, 이후 핫 패스(자주 호출되는 함수)는 JIT 컴파일한다. 이 혼합 전략 덕분에 시작은 빠르고(인터프리터), 장기 실행 성능도 좋다(JIT). 순수 인터프리터라면 반복 실행마다 파싱 비용이 들고, 순수 AOT 컴파일이라면 빌드 시간이 길어진다.

# [UNVERIFIED] 인터프리터란 무엇인가?

## 도입

컴파일러가 코드 전체를 미리 변환하는 것과 달리, 인터프리터는 코드를 한 줄씩 읽으면서 즉시 실행한다. 실행 전에 변환 단계가 없어서 즉시 실행이 가능하다.

---

## 본문

인터프리터(Interpreter)는 고급언어 코드를 한 줄씩 읽고 바로 실행하는 소프트웨어다. 변환된 실행 파일을 생성하지 않는다.

비유: 동시 통역사. 연설자(소스코드)가 말하는 것을 즉시 다른 언어(기계어)로 변환해 청중(CPU)에게 전달한다. 컴파일러는 문서 전체를 미리 번역하는 번역가다.

---

## 종합

브라우저에서 `<script>` 태그 안의 JS가 실행될 때, 초창기 브라우저들은 순수 인터프리터 방식을 썼다. 지금은 V8 같은 JIT 컴파일러가 인터프리터와 컴파일러를 혼합해 사용한다. `node -e "console.log(1+1)"`처럼 짧은 코드를 바로 실행할 수 있는 것이 인터프리터의 즉시 실행 특성을 활용한 예다.

---

---

# [UNVERIFIED] 인터프리터와 컴파일러의 장단점은 무엇인가?

## 도입

인터프리터와 컴파일러는 서로 다른 트레이드오프를 가진다. 어느 쪽이 무조건 낫다고 할 수 없고, 용도에 따라 선택이 달라진다.

---

## 본문

비교:

```
비교 항목        컴파일러                 인터프리터
──────────────────────────────────────────────────
전체 실행 속도   빠름 (미리 최적화됨)     느림 (매번 해석)
시작 속도        느림 (컴파일 시간 소요)  빠름 (즉시 실행)
개발 편의성      불편 (수정→컴파일→실행)  편리 (수정→실행)
플랫폼 이식성    낮음 (기계어는 CPU 종속) 높음 (인터프리터만 있으면 됨)
```

컴파일러의 우위: 전체 코드를 미리 변환하면서 최적화를 수행하기 때문에 실행 파일 자체의 성능이 좋다.

인터프리터의 우위: 긴 코드를 컴파일하는 동안 아무것도 못 하는 컴파일러와 달리, 인터프리터는 첫 줄부터 즉시 실행을 시작할 수 있다.

---

## 종합

TypeScript 프로젝트에서 `ts-node`로 바로 실행하면 인터프리터 방식이고, `tsc`로 빌드 후 `node dist/index.js`로 실행하면 컴파일 방식이다. 개발 중에는 빠른 피드백을 위해 인터프리터 방식이 편리하고, 프로덕션에서는 성능을 위해 미리 컴파일된 형태를 사용하는 것이 일반적인 패턴이다.

# [UNVERIFIED] 고급언어에서 기계어로 번역되는 과정은 어떻게 되는가?

## 도입

고급언어 코드 한 줄이 실제로 기계어로 변환될 때, 어셈블리어라는 중간 단계를 거친다. 그리고 기계어는 연산코드와 주소로 구성된 이진수 명령어다.

---

## 본문

변환 과정 예시 — "메모리 X의 값과 Y의 값을 더해서 Z에 저장하라":

**고급언어:** `z = x + y`

**어셈블리 코드:**
1. 메모리 주소 X의 값을 레지스터 A에 로드
2. 메모리 주소 Y의 값과 레지스터 A의 값을 더함
3. 레지스터 A의 값을 메모리 주소 Z에 저장

**기계어(8bit 예시, 앞 3비트=연산코드, 뒤 5비트=오퍼랜드):**

```
연산코드(3비트)   오퍼랜드(5비트)
───────────────────────────────
001 (LOAD)        00001 (주소 X)
010 (ADD)         00010 (주소 Y)
011 (STORE)       00011 (주소 Z)
```

연산코드 3비트 → 2³ = 8가지 연산 가능. 오퍼랜드 5비트 → 2⁵ = 32개 주소 지정 가능. CPU마다 이 필드 구성이 다르다.

---

## 종합

`let z = x + y` 한 줄이 CPU 수준에서는 레지스터 로드 → ALU 덧셈 → 레지스터 저장이라는 세 명령어로 펼쳐진다. V8이 JIT 컴파일할 때 이 변환을 수행하고, 변환된 기계어가 직접 CPU 위에서 실행된다. 컴파일러를 공부하는 이유 중 하나가 "내가 쓰는 JS 코드가 CPU에서 어떻게 동작하는지"를 이해하기 위해서다.

---

# [UNVERIFIED] 시스템 버스는 무엇이며 하위 버스는 어떻게 구성되는가?

## 도입

CPU, 메모리, I/O Device가 통신하려면 데이터가 이동하는 경로가 필요하다. 이 공통 통신 경로가 시스템 버스다. 시스템 버스는 역할에 따라 세 종류의 하위 버스로 나뉜다.

---

## 본문

**시스템 버스(System Bus)**: CPU, 메모리, I/O Device가 공유하는 통신 경로.

세 하위 버스:

- **Data Bus(데이터 버스)**: 실제 데이터를 전달하는 버스. 양방향(읽기·쓰기 모두 가능).
- **Address Bus(주소 버스)**: 데이터가 있는(또는 저장할) 위치의 주소를 전달하는 버스. 단방향(CPU → 메모리/I/O). 메모리 주소나 I/O 디바이스 포트 번호가 이를 통해 전달된다.
- **Control Bus(제어 버스)**: 읽기/쓰기 신호, 인터럽트 신호 같은 제어 명령을 전달하는 버스. 양방향.

```
CPU ←──── 데이터 버스(양방향) ────→ 메모리 / I/O
CPU ─────→ 주소 버스(단방향) ─────→ 메모리 / I/O
CPU ←──── 제어 버스(양방향) ────→ 메모리 / I/O
```

각 버스의 비트 수는 한 번에 전달할 수 있는 정보의 크기를 결정한다. 주소 버스가 16비트면 2¹⁶ = 64K개의 주소를 지정할 수 있고, 이는 접속 가능한 메모리의 최대 용량이 64KB임을 의미한다.

---

## 종합

버스 비트 수 제한이 역사적으로 중요한 병목이었다. 32비트 주소 버스는 4GB 이상의 RAM을 지정할 수 없어서, 32비트 OS가 4GB 이상의 메모리를 못 쓰는 제약이 생겼다. 64비트로 전환하면서 이 제약이 해소됐다. 개발자가 "32비트 vs 64비트"를 구분하는 이유 중 하나가 바로 이 주소 버스 비트 수의 차이다.

---

---

# [UNVERIFIED] CPU가 시스템 버스와 통신하는 과정은?

## 도입

시스템 버스가 세 종류로 나뉘고, 각각 단방향/양방향이 다르다는 것을 이해하면 CPU가 실제로 어떻게 메모리와 통신하는지 알 수 있다.

---

## 본문

버스 방향성:

- **주소 버스**: 단방향(CPU → 메모리/I/O). CPU만 주소를 지정한다. 메모리나 I/O가 CPU에게 주소를 보낼 필요가 없다.
- **데이터 버스**: 양방향. 읽기 때는 메모리 → CPU, 쓰기 때는 CPU → 메모리.
- **제어 버스**: 양방향. CPU가 제어 신호를 보내고, 장치가 상태 신호를 돌려보낸다.

비트 수의 의미:

- 주소 버스 비트 수 → 지정 가능한 메모리 주소 수 → 최대 메모리 용량 결정
- 데이터 버스 비트 수 → 한 번에 전송 가능한 데이터 크기. 32비트 CPU는 한 번에 32비트씩 전송.

---

## 종합

개발자가 직접 버스를 제어하는 경우는 없지만, 버스 비트 수는 시스템의 하드웨어 한계를 이해하는 데 중요하다. `ArrayBuffer`, `SharedArrayBuffer` 같은 웹 API가 바이트 단위로 메모리를 다루는 것도 결국 이 버스 수준의 메모리 접근 개념과 연결된다.

---

---

# [UNVERIFIED] CPU가 Memory에 데이터를 쓸 때의 과정은?

## 도입

CPU가 메모리에 데이터를 저장하는 과정은 세 버스가 동시에 협력하는 작업이다. 주소, 데이터, 제어 신호가 동시에 전달된다.

---

## 본문

CPU가 메모리에 쓸 때 순서:

1. 저장할 주소값을 **주소 버스**로 전송
2. 저장할 데이터를 **데이터 버스**로 전송 (1과 동시에)
3. **제어 버스**로 쓰기 신호(Write signal = 1) 활성화 (동시에)
4. 메모리는 쓰기가 완료될 때까지 쓰기 신호가 유지됨

```
CPU ───[주소값]──→ 주소 버스 ──→ 메모리
CPU ───[데이터]──→ 데이터 버스 ─→ 메모리  (동시 전송)
CPU ───[Write=1]─→ 제어 버스 ──→ 메모리
                           (쓰기 완료 시까지 유지)
```

**Memory Write Time**: CPU가 주소와 데이터를 버스에 실은 순간부터 메모리에 쓰기가 완료될 때까지의 시간.

---

## 종합

`const x = 42`를 실행할 때 V8이 내부적으로 이 과정을 수행한다. 물론 V8이 직접 버스를 제어하는 것이 아니라 OS와 CPU가 처리하지만, 결국 메모리 주소에 값을 쓰는 물리적 과정은 이 흐름이다. 쓰기 신호가 완료될 때까지 유지된다는 것은 쓰기가 원자적(atomic)으로 완성되어야 함을 보장하는 메커니즘이다.

---

---

# [UNVERIFIED] CPU가 Memory로부터 데이터를 읽을 때의 과정은?

## 도입

쓰기와 달리 읽기는 CPU가 먼저 주소를 보내고, 메모리가 데이터를 돌려보내는 데 일정 시간이 걸린다. 이 지연이 메모리 읽기 시간이다.

---

## 본문

CPU가 메모리에서 읽을 때 순서:

1. 읽을 데이터가 저장된 주소값을 **주소 버스**로 전송
2. **제어 버스**로 읽기 신호(Read signal = 1) 활성화 (동시에)
3. **일정 지연 시간 경과** (메모리가 해당 주소의 데이터를 꺼내는 시간)
4. 메모리가 **데이터 버스**로 데이터를 전송 → CPU가 수신

```
CPU ───[주소값]──→ 주소 버스 ──→ 메모리
CPU ───[Read=1]──→ 제어 버스 ──→ 메모리
                    (지연)
메모리 ←[데이터]── 데이터 버스 ←─ 메모리
```

**Memory Read Time**: CPU가 주소값을 버스에 실은 순간부터 데이터를 받을 때까지의 시간.

지연이 발생하는 이유: 메모리 칩이 해당 주소의 셀에서 데이터를 읽어내는 물리적 과정에 시간이 필요하다. 이 지연이 CPU 클럭 속도보다 훨씬 느려서 캐시메모리가 필요한 이유가 된다.

---

## 종합

읽기 지연이 누적되면 CPU는 데이터를 기다리며 아무 연산도 하지 못하는 "stall" 상태에 빠진다. 캐시메모리가 이 대기 시간을 줄이는 핵심 장치다. `arr[i]`를 루프에서 반복 접근할 때 캐시에 올라있으면 이 지연이 거의 없고, 캐시 미스가 나면 이 전체 과정이 다시 일어난다.

---

---

# [UNVERIFIED] CPU와 I/O Device가 통신하는 과정은?

## 도입

CPU가 메모리와 통신할 때보다 I/O Device와 통신할 때는 한 단계가 더 들어간다. 바로 I/O Controller다. 이 중간자가 필요한 이유와 역할을 이해하면 OS의 I/O 관리가 왜 복잡한지 알 수 있다.

---

## 본문

CPU가 I/O Device와 직접 연결될 수 없는 이유:

- I/O Device 종류가 매우 다양해서 각 장치마다 제어 방법이 다르다
- I/O Device의 데이터 전송 속도가 CPU보다 훨씬 느리다
- 데이터 형식이 CPU와 다르다 (예: 키보드는 문자 코드, 프린터는 도트 패턴)

그래서 시스템 버스와 I/O Device 사이에 **I/O Controller**가 들어간다.

I/O Controller 내부 구조:

- **Status Register**: 현재 I/O 장치의 상태를 나타내는 비트를 저장한다. (예: In Ready 비트, Out Ready 비트)
- **Data Register**: CPU와 I/O Device 사이를 오가는 데이터를 임시 저장한다.

```
CPU ── 시스템 버스 ── I/O Controller ── I/O Device
                      ├─ Status Register
                      └─ Data Register
```

---

## 종합

Node.js에서 `process.stdin`으로 키보드 입력을 받을 때, 물리적으로는 키보드 → I/O Controller(키보드 컨트롤러) → 시스템 버스 → OS → Node.js 경로를 거친다. I/O Controller가 없으면 모든 키보드 입력을 CPU가 직접 폴링해야 하는데, 그러면 CPU가 다른 작업을 전혀 못 하게 된다. 인터럽트 기반 I/O는 이 문제를 해결한다.

---

# [UNVERIFIED] 프로그램이 실행되는 과정을 간단히 설명하면?

## 도입

프로그램 실행의 본질은 CPU가 메모리에서 명령어를 읽어서 실행하는 반복 사이클이다.

---

## 본문

프로그램 실행 과정:

1. 프로그램 코드(실행 파일)가 디스크에 있다
2. 실행 명령이 들어오면 OS가 디스크에서 RAM으로 코드를 복사한다
3. CPU가 주소 버스로 다음 명령어 주소를 메모리에 보낸다
4. 메모리가 데이터 버스로 명령어를 CPU에 돌려보낸다
5. CPU가 명령어를 해석하고 실행한다
6. 실행 결과 데이터를 메모리에 저장한다
7. 3~6을 프로그램이 끝날 때까지 반복한다

```
디스크[app.js] ──OS 로드──→ RAM[명령어]
                               ↕ (주소 버스, 데이터 버스)
                            CPU[명령어 실행]
```

---

## 종합

`node app.js`를 입력하면 OS가 Node.js 실행 파일을 메모리에 올리고, Node.js가 `app.js`를 파싱해 V8에 넘기고, V8이 JS를 기계어로 변환해 CPU에 실행시킨다. 이 전체 과정이 "CPU가 메모리에서 명령어를 읽어 실행하는 사이클"의 다층적 구현이다.

# [UNVERIFIED] 폰노이만 아키텍처란 무엇인가?

## 도입

현대 컴퓨터의 대다수가 따르는 설계 원칙이 폰노이만 아키텍처다. 핵심은 "프로그램 코드와 데이터를 같은 메모리에 저장한다"는 것으로, 이 단순한 원칙이 범용 컴퓨터를 가능하게 했다.

---

## 본문

폰노이만 아키텍처의 구성:

- **Input Device**: 사용자 입력을 받는 장치
- **CPU**: 연산 단위인 ALU와 제어 단위인 CU로 구성
  - CU: 메모리에서 명령어를 읽어 ALU에 전달
  - ALU: 실제 연산 수행
- **Output Device**: 결과를 출력하는 장치
- **Memory**: 프로그램 코드와 데이터를 같은 공간에 저장

폰노이만 vs 하버드 아키텍처:

- **폰노이만**: 코드와 데이터가 같은 메모리 공간에 있다. 구현이 단순하고 비용이 낮다. 범용 컴퓨터에 사용.
- **하버드**: 코드용 메모리와 데이터용 메모리가 분리되어 있다. 구현이 복잡하고 비용이 높다. 고속 I/O가 필수인 DSP(디지털 신호 처리) 전용 시스템에 제한적으로 사용.

폰노이만 아키텍처의 핵심 장점: 하드웨어를 바꾸지 않고 프로그램(소프트웨어)만 교체하면 컴퓨터에 다른 작업을 시킬 수 있다. 이것이 "범용 컴퓨터"를 가능하게 한 결정적 설계 원칙이다.

---

## 종합

폰노이만 아키텍처 덕분에 같은 노트북으로 Chrome을 실행하다가 VS Code를 실행할 수 있다. 하드웨어(회로 배선)를 바꾸지 않고 실행할 프로그램(메모리에 올라온 코드)만 바꾸는 것이다. 폰노이만 이전에는 특정 계산을 위해 회로를 물리적으로 재배치해야 했다. 이 아키텍처가 없었다면 "운영체제"라는 개념 자체가 존재하기 어려웠다.
