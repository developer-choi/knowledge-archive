# [UNVERIFIED] 컴퓨터 시스템의 기본 구성이 어떻게 되나요?

## 도입

수많은 사용자와 애플리케이션이 동시에 하드웨어에 요청을 보낸다. 이 모든 요청을 하드웨어가 직접 받으면 충돌이 발생한다. 그 사이에서 중재하는 것이 OS다.

---

## 본문

컴퓨터 시스템의 계층 구조:

```
사용자 (사람, 기계, 또는 다른 컴퓨터)
      ↓
Application (Chrome, VS Code, Node.js 앱)
      ↓
OS (운영체제) ← 중간 매개체
      ↓
Hardware (CPU, RAM, I/O Device)
```

사용자는 반드시 사람일 필요는 없다. 다른 기계가 API로 요청을 보내거나, 스크립트가 자동으로 명령을 내리는 것도 OS 입장에서는 사용자의 요청이다.

OS는 Application과 Hardware의 중간자다. Application이 하드웨어에 직접 접근하면 충돌·보안 문제가 생기므로, OS가 요청을 받아 하드웨어를 대신 제어하고 결과를 돌려준다.

---

## 종합

Chrome이 화면에 픽셀을 그릴 때 직접 GPU 회로를 제어하는 것이 아니라, OS(Windows의 GDI, macOS의 Core Graphics)가 대신 그린다. 여러 앱이 동시에 화면을 그려도 충돌 없이 각자의 창에 표시될 수 있는 것이 OS 중재의 결과다.

# [UNVERIFIED] OS는 무슨 일을 하나요?

## 도입

OS의 역할은 크게 두 가지다 — 한정된 하드웨어 자원을 효율적으로 배분하는 것과, 프로세스 간 충돌·문제를 예방하고 해결하는 것.

---

## 본문

**역할 1: 하드웨어를 효율적으로 관리한다**

CPU·메모리·I/O 같은 리소스는 한정되어 있는데 수많은 Application이 동시에 요청한다. OS는 이 경쟁을 중재하여 효율적이고 공정하게 배분한다.

- 어떤 프로세스 하나가 CPU를 독점하지 않도록 제한한다
- 중요한 프로세스에게는 더 많은 CPU 시간을 보장한다
- 모든 리소스 할당(CPU 스케줄링, 메모리 할당, 파일 핸들 배분)을 OS가 담당한다

**역할 2: 프로세스 문제를 예방·해결한다**

프로세스들이 OS에게 요청하다가 데드락(Deadlock) 같은 문제가 생길 수 있다. OS는 이런 상황을 미리 감지하거나, 발생 후에 해결하는 역할을 한다.

---

## 종합

Windows Task Manager에서 "CPU 사용률 100%"가 표시될 때 다른 앱이 여전히 동작하는 것은 OS의 CPU 스케줄링 덕분이다. 특정 앱이 무한 루프로 CPU를 독점하려 해도 OS가 강제로 다른 프로세스에게 CPU를 넘긴다. OS가 없으면 프로세스 하나가 전체 시스템을 멈추게 만들 수 있다.

# [UNVERIFIED] 컴퓨터의 부팅 과정에 대해 설명해주세요.

## 도입

전원 버튼을 누른 순간부터 Windows 바탕화면이 나오기까지 무슨 일이 벌어지는가? 핵심은 "RAM에 아무것도 없는 상태에서 OS를 메모리에 올리기까지의 과정"이다.

---

## 본문

부팅 과정 순서:

1. **전원 인가**: 사용자가 PC 전원 버튼을 누르면 메인보드에 전력이 공급된다.
2. **BIOS/UEFI 실행**: CPU가 ROM에 저장된 BIOS(또는 UEFI) 코드를 실행한다. RAM이 아직 아무것도 없는 상태지만 ROM은 전원 없이도 유지되므로 읽을 수 있다.
3. **POST(Power-On Self Test)**: BIOS가 CPU, 메모리, 그래픽 카드 등 주요 하드웨어에 이상이 없는지 진단한다. 이 단계에서 문제가 생기면 비프음이나 블루스크린이 발생한다.
4. **MBR 읽기**: BIOS가 디스크의 MBR(Master Boot Record)을 읽어온다. MBR은 메모리가 아니라 디스크에 있다(RAM은 휘발성이라 전원이 꺼지면 사라지기 때문).
5. **Bootloader 적재**: MBR에서 읽은 정보를 바탕으로 RAM에 Bootloader를 올린다.
6. **OS 커널 로드**: Bootloader가 디스크에 있는 OS 커널 코드를 복사해 메모리에 적재한다.

```
전원 ON
  ↓
ROM(BIOS) 실행 → POST(하드웨어 검사)
  ↓
디스크 MBR 읽기
  ↓
RAM에 Bootloader 적재
  ↓
RAM에 OS 커널 적재
  ↓
OS 실행 시작
```

---

## 종합

부팅이 느린 컴퓨터는 보통 HDD에서 OS를 읽어오는 단계(5→6)가 병목이다. SSD로 교체하면 부팅이 빠른 이유다. "클라우드 서버 인스턴스가 30초 만에 켜진다"는 것도 이 부팅 과정 — OS 이미지를 가상 디스크에서 RAM으로 복사하는 시간이다.

# [UNVERIFIED] 인터럽트가 뭔가요?

## 도입

CPU는 한 번에 하나의 명령어를 순서대로 실행한다. 그런데 더 급한 일이 생겼을 때 현재 작업을 잠깐 중단하고 급한 일을 먼저 처리해야 하는 상황이 있다. 이것이 인터럽트다.

---

## 본문

인터럽트(Interrupt) = 끼어들기. 우선순위가 더 높은 작업이 생겼으니 이것부터 처리해달라는 신호다.

CPU가 어떤 작업을 실행 중일 때 I/O 완료, 키보드 입력, 하드웨어 오류 같은 예외 상황이 발생하면 CPU에게 신호를 보내는 것이 인터럽트다.

인터럽트 구분:

- **H/W 인터럽트 vs S/W 인터럽트**
- **외부(External) 인터럽트 vs 내부(Internal) 인터럽트**
- **I/O 인터럽트 vs Exception 인터럽트**

인터럽트가 없다면 CPU는 I/O가 완료됐는지 확인하기 위해 계속 상태를 폴링(polling)해야 한다. 이것은 CPU를 낭비하는 방식이다. 인터럽트를 통해 "다 됐을 때 알려줘"가 가능해진다.

---

## 종합

Node.js의 이벤트 루프에서 `fs.readFile(callback)`이 비동기로 동작하는 이유가 하드웨어 인터럽트와 연결된다. OS가 파일 I/O를 시작하고, CPU는 다른 일을 한다. I/O Controller가 파일 읽기를 완료하면 CPU에게 인터럽트를 보내고, OS가 이를 받아 Node.js 이벤트 큐에 완료 이벤트를 넣는다. 그러면 이벤트 루프가 `callback`을 실행한다.

# [UNVERIFIED] 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이는 무엇인가요?

## 도입

인터럽트는 발생 주체에 따라 하드웨어 인터럽트와 소프트웨어 인터럽트로 나뉜다. 누가 CPU에게 신호를 보내느냐가 차이다.

---

## 본문

**하드웨어 인터럽트(H/W Interrupt)**

CPU 외부의 하드웨어 장치가 CPU에게 보내는 신호다.

예:
- 키보드 키를 누름 → 키보드 컨트롤러가 CPU에 인터럽트 신호
- I/O 작업 완료 → I/O Controller가 CPU에 신호
- 전원 이상 → 전원 공급 회로가 CPU에 신호
- 오버플로 발생 → ALU가 CPU에 내부 신호

**소프트웨어 인터럽트(S/W Interrupt)**

소프트웨어가 발생시키는 인터럽트다. 대표적인 것이 SVC(Supervisor Call)다.

소프트웨어가 특권이 필요한 위험한 작업(메모리 직접 접근, 시스템 리소스 변경 등)을 요청할 때, SVC가 중간에서 가로채 관리·감독한다. 이것이 시스템 콜의 하드웨어 기반이다.

```
비교
──────────────────────────────────────
H/W 인터럽트  외부 하드웨어가 발생  비동기
S/W 인터럽트  소프트웨어가 발생    동기(의도적 호출)
──────────────────────────────────────
```

---

## 종합

Node.js에서 `fs.readFile()`을 호출하면 소프트웨어 인터럽트(시스템 콜)가 발생해 OS 커널이 제어를 받는다. 파일 읽기가 완료되면 하드웨어 인터럽트(I/O 완료 신호)가 발생해 OS가 다시 알림을 받는다. 비동기 I/O의 라이프사이클이 소프트웨어 인터럽트로 시작하고 하드웨어 인터럽트로 완료된다.

# [UNVERIFIED] 인터럽트의 Flow에 대해 설명해주세요.

## 도입

인터럽트가 발생했을 때 CPU가 어떤 순서로 반응하는지 알면, OS가 멀티태스킹을 어떻게 구현하는지 이해할 수 있다.

---

## 본문

인터럽트 처리 흐름:

1. **Interrupt 발생**
2. **CPU 현재 작업 중지**
3. **현재 수행 중이던 명령어의 주소값을 PC에 저장** ← 나중에 복귀를 위한 필수 단계
4. **인터럽트가 요청한 위치(ISR 주소)로 PC 변경** → 인터럽트 서비스 루틴 실행
5. **ISR 처리 완료**
6. **저장된 주소값으로 복귀** → 중단된 원래 작업 재개

**I/O 리퀘스트에 의한 흐름:**

```
I/O Request 시작 (예: fs.readFile 호출)
      ↓
I/O Controller에 작업 위임, CPU는 다른 일 수행
      ↓
I/O 완료 → I/O Controller가 CPU에 인터럽트 전송
      ↓
CPU: 현재 작업 PC에 저장
      ↓
인터럽트 처리 (OS가 완료 데이터 수신)
      ↓
원래 작업으로 복귀
      ↓
(반복)
```

---

## 종합

Windows Task Manager에서 CPU 사용률이 낮은데도 프로그램이 I/O를 많이 수행하고 있는 경우, CPU는 대부분 인터럽트를 기다리거나 다른 프로세스를 실행하는 상태다. Node.js의 논블로킹 I/O 모델이 높은 동시성을 처리할 수 있는 이유는 하나의 스레드가 I/O 완료 인터럽트를 이벤트로 받아 처리하는 구조이기 때문이다.

# [UNVERIFIED] 운영체제가 사용자의 편의를 위해 제공하는 기능은 뭐가 있나요?

## 도입

OS는 하드웨어를 관리하는 것 외에 사용자와 직접 상호작용하는 편의 기능도 제공한다. 이 기능들이 없으면 모든 작업을 CPU 명령어 수준에서 직접 해야 한다.

---

## 본문

OS가 사용자 편의를 위해 제공하는 기능:

- **GUI / CLI**: 사용자가 명령을 내리는 인터페이스. PC는 GUI(그래픽), 서버 환경은 주로 CLI(터미널).
- **프로그램 실행**: OS가 프로그램을 메모리에 적재하고, 실행하고, 종료를 지원한다.
- **I/O 수행**: 실행 중인 프로그램의 입출력 요청을 처리한다.
- **파일 시스템 조작**: 파일·디렉토리의 생성·읽기·수정·삭제(CRUD)를 지원하고, 사용자 권한도 관리한다.
- **리소스 할당**: CPU 시간, 메모리 공간, 파일 핸들 등을 프로세스에 배분한다.
- **로깅(Logging)**: 어떤 프로그램이 얼마나 많은 자원을 언제 사용했는지 추적한다.

---

## 종합

`node app.js`를 터미널에 입력하는 것이 CLI → OS → 프로그램 실행 흐름이다. `fs.readFile()`이 파일을 읽는 것이 I/O 수행이고, `fs.mkdir()`가 디렉토리를 만드는 것이 파일 시스템 조작이다. OS 기능이 없으면 Node.js 앱이 할 수 있는 일이 거의 없다 — 파일도 못 읽고, 네트워크도 못 쓰고, 화면에 출력도 못 한다.

# [UNVERIFIED] 시스템 콜이 뭔가요?

## 도입

사용자(애플리케이션)가 OS에게 서비스를 요청하는 수단이 시스템 콜이다. OS는 시스템 콜을 경계로 두 영역으로 나뉜다 — 사용자가 접근하는 영역과 OS 커널이 직접 실행되는 영역.

---

## 본문

시스템 콜(System Call)은 운영체제에게 서비스를 요청하는 메커니즘이다.

```
사용자 측 요청 (CLI, GUI)
       ↕
  <<< 시스템 콜 >>>
       ↕
OS가 제공하는 서비스들
(프로그램 실행, 파일 조작, I/O, 리소스 할당 등)
```

시스템 콜의 두 가지 모드:

- **사용자 모드(User Mode)**: 애플리케이션이 실행되는 모드. 하드웨어에 직접 접근하지 못한다.
- **커널 모드(Kernel Mode)**: OS 커널이 실행되는 모드. 하드웨어에 직접 접근할 수 있다.

사용자 모드에서 커널 모드로 전환하는 것이 시스템 콜의 핵심 역할이다. 사용자가 직접 메모리 주소를 건드리거나 하드웨어를 제어하면 보안·안정성 문제가 생기기 때문에, OS가 중간에서 검증 후 대신 처리한다.

---

## 종합

Node.js의 `fs.readFile()`이 파일을 읽을 때 내부적으로 `read()` 시스템 콜을 발생시켜 OS 커널에 요청을 보낸다. 커널이 파일을 읽어서 결과를 Node.js에 돌려준다. 시스템 콜이 없으면 Node.js가 직접 SSD 컨트롤러와 통신해야 하는데, 이건 보안 문제이자 하드웨어 종속 문제다.

# [UNVERIFIED] cp in.txt out.txt 명령어를 입력하면 구체적으로 어떤 일이 발생하나요?

## 도입

단순한 파일 복사 명령 하나가 OS 레벨에서 여러 시스템 콜과 절차로 분해된다. 이 예시로 시스템 콜이 실제로 어떻게 쓰이는지 확인할 수 있다.

---

## 본문

`cp in.txt out.txt` 처리 과정:

1. 사용자가 시스템 콜을 통해 "in.txt를 out.txt로 복사해줘" 요청을 OS에 보낸다.
2. 커널이 요청을 처리:
   - 입력 파일명(`in.txt`)과 출력 파일명(`out.txt`)을 파라미터로 받는다.
   - `in.txt`를 연다 → 파일이 없으면 에러 반환.
   - `out.txt`를 생성한다 → 이미 파일이 있으면 에러 반환(또는 덮어쓰기).
   - `in.txt`에서 한 줄 읽고 → `out.txt`에 한 줄 쓴다. 더 이상 읽을 내용이 없을 때까지 반복.
   - 두 파일을 닫는다.
   - 완료 메시지를 사용자에게 반환한다.

```
사용자: "cp in.txt out.txt"
      ↓ (시스템 콜)
OS 커널:
  open(in.txt)  → 파일 핸들 획득
  create(out.txt) → 새 파일 생성
  loop:
    read(in.txt) → 데이터 읽기
    write(out.txt) → 데이터 쓰기
  close(in.txt), close(out.txt)
      ↓
사용자: 완료 메시지 수신
```

사용자가 직접 파일 핸들을 열고 주소를 계산해서 복사하면 문제가 생길 가능성이 높다. 시스템 콜을 통해 OS에 위임하는 것이 보안과 안정성을 보장하는 방법이다.

---

## 종합

Node.js에서 `fs.copyFile('in.txt', 'out.txt')`을 호출하면 내부적으로 이 시스템 콜 시퀀스가 실행된다. 개발자가 OS 내부 절차를 직접 구현할 필요 없이 OS API(시스템 콜)를 통해 안전하게 파일 작업을 수행할 수 있다.

# [UNVERIFIED] 시스템 콜들은 어떤 것들이 있는지 나열해주세요.

## 도입

시스템 콜은 OS가 제공하는 서비스의 종류만큼 다양하다. 범주별로 묶어서 파악하면 OS가 어떤 기능을 담당하는지 전체 그림이 보인다.

---

## 본문

시스템 콜 주요 범주:

**프로세스 제어**
- 프로세스 생성·종료·로드·실행
- wait(대기), signal(신호 보내기)
- 메모리 할당·해제

**파일 관리**
- 파일 열기·닫기
- 읽기·쓰기·수정
- 생성·삭제·복사·이동
- 파일 권한 관리

**장치 관리**
- I/O 장치 연결·분리 요청

**통신**
- 통신 연결 생성
- 데이터 송수신
- 전송 상태 확인

시스템 콜 없이 사용자가 직접 할 수 있는 것은 거의 없다 — 파일 읽기, 화면 출력, 네트워크 통신, 프로세스 생성 모두 시스템 콜을 거친다.

```
Node.js 앱에서 자주 발생하는 시스템 콜 매핑
─────────────────────────────────────────────
fs.readFile()    → read() 시스템 콜
fs.writeFile()   → write() 시스템 콜
net.connect()    → socket(), connect() 시스템 콜
child_process()  → fork()/exec() 시스템 콜
```

---

## 종합

Node.js 앱은 시스템 콜 없이는 아무것도 할 수 없다. `console.log("hello")`도 stdout에 write() 시스템 콜을 호출한다. OS가 시스템 콜 인터페이스를 제공하기 때문에 Node.js가 Windows/macOS/Linux에서 모두 동작할 수 있다 — 각 OS의 시스템 콜을 Node.js 내부에서 추상화하기 때문이다.
