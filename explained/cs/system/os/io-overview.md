# [UNVERIFIED] I/O 장치들이 시스템 버스에 직접 접속하지 못하는 이유는?

## 도입

CPU와 메모리는 시스템 버스로 직접 연결된다. 그런데 키보드, 프린터, USB 드라이브 같은 I/O 장치들은 왜 직접 연결이 안 될까? 세 가지 근본적인 이유가 있다.

---

## 본문

**이유 1: 다양한 종류 → 각각 다른 제어 방법**

I/O 장치들은 종류가 매우 다양하고, 장치마다 제어 방법이 다르다. 그 제어 회로를 모두 CPU 내부에 포함시키는 것은 불가능하다. 새로운 I/O 장치가 나올 때마다 CPU를 다시 설계해야 하기 때문이다. I/O Controller가 중간에서 각 장치의 특성을 처리한다.

**이유 2: 속도 차이**

I/O 장치들의 데이터 전송·처리 속도가 CPU보다 훨씬 느리다. 키보드는 사람이 타이핑하는 속도, 프린터는 인쇄 속도에 맞춰 동작한다. 고속의 시스템 버스와 직접 연결하면 버스가 느린 I/O를 기다리느라 다른 통신이 막힌다.

**이유 3: 데이터 형식 불일치**

I/O 장치들이 사용하는 데이터 형식이 CPU와 다르다. 키보드는 문자 코드를, 화면은 픽셀 데이터를, 네트워크 카드는 패킷을 다룬다. CPU가 이것을 직접 해석하면 CPU 설계가 무한히 복잡해진다.

```
시스템 버스 ─ CPU, RAM (직접 연결, 고속, 규격 통일)

시스템 버스 ─ I/O Controller ─ I/O Device
             (중간 변환, 속도 맞춤, 형식 변환)
```

---

## 종합

I/O Controller 없이 시스템 버스에 키보드를 직접 연결한다면, 키보드를 하나 바꿀 때마다 CPU 버스 프로토콜을 바꿔야 한다. I/O Controller는 "다양한 I/O 장치와 통일된 시스템 버스 사이의 번역가"다. 드라이버(driver) 소프트웨어가 OS에 필요한 이유도 여기 있다 — 새 I/O 장치가 나오면 그 장치의 Controller와 통신하는 방법을 OS에 알려주는 소프트웨어가 드라이버다.

# [UNVERIFIED] I/O Controller의 주요 기능은?

## 도입

I/O Controller는 CPU와 I/O 장치 사이의 중간자다. CPU가 하는 일에서 I/O 장치를 직접 상대하는 부분을 대신 처리한다.

---

## 본문

I/O Controller의 3가지 주요 기능:

**1. I/O 장치 제어**
CPU로부터 명령을 받아 연결된 I/O 장치를 실제로 제어한다. 프린터 I/O Controller가 프린터 헤드 이동, 잉크 분사를 직접 제어하는 것처럼.

**2. CPU와 통신 / I/O 장치와 통신**
양쪽 인터페이스를 모두 처리한다.
- CPU 측: 시스템 버스 프로토콜로 통신
- I/O 장치 측: 각 장치 고유의 프로토콜로 통신

**3. 데이터 버퍼링(Data Buffering)**
CPU와 I/O 장치의 속도 차이를 완충한다. I/O Controller 안의 Data Register가 버퍼 역할을 한다. CPU가 데이터를 빠르게 보내도 I/O 장치가 처리할 준비가 될 때까지 Data Register에 임시 저장한다.

```
I/O Controller 내부
┌─────────────────────────────┐
│ Status Register  현재 장치 상태 (Ready/Busy) │
│ Data Register    임시 데이터 버퍼            │
└─────────────────────────────┘
      ↕                 ↕
  시스템 버스         I/O Device
  (CPU 측)           (장치 측)
```

---

## 종합

Node.js에서 `fs.createReadStream()`으로 파일을 읽을 때 내부적으로 SSD의 I/O Controller와 OS가 협력하여 데이터를 버퍼에 채우고, 준비됐을 때 Node.js에게 이벤트를 발생시킨다. I/O Controller의 데이터 버퍼링이 이 비동기 I/O의 하드웨어 기반이다. 버퍼링이 없으면 CPU는 I/O가 완료될 때까지 아무것도 못 하고 기다려야 한다.
