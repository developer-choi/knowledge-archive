# [UNVERIFIED] 디지털 데이터란 무엇인가?

## 도입

컴퓨터가 처리하는 모든 정보 — 숫자, 문자, 이미지, 영상 — 는 내부적으로 0과 1의 조합으로 표현된다. 이것이 디지털 데이터다. "디지털"이 아날로그와 대비되는 이유는 연속적인 값이 아니라 불연속적인 이산(discrete) 값만 사용하기 때문이다.

---

## 본문

디지털 데이터는 0과 1, 두 가지 상태만으로 표현되는 정보다.

왜 0과 1인가: 전기 회로는 "전압이 있다(1)" / "전압이 없다(0)" 두 상태를 가장 안정적으로 구분할 수 있기 때문이다. 전압이 애매한 중간값이면 잡음(noise)에 취약하다. 두 상태만 사용하면 잡음이 생겨도 0인지 1인지 확실히 판단할 수 있다.

모든 정보는 0과 1의 조합으로 인코딩된다:

- 숫자 42 → `00101010` (이진수)
- 문자 'A' → `01000001` (ASCII)
- 이미지 → 픽셀마다 RGB 값을 이진수로 표현한 배열

---

## 종합

JS에서 `typeof 42 === 'number'`라고 해도, CPU 레벨에서 `42`는 `00000000 00000000 00000000 00101010`이라는 이진수다. 디지털 데이터가 없다면 컴퓨터는 존재할 수 없다 — 아날로그 회로로 복잡한 계산을 하는 것은 잡음 문제로 실용적이지 않기 때문이다.

---

# [UNVERIFIED] bit란 무엇인가?

## 도입

디지털 데이터의 최소 단위가 비트(bit)다. 0 또는 1, 두 가지 값 중 하나를 담을 수 있는 가장 작은 정보 단위다.

---

## 본문

비트(bit, binary digit)는 정보를 나타내는 최소 단위다.

- 값: 0 또는 1, 두 가지만 가질 수 있다
- "컴퓨터가 처리할 수 있는 가장 작은 단위의 정보"

비트 수와 표현 가능한 경우의 수:

```
비트 수   경우의 수   예시
────────────────────────────────
1비트     2¹ = 2    0, 1
2비트     2² = 4    00, 01, 10, 11
3비트     2³ = 8    000 ~ 111
8비트     2⁸ = 256  0 ~ 255
32비트    2³² ≈ 42억
64비트    2⁶⁴ ≈ 1800경
```

비트가 의미 있는 정보가 되려면 무엇이 0이고 무엇이 1인지 약속(인코딩 규칙)이 필요하다.

---

## 종합

JS의 `Number` 타입은 64비트 부동소수점(IEEE 754 double)으로 표현된다. `Number.MAX_SAFE_INTEGER`가 약 900조인 것은 64비트 중 부호·지수 비트를 제외한 52비트로 정수를 표현하기 때문이다. 비트 하나의 차이가 표현 가능한 값의 범위를 두 배씩 바꾼다.

---

# [UNVERIFIED] byte란 무엇인가?

## 도입

비트 하나로는 너무 작아서 실용적인 정보를 담기 어렵다. 컴퓨터는 여러 비트를 묶어서 더 큰 단위로 처리한다. 가장 기본적인 묶음 단위가 바이트(byte)다.

---

## 본문

바이트(byte)는 8비트를 하나로 묶은 단위다. 컴퓨터가 데이터를 읽고 쓸 때 가장 기본적으로 사용하는 단위다.

- 1 byte = 8 bits
- 1 byte로 표현 가능한 경우의 수: 2⁸ = 256 (0~255)

```
크기 단위
1 byte    = 8 bits
1 KB      = 1,024 bytes  (2¹⁰)
1 MB      = 1,024 KB
1 GB      = 1,024 MB
1 TB      = 1,024 GB
```

컴퓨터 메모리의 주소는 바이트 단위로 지정된다. `arr[0]`부터 `arr[1]`까지가 1바이트 단위로 구분되는 것도 이 때문이다. (`Uint8Array`의 `8`이 1바이트를 뜻한다.)

---

## 종합

`Buffer.alloc(1024)`는 1024바이트(1KB) 메모리를 할당한다. Node.js에서 HTTP 요청 본문을 청크(chunk)로 읽을 때 단위도 바이트다. 파일 크기가 "12.4 MB"로 표시되는 것도 바이트 단위로 계산된 결과다. 바이트가 컴퓨터 저장과 전송의 기본 화폐 단위라는 것을 이해하면, 네트워크 대역폭이나 메모리 계산이 직관적으로 다가온다.

---

# [UNVERIFIED] Character Code란 무엇인가?

## 도입

컴퓨터는 0과 1만 이해한다. 그런데 사용자는 'A', 'B', '안' 같은 문자를 다루고 싶다. 문자와 이진수 사이의 대응표가 필요하다. 이것이 Character Code(문자 코드)다.

---

## 본문

Character Code는 문자와 숫자(이진수) 사이의 대응표다. 컴퓨터에서 문자를 입력하고 화면에 표시할 수 있는 것은 이 대응표 덕분이다.

```
예시 (가상의 4비트 코드):
A = 0001
B = 0010
C = 0011
...
```

실제 코드: `'A'`를 저장할 때 컴퓨터는 `65`(0x41)라는 숫자를 저장하고, 출력할 때는 그 숫자를 보고 `A`를 화면에 그린다.

Character Code가 없으면 문자를 다루는 것이 불가능하다 — 컴퓨터는 숫자만 알고 문자는 모르기 때문이다. 모든 문자 처리 소프트웨어는 Character Code를 전제로 한다.

---

## 종합

JS에서 `'A'.charCodeAt(0) === 65`가 성립하는 것이 Character Code(ASCII)의 직접적 노출이다. `String.fromCharCode(65) === 'A'`도 마찬가지다. 한글 입력이 가능한 것은 한국어 문자에도 Character Code(Unicode)가 정의되어 있기 때문이다.

---

# [UNVERIFIED] ASCII Code란 무엇이며, 한계는 무엇인가?

## 도입

가장 먼저 널리 쓰인 Character Code가 ASCII다. 영어권 환경에서 필요한 문자들을 정의했지만, 처음부터 전 세계 언어를 고려하지 않은 것이 한계가 됐다.

---

## 본문

ASCII(American Standard Code for Information Interchange)는 7비트(128개 코드)로 다음을 표현한다:

- 숫자(0~9): `'0'` = 48, `'9'` = 57
- 대문자(A~Z): `'A'` = 65, `'Z'` = 90
- 소문자(a~z): `'a'` = 97, `'z'` = 122
- 특수 기호(!, @, #, ... 등)
- 제어 문자(줄바꿈, 탭 등)

```
ASCII 일부 예시
0x41 = 65  → 'A'
0x61 = 97  → 'a'
0x30 = 48  → '0'
0x0A = 10  → 줄바꿈(LF)
```

**한계**: 7비트 = 128개 코드밖에 없어서 영어 이외의 언어(한국어, 중국어, 아랍어 등)를 표현할 수 없다. 각 나라가 자체 확장 코드를 만들면서 호환성 문제가 발생했다.

---

## 종합

과거 ASCII 시대에는 한글 파일을 다른 나라 컴퓨터에서 열면 깨지는 문제가 흔했다. `Content-Type: text/html; charset=UTF-8`이라는 헤더가 있는 이유가 바로 "이 텍스트는 어떤 Character Code를 쓰는지"를 명시하기 위해서다. ASCII 한계를 해결한 것이 Unicode다.

---

# [UNVERIFIED] Unicode란 무엇인가?

## 도입

ASCII가 영어만 다룰 수 있는 한계를 가졌다면, Unicode는 전 세계 모든 언어의 문자를 하나의 표준으로 정의하려는 시도다. 현재 인터넷과 소프트웨어의 기본 문자 체계가 됐다.

---

## 본문

Unicode는 전 세계 문자를 하나의 코드 체계에 포함하는 Character Code 표준이다. ASCII의 한계(128개 코드, 영어 전용)를 해결한다.

현재 Unicode는 100만 개 이상의 코드 포인트를 정의한다. 한국어, 중국어, 아랍어, 이모지, 고대 문자까지 포함한다.

Unicode를 인코딩하는 방식이 여러 가지 있다:

- **UTF-8**: 가장 널리 쓰인다. 영어(ASCII 호환)는 1바이트, 한글은 3바이트로 가변 인코딩.
- **UTF-16**: JS 엔진 내부에서 문자열을 저장하는 방식.
- **UTF-32**: 모든 문자를 4바이트로 고정 인코딩.

**JS와 Unicode:** JS 문자열은 내부적으로 UTF-16을 사용한다. `'한'.length === 1`이지만 이모지 `'😀'.length === 2`인 것은 일부 이모지가 UTF-16에서 2개의 코드 유닛(surrogate pair)을 차지하기 때문이다.

---

## 종합

`<meta charset="UTF-8">`이 HTML 파일 상단에 있는 것은 이 파일이 UTF-8로 인코딩된 Unicode를 사용한다고 브라우저에 알려주는 것이다. Node.js에서 `Buffer.from('안녕', 'utf8').length === 6`인 것은 '안'과 '녕' 각각이 UTF-8에서 3바이트를 차지하기 때문이다. Unicode 덕분에 한국어 도메인, 다국어 API 응답, 이모지가 동작할 수 있다.
