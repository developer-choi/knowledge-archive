# [UNVERIFIED] CPU가 명령어를 실행할 때 수행하는 동작들을 순서대로 설명해주세요.

## 도입

CPU가 하는 일은 근본적으로 반복이다 — 메모리에서 명령어를 읽고, 해석하고, 실행하고, 결과를 저장한다. 이 사이클을 이해하면 프로그램 실행의 가장 낮은 수준을 파악할 수 있다.

---

## 본문

CPU가 명령어를 실행할 때의 5단계:

**1. Instruction Fetch (명령어 인출)**
기억장치(메모리)에 저장된 명령어를 CPU 안으로 읽어오는 단계다. Program Counter(PC)가 가리키는 주소의 명령어를 가져온다.

**2. Instruction Decode (명령어 해독)**
가져온 명령어가 무슨 연산을 하라는 것인지 해석하는 단계다. Control Unit이 Operation Code를 읽어 어떤 회로를 활성화할지 결정한다.

**3. Data Fetch (데이터 인출)**
명령어 실행에 필요한 데이터를 메모리나 I/O 장치에서 읽어오는 단계다. 데이터가 필요한 명령어에서만 수행된다.

**4. Data Process (데이터 처리)**
ALU가 산술 연산 또는 논리 연산을 수행하는 단계다.

**5. Data Store (데이터 저장)**
연산 결과를 메모리에 저장하는 단계다.

```
Fetch → Decode → (Data Fetch) → (Data Process) → (Data Store)
  ↑_______________________________________________|
  (다음 명령어로 PC 증가 후 반복)
```

3~5단계는 데이터가 필요한 명령어에서만 실행된다. `NOP`(No Operation) 같은 명령어는 1~2단계만 거친다.

---

## 종합

V8이 `a + b`를 JIT 컴파일하면, CPU는 이 5단계를 수십 나노초 안에 처리한다. 3GHz CPU는 초당 30억 번의 클럭 사이클을 가지며, 각 명령어가 여러 사이클에 걸쳐 이 단계들을 처리한다. 파이프라이닝 기술은 이 단계들을 겹쳐 실행해 처리량을 높인다.

# [UNVERIFIED] CPU의 기본 구조를 설명해주세요.

## 도입

CPU가 명령어를 실행하는 5단계는 CPU 내부의 여러 구성 요소가 분업해서 처리한다. CPU를 열어보면 네 가지 주요 구성 요소가 있다.

---

## 본문

CPU 내부 4가지 구성 요소:

**1. ALU (Arithmetic and Logic Unit, 산술논리연산장치)**
산술 연산(+, -, ×, ÷)과 논리 연산(AND, OR, NOT, XOR)을 처리하는 회로 집합이다. 실제 "계산"이 일어나는 곳이다.

**2. 레지스터 세트 (Register Sets)**
CPU 내부에 있는 가장 빠른 저장장치다. ALU가 연산하는 동안 피연산자와 중간 결과를 임시 보관한다. 메모리 계층에서 가장 빠르고 가장 작다.

**3. CU (Control Unit, 제어장치)**
명령어를 인출·해독하고, ALU와 레지스터·메모리에 제어 신호를 보내 각 단계가 올바르게 실행되도록 조율한다.

**4. CPU 내부 버스 (Internal Bus)**
CPU 내부 구성 요소끼리 데이터를 주고받는 경로다. 외부의 시스템 버스와 직접 연결되지 않으며, 버퍼(MAR, MBR)를 통해 외부 버스와 연결된다.

```
CPU 내부
┌────────────────────────────────────────┐
│  ┌──────┐    내부 버스    ┌──────────┐  │
│  │  CU  │ ←────────────→ │레지스터  │  │
│  └──────┘                │ 세트     │  │
│      ↕                   └──────────┘  │
│  ┌──────┐                    ↕          │
│  │ ALU  │ ←──────────────────          │
│  └──────┘                              │
│  (내부 버스는 시스템 버스와 직접 연결 안 됨)  │
└────────────────────────────────────────┘
```

---

## 종합

`a + b`를 실행할 때: CU가 명령어를 인출·해독하고, 레지스터에 `a`와 `b` 값을 올리고, ALU가 덧셈을 수행하고, 결과가 레지스터에 저장된다. Windows Task Manager의 CPU 코어 사용률은 이 사이클이 얼마나 바쁘게 돌아가는지를 나타낸다.

# [UNVERIFIED] 명령어 사이클(Instruction Cycle)이 뭔가요?

## 도입

CPU가 프로그램을 실행한다는 것은 명령어를 하나씩 처리하는 반복 사이클이다. 이 사이클이 전원이 꺼지거나 오류가 발생하기 전까지 계속 돌아간다.

---

## 본문

명령어 사이클(Instruction Cycle)은 CPU가 하나의 명령어를 완전히 실행하는 데 필요한 전체 과정이다.

두 단계로 구성된다:

- **명령어 인출 단계(Fetch Cycle)**: CPU가 메모리에서 다음 실행할 명령어를 읽어오는 단계. Program Counter(PC)가 다음 명령어 주소를 가리킨다.
- **명령어 실행 단계(Execute Cycle)**: 인출한 명령어를 해독하고 실행하는 단계. 데이터 인출·처리·저장이 여기에 포함된다.

```
프로그램 실행 시작
     ↓
[Fetch] ← PC가 가리키는 주소의 명령어를 인출
     ↓
[Execute] ← 명령어 해독·실행
     ↓
PC 증가 (다음 명령어 주소로)
     ↓
[반복] ← 전원 차단 또는 복구 불가 오류 발생 전까지
```

---

## 종합

`for (let i = 0; i < 1000; i++)`는 수천 번의 명령어 사이클로 분해된다. PC(Program Counter)가 루프 시작 주소로 되돌아갈 때마다 사이클이 반복된다. 인터럽트가 발생하면 사이클 중간에 CPU가 다른 주소로 점프했다가 다시 돌아오는 것도 이 사이클 개념 위에서 동작한다.

---

# [UNVERIFIED] CPU 내부 레지스터들은 뭐가 있나요?

## 도입

레지스터는 CPU가 명령어를 실행하는 동안 임시로 값을 보관하는 공간이다. 각 레지스터는 특정 역할을 담당하며, 이 역할 분담이 명령어 사이클을 가능하게 한다.

---

## 본문

주요 CPU 내부 레지스터:

- **PC (Program Counter)**: 다음에 인출될 명령어의 메모리 주소를 저장한다. 명령어가 실행될 때마다 자동으로 다음 주소로 증가한다. 이것이 없으면 CPU가 어디서부터 실행해야 하는지 알 수 없다.
- **AC (Accumulator)**: 산술·논리 연산의 결과를 일시적으로 저장하는 레지스터. ALU의 출력이 여기에 쌓인다.
- **IR (Instruction Register)**: 가장 최근에 인출된 명령어를 저장한다. CU가 이 레지스터를 읽어 명령어를 해독한다.
- **MAR (Memory Address Register)**: PC에서 온 주소값이 주소 버스로 나가기 전에 임시 저장되는 레지스터. 버스 연결 순서: PC → MAR → 주소 버스.
- **MBR (Memory Buffer Register)**: 메모리와 CPU 사이를 오가는 데이터를 임시 저장하는 레지스터. 버스 연결 순서: 메모리 ↔ 데이터 버스 ↔ MBR.

```
레지스터 흐름 (명령어 인출 시)
PC → MAR → 주소 버스 → 메모리
메모리 → 데이터 버스 → MBR → IR
```

---

## 종합

PC가 없으면 CPU는 다음에 어떤 명령어를 실행해야 할지 모른다. MAR과 MBR이 없으면 내부 버스와 외부 버스(시스템 버스) 사이의 연결이 끊어진다. 각 레지스터는 명령어 사이클의 특정 단계를 담당하며, 이 분업 덕분에 CPU가 명령어를 고속으로 처리할 수 있다.

# [UNVERIFIED] 인터럽트와 인터럽트의 처리 과정을 설명해주세요.

## 도입

CPU는 원래 하나의 명령어 사이클을 끝내고 다음 명령어로 넘어가는 순차적 흐름으로 작동한다. 그런데 예상치 못한 이벤트가 발생했을 때 현재 작업을 잠깐 멈추고 그것을 먼저 처리해야 하는 경우가 있다. 이것이 인터럽트 메커니즘이다.

---

## 본문

인터럽트(Interrupt)는 프로그램 실행 도중 CPU에게 "현재 작업을 중단하고 이것을 먼저 처리해달라"는 신호다. CPU 외부(I/O 장치 등)나 내부(소프트웨어, 오류)에서 발생한다.

인터럽트 처리 과정:

1. 외부에서 인터럽트 요청이 들어온다
2. CPU가 어떤 장치/이벤트에서 인터럽트를 요구했는지 확인한다
3. 현재 수행 중인 작업을 중지하고, **현재 명령어 주소값을 PC에 저장**한다 (나중에 복귀를 위해)
4. 해당 **인터럽트 서비스 루틴(ISR, Interrupt Service Routine)**을 호출한다
5. ISR 처리가 완료되면 저장했던 주소로 복귀하여 원래 작업을 재개한다

```
원래 작업 실행 중
       ↓
인터럽트 발생
       ↓
현재 PC 주소 저장 ← 복귀를 위해 필수
       ↓
ISR 실행 (인터럽트 처리)
       ↓
저장된 PC로 복귀 → 원래 작업 재개
```

**인터럽트 서비스 루틴(ISR)**: 특정 인터럽트를 처리하기 위해 미리 작성된 루틴(함수). 인터럽트 종류마다 다른 ISR이 있다.

---

## 종합

Node.js의 이벤트 루프와 개념적으로 비슷하다. `setTimeout(fn, 0)`은 현재 실행 중인 코드를 방해하지 않고 완료 후에 `fn`을 실행하는데, 하드웨어 인터럽트는 OS 수준에서 같은 원리로 "나중에 처리"가 아니라 "지금 즉시 끼어들어 처리"한다. 키보드를 누르면 화면에 바로 글자가 나타나는 것이 키보드 인터럽트 → OS ISR → 애플리케이션 이벤트로 이어지는 흐름의 결과다.

# [UNVERIFIED] 연산의 종류는 뭐가 있나요?

## 도입

CPU가 실행하는 명령어들은 연산의 종류에 따라 분류할 수 있다. 이 분류를 알면 어떤 명령어가 어떤 하드웨어 자원을 사용하는지 이해할 수 있다.

---

## 본문

CPU 연산의 5가지 종류:

- **데이터 전송**: 레지스터끼리, 기억장치끼리, 레지스터와 기억장치 사이에 데이터를 이동한다. `LOAD`, `STORE`, `MOVE` 명령어가 여기에 해당한다.
- **산술 연산**: 덧셈, 뺄셈, 곱셈, 나눗셈. ALU의 산술 연산장치가 처리한다.
- **논리 연산**: AND, OR, NOT, XOR. ALU의 논리 연산장치가 처리한다. 비트 마스킹이나 플래그 조작에 사용된다.
- **입출력 연산**: CPU와 I/O 장치 사이의 데이터 교환. 키보드 입력, 화면 출력 등이 이 명령어로 수행된다.
- **프로그램 제어**: 명령어 실행 순서를 변경한다. `if`, `for` 같은 분기(branch)와 함수 호출(subroutine call)이 여기 해당한다. PC(Program Counter) 값을 바꿔서 다른 주소로 점프한다.

---

## 종합

JS의 `if (x > 0) { ... }` 한 줄이 CPU에서는 비교 연산(산술) → 조건 분기(프로그램 제어) 두 종류의 명령어로 분해된다. `console.log()`는 입출력 연산을 포함한다. 고급언어의 모든 구문이 결국 이 다섯 가지 연산의 조합이다.

# [UNVERIFIED] 명령어 구성요소는 어떻게 되나요?

## 도입

CPU가 하나의 명령어를 읽으면, 그 안에 "무엇을 할지"와 "무엇을 가지고 할지"가 담겨 있다. 명령어는 구조화된 이진수 패턴이다.

---

## 본문

명령어의 3가지 구성요소:

- **Operation Code(연산 코드, Opcode)**: 수행될 연산이 무엇인지를 나타낸다. `LOAD`, `ADD`, `STORE`, `JUMP` 같은 동사에 해당한다. CPU는 이 필드를 보고 어떤 회로를 활성화할지 결정한다.
- **Operand(오퍼랜드)**: 연산을 수행하는 데 필요한 데이터 또는 데이터의 메모리 주소를 나타낸다. `ADD 5, 3`에서 `5`와 `3`이 오퍼랜드다.
- **Next Instruction Address(다음 명령어 주소)**: 조건 분기 명령어에서 다음에 실행할 명령어의 주소를 직접 지정한다. 순차 실행(PC + 1)이 기본이지만, 분기 명령어는 이 필드가 PC를 덮어쓴다.

```
명령어 비트 구조 (8비트 예시)
[ Opcode(3비트) ][ Operand(5비트) ]
  어떤 연산?      어디 있는 데이터?
```

---

## 종합

`if (x > 0) goto label`이 기계어 수준에서는 비교 명령(opcode=CMP, operand=x주소), 조건 점프 명령(opcode=JG, operand=label주소)으로 변환된다. Next Instruction Address 필드가 없다면 루프나 함수 호출을 구현할 수 없다 — 항상 다음 순서 명령어만 실행하는 직선 코드만 가능해진다.

# [UNVERIFIED] ALU가 하는 일과 ALU의 구성요소는 어떻게 되나요?

## 도입

CPU에서 실제 계산이 일어나는 곳이 ALU다. 덧셈 하나도 ALU가 없으면 불가능하다. ALU의 구성을 알면 CPU가 어떻게 음수를 다루고, 비트를 이동시키고, 오버플로를 감지하는지 이해할 수 있다.

---

## 본문

ALU(Arithmetic and Logic Unit)는 산술 연산과 논리 연산을 수행하는 회로 집합이다.

ALU의 5가지 구성 요소:

- **산술 연산장치**: 덧셈, 뺄셈, 곱셈, 나눗셈을 수행하는 회로.
- **논리 연산장치**: AND, OR, NOT, XOR 같은 논리 연산을 수행하는 회로. 비트 마스킹, 플래그 처리에 사용된다.
- **시프트 레지스터(Shift Register)**: 비트들을 좌우로 이동시킨다. 좌측 시프트는 2배 곱셈, 우측 시프트는 2로 나누기와 동치인 경우가 많아 빠른 연산에 활용된다.
- **보수기(Complementer)**: 2의 보수를 계산하는 장치. 컴퓨터가 뺄셈을 할 때 `a - b`를 `a + (-b)`로 변환하는데, `-b`를 2의 보수로 표현하기 위해 필요하다.
- **상태 레지스터(Status Register, Flag Register)**: 연산 결과의 상태(Zero 플래그, Carry 플래그, Overflow 플래그 등)를 저장한다. `if (x == 0)` 같은 조건 분기가 이 레지스터의 플래그를 확인해서 동작한다.

```
ALU 구성
┌──────────────────────────────────────┐
│ 산술 연산장치  +  -  ×  ÷           │
│ 논리 연산장치  AND  OR  NOT  XOR     │
│ 시프트 레지스터  << >>               │
│ 보수기          2's complement       │
│ 상태 레지스터   Zero Carry Overflow  │
└──────────────────────────────────────┘
```

---

## 종합

`42 - 10`을 CPU에서 실행하면: 보수기가 `10`을 `-10`(2의 보수)으로 변환하고, 산술 연산장치가 `42 + (-10)`을 계산하고, 상태 레지스터에 결과 플래그를 기록한다. 보수기가 없으면 뺄셈 회로를 별도로 만들어야 해서 회로가 두 배로 복잡해진다. 상태 레지스터가 없으면 조건 분기(`if`, `while`)를 구현할 수 없다.

# [UNVERIFIED] CU의 내부 구조를 설명해주세요.

## 도입

CU(Control Unit)는 CPU의 지휘자다. 명령어를 해독하고 ALU·레지스터·메모리에 신호를 보내 각 단계가 올바르게 실행되도록 조율한다. 마이크로 프로그래밍 방식의 CU는 내부에 자체적인 작은 프로그램 실행 구조를 갖추고 있다.

---

## 본문

CU의 6가지 구성 요소:

- **명령어 해독기(Instruction Decoder)**: IR(Instruction Register)에서 명령어의 Operation Code를 읽어 해독한다. "이 명령어는 ADD다"를 판단하는 회로.
- **제어 주소 레지스터(CAR, Control Address Register)**: 다음에 실행할 마이크로 명령어의 주소를 저장하는 레지스터. 마이크로 프로그램 안에서의 PC 역할.
- **제어 기억장치(Control Memory)**: 마이크로 프로그램을 저장하는 장치. 각 기계어 명령어를 처리하는 세부 마이크로 명령어 시퀀스가 여기에 저장된다.
- **제어 버퍼 레지스터(CBR, Control Buffer Register)**: 제어 기억장치에서 읽힌 마이크로 명령어를 일시적으로 저장한다. 실행 전 임시 보관소.
- **서브루틴 레지스터(SBR, Subroutine Register)**: 마이크로 프로그램에서 서브루틴이 호출될 때 현재 CAR 값을 보존한다. 복귀 주소 저장 역할.
- **순서 제어 모듈(Sequencing Module)**: 마이크로 명령어의 실행 순서를 결정한다. 다음 CAR 값을 계산한다.

```
CU 동작 흐름
IR(기계어) → 명령어 해독기 → CAR 설정
CAR → 제어 기억장치 → CBR(마이크로 명령어)
CBR → ALU/레지스터/메모리에 제어 신호
순서 제어 모듈 → 다음 CAR 계산 → 반복
```

---

## 종합

기계어 명령어 하나가 여러 마이크로 명령어로 구성된다는 것이 CU 구조의 핵심이다. `ADD AX, BX` 한 줄이 "MAR에 주소 로드 → MBR에서 데이터 읽기 → ALU에 덧셈 신호 → 결과 AC에 저장"이라는 여러 마이크로 명령어 시퀀스로 분해된다. 개발자는 기계어 수준까지 내려가지 않지만, JS 엔진이 JIT 컴파일하여 만든 기계어는 이 CU가 마이크로 명령어로 처리한다.
