# Digest

공식문서를 학습하고, 중요한 내용을 Q&A로 knowledge/ 문서에 저장한다.

## 공통 규칙

모든 모드에서 Q&A를 저장할 때 적용되는 규칙이다.

### 내용 분류
- 영어 원문이 있으면 그대로 보존하여 Official Answer로 사용
- **단일 출처 원칙**: Official Answer는 단일 출처의 원문만 사용한다. 여러 페이지의 문장을 하나의 Official Answer로 합성하지 않는다. 출처가 다른 내용을 보충하려면 질문을 분리하거나 Annotation으로 처리한다.
- 한글 메모는 Annotation으로 분류
- **영어 원문 없이 한글만 제공된 경우**: User Answer로 작성. AI가 영어 Official Answer를 생성하지 않는다

### 이미지 alt text 필터링
사용자가 붙여넣은 텍스트에는 이미지의 alt text가 섞여 들어올 수 있다. 주어-서술어 구조가 없는 나열형 문장, 이미지 캡션으로 추정되는 문장은 Official Answer에서 제외한다. 해설 시에도 본문과 구분하여 취급한다.

### 범위 제한
사용자가 제공한 내용의 범위 내에서만 Q&A를 작성한다. 출처 URL을 방문하여 사용자가 언급하지 않은 추가 내용을 가져오지 않는다.

### TODO 매칭
Q&A를 새로 만들기 전에, 기존 knowledge/ 문서에서 TODO를 2단계로 검색한다.

**1단계: `[TODO]` 질문 검색**
- 사용자 내용과 관련된 `[TODO]` 질문이 있는지 검색한다
- **매칭됨**: 해당 `[TODO]` 질문의 Answer 섹션에 내용을 채우고, Questions 목록과 Answers 제목 양쪽에서 `[TODO]` 접두사를 제거

**2단계: 답변/annotation 내 `TODO` 검색**
- 1단계 이후, 답변이나 annotation 본문에 `TODO` 문자열이 포함된 항목을 추가로 검색한다
- **매칭됨**: 해당 `TODO` 부분만 교체/보충한다. 질문이나 전체 답변을 교체하지 않는다

**매칭 안 됨**: 새 Q&A를 작성

TODO 답변을 채울 때, 해당 질문에 대한 **직접적인 답**이 되는 내용을 작성한다. 같은 파일 내 다른 답변의 문장을 그대로 복사하지 않는다.

## 모드 1: 대화형 학습 (ON/OFF)

### ON — 모드 시작

[production-guide.md](../../contexts/production-guide.md)의 **Before** 실행.

출처 URL을 기록하고, WebFetch로 해당 페이지의 텍스트 콘텐츠를 가져와 **원본 참조 텍스트**로 보관한다. 이후 단락 검증에 사용한다.

아래 안내를 출력한다:

```
digest 모드 시작.
단락을 붙여넣으면 해설합니다.
기억할 만한 내용은 질문으로 만들어 제안합니다.
승인하면 즉시 파일에 저장합니다.
나중에 더 파고들 주제가 있으면 TODO 질문을 요청해 주세요.
단락을 붙여넣어 주세요.
```

### 루프 — 단락 처리

사용자가 단락을 붙여넣을 때마다:

#### 0. 원본 페이지 검증

붙여넣은 단락의 핵심 문장이 원본 참조 텍스트에 포함되어 있는지 확인한다.

- **포함됨**: 다음 단계로 진행
- **포함되지 않음**: 해설을 진행하지 않고 아래 안내를 출력한다:

```
이 단락은 현재 URL 페이지에 없는 내용입니다.
다른 페이지의 단락을 붙여넣으셨나요?
새 URL을 알려주시면 해당 페이지로 전환합니다.
```

사용자가 새 URL을 제공하면 WebFetch로 다시 가져와 원본 참조 텍스트를 교체하고, 해당 단락의 해설을 이어서 진행한다.

#### 1. 해설

단락이 사용자의 학습 목적에 가치가 없다고 판단하면 (하드웨어 표준 세부사항, 관리 체계, 사용자 관심 밖의 전문 분야 등), line-by-line 해설 없이 "넘어간다"로 건너뛴다. 사유도 한 줄 이내로 짧게 쓴다.

가치가 있는 단락에 대해서는 정밀 해석(Line-by-Line) 방식으로 원문을 해설한다.

- 영어 원문을 **한 문장씩** `>` 블록쿼트로 인용한다
- 각 문장 바로 아래에 한글로 개발적 맥락을 풍부하게 풀어 설명한다 — 예시, 배경지식, "왜 이렇게 동작하는가", "이게 없으면 어떻게 되는가"까지 포함한다

#### 1-0. 해설 완전성 검증

해설을 사용자에게 출력하기 전에 완전성을 검증한다.

1. 사용자가 붙여넣은 단락에서 문장 수 + 코드 블록 수를 센다
2. 해설의 블록쿼트(`>`) 수와 비교한다
3. 불일치 시 누락분을 채운 뒤 출력한다

#### 1-1. 요약

해설이 끝나면 단락 전체의 핵심을 비유와 함께 3줄 이내로 정리한다.

#### 2. 중복 검사

해설 후, 기억할 가치가 있는 내용이면 질문을 만들기 **전에** 기존 knowledge/ 문서에 동일한 내용이 있는지 확인한다.

- 태그, 파일명, 폴더 기반으로 검색
- 이미 있으면 사용자에게 알리고 건너뛴다

#### 3. 기존 답변 보충 검토

새 질문을 만들기 전에, 현재 단락의 내용이 기존 Q&A의 답변을 보강할 수 있는지 검토한다.

- 기존 답변에 구체적 예시, 안티패턴, 공식 원문 근거를 추가할 수 있으면 Annotation으로 보충을 제안한다
- 새 질문으로 만들 만큼 독립적이지 않지만, 기존 답변을 풍부하게 만드는 내용이 대상이다
- 보충 제안도 사용자 승인 후 즉시 저장한다

#### 4. 질문 제안

중복이 아닌 경우, 질문을 제안한다. 질문 작성 시 아래 원칙을 따른다:

- **면접 질문처럼**: 맥락 없이 읽어도 무슨 말인지 이해되어야 한다. 전문 용어로 냅다 시작하지 말고, 상황/배경을 질문 안에 포함한다.
- **문제→해결 구조**: "이런 문제가 있다. 어떻게 해결하는가?" 형태가 recall에 좋다.
- **리스트 암기 금지**: "3가지는?" 식의 나열 질문 대신, 왜 그런지를 묻는 질문으로 만든다.

질문 구조:
- **새 개념 등장 시 정의 질문 필수**: 문서에서 새로운 기술/개념(예: subagent, agent team)이 처음 등장하면, "X란 무엇인가?" 형태의 기본 정의 질문을 반드시 먼저 제안한다. 심화/메커니즘 질문은 정의 질문 이후에 제안한다.
- **정론 먼저, 꼬리질문은 그 다음**: 먼저 개념 자체를 담백하게 묻는 질문(정론)을 만들고, 그 안에서 꼬집을 수 있는 세부 사항을 꼬리질문으로 붙인다. 오개념 타파 질문을 억지로 만들지 않는다 — 정론이 있어야 꼬리질문이 의미가 있다.
- 부모 질문과 꼬리질문의 Official Answer는 내용이 겹쳐도 괜찮다. 단, 완전히 동일한 복붙은 피한다.

질문 유형:
- 일반 질문: "X란 무엇인가?"
- 오개념 타파: "X는 Y를 하는가?" (답이 아니오인 질문)
- 실무 연결: "X가 실제로 해결하는 문제는?"
- 문제→해결: "이런 문제가 있었다. 어떻게 해결하는가?"
- **예시 해석**: 단락에 실제 코드/설정/프로토콜 예시가 포함된 경우, 예시를 질문 본문에 그대로 싣고 "각 부분의 역할을 설명하라" 형태로 만든다. 복습 시 예시를 보고 직접 해석한 뒤 답을 확인하는 훈련이 된다.

제안할 때 질문과 함께 **한글 해설만** 보여준다. 해설 내 `>` 블록쿼트로 인용된 영어 원문이 곧 저장될 Official Answer이므로, Official Answer 전문을 별도 블록으로 반복하지 않는다.
사용자가 승인하면 아래 **승인 시 즉시 저장** 절차를 실행한다. 거부하면 버린다.

단락마다 무조건 질문을 만들려고 하지 않는다. 스킵 기준은 **"알 필요가 없는 내용"**이다 (하드웨어 표준 세부사항, 관리 체계 등). **"쉬운 내용", "이미 알 만한 내용", "상식"은 스킵 사유가 아니다** — 알아야 하는 내용이면 난이도와 무관하게 질문을 제안한다. 가치가 없다고 판단하면 사유와 함께 "이건 ~한 이유로 학습 안 해도 될 것 같은데 넘어갈까요?"로 사용자에게 확인받는다. 최종 판단은 사용자가 한다.

#### 5. 승인 시 즉시 저장

사용자가 질문을 승인하면 즉시 파일에 저장한다. OFF까지 미루지 않는다.

**세션 첫 승인 시:**
- [production-guide.md](../../contexts/production-guide.md)의 **After** 전체 실행 (파일 배치, 구조 검증, 태그 검증)
- 대상 파일을 결정하고 Q&A를 작성하여 저장

**같은 파일에 추가 승인 시:**
- 기존 파일에 Q&A를 추가 (Questions 목록과 Answers 섹션 모두)
- 구조 검증, 분할 경고만 재실행

**다른 주제/파일 대상 승인 시:**
- 파일 배치를 다시 결정하고 새 파일에 저장

승인 시점에 사용자가 붙여넣은 원문이 대화 맥락에 가까이 있으므로, 원문 대조가 자연스럽게 정확해진다.

### OFF — 모드 종료

모드를 종료한다. 승인된 질문은 이미 파일에 저장되어 있으므로 추가 작업은 없다.

## 모드 2: 즉시 저장

사용자가 학습 내용을 "필기해줘", "기록해줘" 등과 함께 제공하면, 해설 없이 바로 Q&A를 만들어 저장한다.

### 작업 순서

1. [production-guide.md](../../contexts/production-guide.md)의 **Before** 읽기
2. 내용 파악: 공통 규칙(내용 분류, 범위 제한) 적용
3. TODO 매칭: 공통 규칙 적용
4. Q&A 작성: [content-format.md](../../contexts/content-format.md)에 따라 작성
5. [production-guide.md](../../contexts/production-guide.md)의 **After** 실행
