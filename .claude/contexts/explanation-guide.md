# 해설 가이드

digest, review, explain에서 해설·설명·비유·예시를 만들 때 **항상** 적용한다.
특정 단계(Official Answer 해설, 최종 해설 등) 한정이 아니다.
사용자 질문에 답할 때, 재설명할 때, 비유를 바꿀 때도 포함한다.

---

## 1. 포맷 — 영어 원문 해설의 3단계 흐름

영어 공식문서 원문을 한글로 해설할 때 사용하는 구조.

> **본문 작성 전 §2 톤 가이드를 재확인한다.**

### 흐름

1. **도입**: 원문 이해에 필요한 사전 지식.
2. **본문**: 원문을 콤마·and·접속사 단위로 끊고, 끊은 조각마다 그 안의 중요 영단어를 해설.
3. **종합**: 끊은 조각들을 다시 엮어 전체 흐름과 실무 맥락을 설명.

### 도입

- 둘 중 하나(또는 둘 다)를 2~4문장으로 제공한다:
  - **사전 배경지식**: 원문을 이해하기 위해 먼저 알아야 할 개념·용어·맥락.
  - **해설 요약본**: 원문이 전달하는 핵심 주장의 한글 요약.

### 본문

각 조각마다:

1. 원문 조각을 `>` 블록쿼트로 인용한다.
2. 그 안의 중요 영단어를 `- **단어**: 이 문장에서의 의미` 형식으로 해설한다.

**끊는 기준**

- 기본은 콤마·and·접속사 단위.
- 단위가 너무 짧거나 해설할 단어가 없으면 다음 단위까지 묶는다.
- 한 문장이 짧고 명확하면 굳이 쪼개지 않는다.

### 종합

- 끊은 조각들이 어떻게 하나의 전체를 이루는지 3~5문장으로 엮는다.
- 개별 단어 해설의 반복이 아니라, 원문 전체의 뼈대·흐름·의의를 다시 조립한다.
- "왜 이렇게 동작하는가", "이게 없으면 어떻게 되는가", 실무에서 어디서 마주치는가 같은 의역적 맥락을 여기서 통합한다.

## 2. 스타일 — 비유와 예시의 톤

### 예시는 사용자 기술 스택 기반으로

사용자는 프론트엔드(JS/HTML/CSS) 개발자다.
비유나 코드 예시를 들 때 JS/브라우저/Node.js 생태계에서 가져온다.

- ✅ `console.log` 실행 흐름, 이벤트 루프, Web Worker, `async/await`
- ❌ Java의 `synchronized`, C의 `pthread`, Python의 GIL 중심 설명

다른 언어를 언급하는 것 자체는 괜찮지만, 핵심 비유와 첫 예시는 JS 기반이어야 한다.

### 개념 설명 시 JS 흐름으로 풀어주기

추상적인 CS 개념을 설명할 때, 사용자가 익숙한 JS 개발 흐름에 대입하여 설명한다.
실생활 비유보다 **실제 코드 실행 예시**를 우선한다.

예 (프로세스/스레드):
- **프로그램** = `app.js` 파일 (디스크에 저장된 코드). `.exe`와 달리 CPU가 직접 못 읽으니 Node.js가 대신 실행
- **프로세스** = `node app.js` 실행 시 OS가 만든 자원 묶음
- **스레드** = 프로세스 안에서 코드를 한 줄씩 순서대로 실행해나가는 흐름

예 (프로세스 vs 스레드 역할 구분):
- for문 1억번 도는 `app.js`를 `node app.js`로 실행했을 때:
  - **프로세스가 하는 것**: 메모리 할당(`i` 변수 저장 공간), `stdout` 파일 핸들 확보, V8이 변환한 기계어 코드 적재
  - **스레드가 하는 것**: `i = 0` → 조건 확인 → 본문 실행 → `i++` → ... 를 1억번 순서대로 실행
  - 스레드가 `i`를 읽고 쓸 수 있는 건, 프로세스가 메모리를 확보해뒀기 때문

### "이게 없으면 어떻게 되는가"를 함께 설명하기

개념이나 값을 설명할 때, 그것이 없으면 어떤 문제가 생기는지를 함께 제시한다.
"무엇인가"만 설명하면 추상적으로 느껴지지만, "없으면 어떻게 되는가"를 붙이면 존재 이유가 구체화된다.

- 예: "Program Counter 초기값 — 이게 없으면 OS가 CPU를 처음 줄 때 어디서 시작할지 몰라 프로세스를 띄울 수 없다"
- 예: "Memory Limit — 이게 없으면 프로세스 하나가 메모리를 무제한 점유해 다른 프로세스가 죽는다"

종합 단계뿐 아니라 본문의 각 조각 해설에서도 자연스럽게 넣는다.

### Official Answer 핵심 단어의 "왜"를 짚기

Official Answer에서 특정 단어가 선택된 이유를 설명한다.
정의를 외우는 것이 아니라, 왜 그 단어가 쓰였는지 이해시키는 것이 목표다.

- 예: "thread는 왜 sequence인가?" → 명령어가 순서대로 하나씩 이어지는 흐름이니까
- 예: "program은 왜 passive인가?" → 스스로 아무것도 하지 않는 정적 파일이니까

### 좋은 비유는 축적

자율적으로 만든 비유 중 사용자가 별도 교정 없이 받아들인 것은 [analogy-examples.md](./analogy-examples.md)에 즉시 추가한다.

축적 판단 기준:
- (a) 사용자 코드 흐름(JS/Node.js/브라우저)에 매핑된다
- (b) 사용자가 톤을 교정하지 않았다
- (c) 같은 개념의 기존 사례보다 명확하다 — 더 나으면 교체, 비슷하면 생략

형식: `## 개념명` + 비유 본문 1-2단락. 출처 세션 메모는 남기지 않는다.

## 3. 규칙 — 해야/하지 말아야

### 본문 영단어 선별 기준

- `the`·`a` 같은 자명한 관사·기능어는 해설하지 않는다.
- 개발자가 일상적으로 아는 단어(`function`, `return` 등)는 생략한다.
- 직역만으로 의미가 모호하거나, 단어 선택의 의도가 중요한 단어를 우선한다.
- "왜 이 단어가 쓰였는가"를 짚는 것이 목표다 (→ 상세는 §2 "Official Answer 핵심 단어의 '왜'를 짚기").

### 금지

- **영문법 구조 분석 금지**: "이 문장은 A comprises B, C, D 구조다" 같은 구조 분석은 도입·본문·종합 어디에도 넣지 않는다.
- **다음 질문의 답 미리 노출 금지**: 도입에서 뒤이어 나올 질문의 정답을 미리 알려주지 않는다.
