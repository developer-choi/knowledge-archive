# 해설 가이드

digest, review, explain에서 해설·설명·비유·예시를 만들 때 **항상** 적용한다.
특정 단계(Official Answer 해설, 최종 해설 등) 한정이 아니다.
사용자 질문에 답할 때, 재설명할 때, 비유를 바꿀 때도 포함한다.

---

## 0. known→unknown 원칙 (모든 설명에 선행 적용)

좋은 설명의 유일한 조건: 학습자가 어렴풋이라도 아는 것에서 출발해, 아직 모르는 것에 도달하게 한다. 모르는 단어로 설명하면 못 알아듣고, 아는 것을 또 정의하면 지루하다.

### 발판 높이 보정 (설명 생성 전 1회)

1. 학습자 베이스라인: 프론트엔드(JS/HTML/CSS) 개발자. JS·브라우저·Node.js 생태계 개념은 정의 없이 발판으로 쓴다.
2. `explained/` 트리 조회로 베이스라인을 보정한다:
   - 폴더·파일명을 훑어 학습자가 어느 영역을 공부했는지 대략 파악한다(예: `cs/network/` 아래 파일이 많으면 네트워크 기초는 깔려 있다).
   - 설명할 개념과 관련된 explained 파일을 열어 H1 헤딩(질문 제목)만 읽는다. 어떤 개념이 헤딩으로 다뤄졌으면, 그 개념은 학습자가 한 번 이상 설명을 받은(=어렴풋이 아는) 것이므로 발판으로 쓴다.
   - **발판으로 인용하는 헤딩은 파일에서 실제로 읽어 확인한 H1 원문 그대로 쓴다.** "파일이 있으니 그 개념도 다뤘겠지" 식으로 존재하지 않는 헤딩을 만들어내지 않는다. 파일은 있으나 해당 개념의 H1이 없으면 그건 발판이 아니다 — 미학습 용어로 취급한다.
   - 발판이 다루는 **세부 측면**(예: "HTTP가 TCP 위에서 동작")을 헤딩 자리에 적지 않는다. 인용은 H1 제목 원문(예: "HTTP란 무엇인가?")이고, 그 측면은 설명 본문에서 풀어 쓴다. 제목을 측면으로 바꿔 적으면 실제 H1과 어긋난다.

### 설명 중 행동 규칙

- 발판 = 베이스라인 + explained 헤딩에 있는 개념. 이 둘은 정의 없이 비유·선행지식으로 바로 쓴다.
- **발판으로 확인된 개념은 재정의하지 않는다.** "X란 ~이다"로 다시 풀지 말고, 이미 아는 것으로 전제하고 새 개념을 그 위에 얹는다. 아는 것을 또 정의하면 지루하다(known→unknown 위반).
- 베이스라인도 아니고 explained 헤딩에도 없는 용어는, 첫 등장하는 그 자리에서 한 줄로 정의한 뒤 쓴다. 미뤄 던지지 않는다.
- 같은 뜻을 발판 용어로 풀 수 있으면, 미학습 용어를 애초에 꺼내지 않는다.
- **설명 본문에 발판 보정 과정·캐시 여부를 노출하지 않는다.** "이건 이미 설명한 내용이라", "캐시에 있어서" 같은 메타 멘트는 학습자에게 가치가 없다. 보정은 내부에서만 하고, 본문은 곧장 설명으로 들어간다.

## 1. 포맷 — 영어 원문 해설의 3단계 흐름

영어 공식문서 원문을 한글로 해설할 때 사용하는 구조.

> **본문 작성 전 §2 톤 가이드를 재확인한다.**

### 흐름

1. **도입**: 원문 이해에 필요한 사전 지식.
2. **본문**: 원문을 콤마·and·접속사 단위로 끊고, 끊은 조각마다 그 안의 중요 영단어를 해설.
3. **종합**: 끊은 조각들을 다시 엮어 전체 흐름과 실무 맥락을 설명.

### 도입

- 둘 중 하나(또는 둘 다)를 2~4문장으로 제공한다:
  - **사전 배경지식**: 원문을 이해하기 위해 먼저 알아야 할 개념·용어·맥락.
  - **해설 요약본**: 원문이 전달하는 핵심 주장의 한글 요약.

### 본문

각 조각마다:

1. 원문 조각을 `>` 블록쿼트로 인용한다.
2. **한글 의역을 한 줄로 적는다** — 원문 의미를 한국어로 풀어 사용자가 영어 → 한국어 변환 부담을 지지 않게 한다. 단어 해설만으로는 문장 전체의 뜻이 잡히지 않는다.
3. 그 안의 중요 영단어를 `- **단어**: 이 문장에서의 의미` 형식으로 해설한다.

**끊는 기준**

- 기본은 콤마·and·접속사 단위.
- 단위가 너무 짧거나 해설할 단어가 없으면 다음 단위까지 묶는다.
- 한 문장이 짧고 명확하면 굳이 쪼개지 않는다.

### 종합

- 끊은 조각들이 어떻게 하나의 전체를 이루는지 3~5문장으로 엮는다.
- 개별 단어 해설의 반복이 아니라, 원문 전체의 뼈대·흐름·의의를 다시 조립한다.
- "왜 이렇게 동작하는가", "이게 없으면 어떻게 되는가", 실무에서 어디서 마주치는가 같은 의역적 맥락을 여기서 통합한다.
- 개념이 §4 다이어그램 트리거에 걸리면 시각화를 추가한다. 위치는 본문 또는 종합 자율.

## 2. 스타일 — 비유와 예시의 톤

### 예시는 사용자 기술 스택 기반으로

사용자는 프론트엔드(JS/HTML/CSS) 개발자다.
비유나 코드 예시를 들 때 JS/브라우저/Node.js 생태계에서 가져온다.

- ✅ `console.log` 실행 흐름, 이벤트 루프, Web Worker, `async/await`
- ✅ Chrome 동작 방식 (탭 = 별도 프로세스, 메인 프로세스와 탭 프로세스 간 IPC 등) — OS/CS 개념 설명 시 적극 활용
- ✅ Windows Task Manager — 실행 중인 프로세스 목록, PID, CPU/메모리 사용량을 직접 볼 수 있는 도구. 프로세스 개념 설명 시 "Task Manager에서 보이는 그것"으로 연결하면 직관적
- ❌ Java의 `synchronized`, C의 `pthread`, Python의 GIL 중심 설명

다른 언어를 언급하는 것 자체는 괜찮지만, 핵심 비유와 첫 예시는 JS/Chrome 기반이어야 한다.

### 개념 설명 시 JS 흐름으로 풀어주기

추상적인 CS 개념을 설명할 때, 사용자가 익숙한 JS 개발 흐름에 대입하여 설명한다.
실생활 비유보다 **실제 코드 실행 예시**를 우선한다.

예 (프로세스/스레드):
- **프로그램** = `app.js` 파일 (디스크에 저장된 코드). `.exe`와 달리 CPU가 직접 못 읽으니 Node.js가 대신 실행
- **프로세스** = `node app.js` 실행 시 OS가 만든 자원 묶음
- **스레드** = 프로세스 안에서 코드를 한 줄씩 순서대로 실행해나가는 흐름

예 (프로세스 vs 스레드 역할 구분):
- for문 1억번 도는 `app.js`를 `node app.js`로 실행했을 때:
  - **프로세스가 하는 것**: 메모리 할당(`i` 변수 저장 공간), `stdout` 파일 핸들 확보, V8이 변환한 기계어 코드 적재
  - **스레드가 하는 것**: `i = 0` → 조건 확인 → 본문 실행 → `i++` → ... 를 1억번 순서대로 실행
  - 스레드가 `i`를 읽고 쓸 수 있는 건, 프로세스가 메모리를 확보해뒀기 때문

### "이게 없으면 어떻게 되는가"를 함께 설명하기

개념이나 값을 설명할 때, 그것이 없으면 어떤 문제가 생기는지를 함께 제시한다.
"무엇인가"만 설명하면 추상적으로 느껴지지만, "없으면 어떻게 되는가"를 붙이면 존재 이유가 구체화된다.

- 예: "Program Counter 초기값 — 이게 없으면 OS가 CPU를 처음 줄 때 어디서 시작할지 몰라 프로세스를 띄울 수 없다"
- 예: "Memory Limit — 이게 없으면 프로세스 하나가 메모리를 무제한 점유해 다른 프로세스가 죽는다"

종합 단계뿐 아니라 본문의 각 조각 해설에서도 자연스럽게 넣는다.

### Official Answer 핵심 단어의 "왜"를 짚기

Official Answer에서 특정 단어가 선택된 이유를 설명한다.
정의를 외우는 것이 아니라, 왜 그 단어가 쓰였는지 이해시키는 것이 목표다.

- 예: "thread는 왜 sequence인가?" → 명령어가 순서대로 하나씩 이어지는 흐름이니까
- 예: "program은 왜 passive인가?" → 스스로 아무것도 하지 않는 정적 파일이니까

### 좋은 비유는 축적

자율적으로 만든 비유 중 사용자가 별도 교정 없이 받아들인 것은 [analogy-examples.md](./analogy-examples.md)에 즉시 추가한다.

축적 판단 기준:
- (a) 사용자 코드 흐름(JS/Node.js/브라우저)에 매핑된다
- (b) 사용자가 톤을 교정하지 않았다
- (c) 같은 개념의 기존 사례보다 명확하다 — 더 나으면 교체, 비슷하면 생략

형식: `## 개념명` + 비유 본문 1-2단락. 출처 세션 메모는 남기지 않는다.

## 3. 규칙 — 해야/하지 말아야

### 본문 영단어 선별 기준

- `the`·`a` 같은 자명한 관사·기능어는 해설하지 않는다.
- 개발자가 일상적으로 아는 단어(`function`, `return` 등)는 생략한다.
- 직역만으로 의미가 모호하거나, 단어 선택의 의도가 중요한 단어를 우선한다.
- "왜 이 단어가 쓰였는가"를 짚는 것이 목표다 (→ 상세는 §2 "Official Answer 핵심 단어의 '왜'를 짚기").

### 금지

- **영문법 구조 분석 금지**: "이 문장은 A comprises B, C, D 구조다" 같은 구조 분석은 도입·본문·종합 어디에도 넣지 않는다.
- **다음 질문의 답 미리 노출 금지**: 도입에서 뒤이어 나올 질문의 정답을 미리 알려주지 않는다.
- **세션 맥락 표현 금지**: explained 문서는 현재 학습 세션에 없던 독자가 나중에 읽는 참고 문서다. 아래 표현은 금지한다.
  - ❌ "세션 중 헷갈렸던 부분이다", "세션 중 가장 막힌 지점이다", "(세션 중 정밀화한 구분)"
  - ❌ "이번에 헷갈린 게", "아까 물어본 것처럼", "학습 중 큰 의문이었다"
  - 세션에서 나온 오해·재설명의 **내용**은 포함하되, **독자 관점의 보편적 표현**으로 재구성한다.
  - ✅ "흔히 혼동하는 지점이다", "헷갈리기 쉬운 부분이다", "주의할 점이 있다"
- **신조어·불명확한 비유 금지**: 사전에 없거나 뜻이 바로 안 잡히는, AI가 즉석에서 만들어낸 압축 표현·비유(예: "외따로", "맨몸 호출", "오검", "증식")로 설명하지 않는다. 일상어로 먼저 전달하고, 비유가 꼭 필요하면 그 자리에서 풀어 쓴다. (§0의 "미학습 용어는 그 자리에서 정의"가 정식 용어를 다룬다면, 이건 정식 용어도 아닌 즉석 신조어를 겨냥한다.)
  - ❌ "renderHook을 **외따로** 아는 게 아니라" / "**맨몸 호출**은 Invalid hook call" / "예시가 복잡→**오검**·사용사례마다 예시 **증식**"
  - ✅ "renderHook을 **맥락 없이 단독으로** 아는 게 아니라" / "**컴포넌트 없이 직접 호출하면** Invalid hook call" / "예시가 복잡→**훅 아닌 예시 탓에 실패**·사용사례마다 예시가 **늘어남**"
- **작업용 내부 라벨·약칭 노출 금지**: 제안·검토 과정에서 쓰는 임시 기호(`Q1`/`Q2`, `(A)`/`(B)`, `primary`, 코드 실행용 임시 이름 `runSuite` 등)를 설명·대화에 그대로 내보내지 않는다. 학습자에겐 뜻 없는 내부 기호이므로, 가리키려는 대상을 풀어 쓴다.
  - ❌ "Q1의 (A) 케이스를 primary로 저장할까요?" → 사용자: "은어 써서 나한테 알려주지 마. runSuite는 갑자기 나와서 뭔지 모르겠어."
  - ✅ "첫 번째 질문의 '재사용 훅' 경우를 기본 질문으로 저장할까요?"
- **AI 추론을 공식 권장·확립된 사실로 단정 금지**: 공식 문서에 없는 판단·기준을 "공식 권장", "~라고 한다"처럼 출처가 있는 양 서술하지 않는다. 공식 여부가 중요하면 1차 소스를 직접 확인해 인용하고, 확인 못 했으면 추정임을 명시한다.
  - ❌ (사례) test.extend와 beforeEach의 사용 기준을 공식 근거 없이 단정 → 사용자: "이거 전부 공식문서 권장사항이야? 저건 우리 뇌피셜이니까 공신력 있는 곳 찾아봐."
  - ✅ 교정: 공식 문서에서 확인되면 그 원문을 근거로 쓰고, 아니면 "공식 규정은 아니고 통상 이렇게 쓴다"로 구분해 표기.

## 4. 다이어그램

설명 중 아래 트리거에 걸리는 부분이 나오면 **반드시** 다이어그램을 함께 작성한다. 위치는 본문(해당 조각 옆) 또는 종합(전체 시각화) 자율. 한 질문에 여러 트리거가 걸리면 다이어그램도 여러 개 — 개수는 가변이다.

### 트리거 4종

| 트리거 | 언제 | 형식 |
|---|---|---|
| **구조(Structure)** | "구성된다", "포함", "계층" — 요소 2개 이상 | 트리 / 네스티드 박스 |
| **흐름(Flow)** | "→", "이후", "단계", 상태 전이 | 화살표 / 상태도 |
| **공간(Layout)** | 메모리 영역, 레지스터, 디스크 블록 | 박스 레이아웃 |
| **비교(Compare)** | "vs", "차이", "반면" — 2개 이상 대상 | side-by-side 매트릭스 |

### 예시 (구조 트리거)

```
프로세스
├── program code ──────────────────────→ 텍스트 영역
├── system resources
│   └── memory
│       ├── 텍스트 영역   실행 기계어
│       ├── 데이터 영역   전역변수
│       ├── 스택          지역변수·매개변수
│       └── 힙            런타임 동적 할당
├── permissions
└── data structures → PCB (RAM)
    ├── initiate  PID, PC 초기값, 권한 등
    ├── control   CPU 레지스터(CPU 칩) → PCB(RAM) 백업
    └── coordinate 스케줄링·우선순위
```

### 코드블록 fence 규칙

- 다이어그램: **fence 언어 없는** 코드블록 (```` ``` ````)
- 코드 예시: fence 언어 명시 (```` ```js ````, ```` ```py ````, ```` ```ts ```` 등)

이 차이로 review/exam이 다이어그램과 코드 예시를 구분한다.

### 사용자 학습 흐름

- **explain**: AI가 트리거에 맞춰 explained에 다이어그램을 박는다.
- **review/exam**: 트리거 질문 출제 시 "옆에 그려보세요" 안내. 평가 후 explained의 다이어그램을 그대로 인용 → 사용자가 본인 그림과 self-check.

## 5. GOTCHAS — 자주 틀리는 표현

### 비유 vs 예시를 구분해서 써라

- **예시(예시)**: 해당 개념의 실제 인스턴스. OS 문서라면 OS 레벨의 실제 동작이나 시스템 콜이 예시다.
  - 맞음: "I/O 대기 예시 — `read()` 시스템 콜 블로킹"
  - 맞음: "자발적 양보 예시 — `sched_yield()` 시스템 콜"
- **비유(비유)**: 개념을 이해시키기 위해 다른 도메인에서 빌려온 것. JS 코드로 OS 개념을 설명하면 그것은 비유다.
  - 맞음: "비유 — `fs.readFile()` 대기가 OS I/O 블로킹과 비슷한 구조"
  - 틀림: 테이블 "예시" 컬럼에 `fs.readFile()`을 넣는 것 (예시처럼 보이지만 실제론 비유)

실천 규칙:
- 테이블에 "예시" 컬럼을 만들면 OS 레벨 실제 인스턴스만 넣는다.
- JS/브라우저 도메인에서 끌어온 것은 별도 "JS 비유" 섹션에 넣고 명확히 "비유"라고 레이블한다.
