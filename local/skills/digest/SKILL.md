# Digest

공식문서를 학습하고, 중요한 내용을 Q&A로 [`knowledge/`](../local/contexts/directory-roles.md) 문서에 저장한다.

## 공통 규칙

모든 모드에서 Q&A를 저장할 때 적용되는 규칙이다.

### 내용 분류
- 영어 원문이 있으면 그대로 보존하여 Official Answer로 사용
- **원문 합성 금지**: 여러 출처의 문장을 단어 단위로 짜깁기하여 새로운 문장을 만들지 않는다. 단, 서로 다른 출처의 원문 문장을 그대로 이어 붙이는 것은 허용한다.
- **사용자가 직접 작성한 한글 메모**는 User Answer로 분류
- **영어 원문 없이 한글만 제공된 경우**: User Answer로 작성한다 (AI의 OA 신규 생성 금지 — [content-format.md](../local/contexts/content-format.md) 참고).

### 해설은 knowledge에 저장하지 않고, OFF 시 explained로 보낸다

대화형 모드의 line-by-line 해설을 **knowledge/ 파일에 저장하지 않는다** — 비유·통찰 등 보조 내용은 explained/에 담당하며 knowledge/에 중복 보관하지 않는다.

대신 **OFF 시점에 확정된 질문 전체를 `explained/<rel>.md`로 저장**한다. `/explain`과 **동일 양식**(`# <질문>` H1 + `## 도입` / `## 본문` / `## 종합`, H2 사이 `---`)을 쓰고, 세션 중 사용자가 헷갈려서 풀어준 **오해·재설명도 본문에 녹여 포함**한다 (`/explain`이 재설명을 본문에 녹이는 방식과 통일 — 별도 `## 오해 교정` 섹션을 만들지 않는다). 이로써 digest 세션에서만 드러난 사용자 오해가 휘발되지 않고 학습 캐시로 남는다.

소유권: 해당 질문 섹션이 `explained/<rel>.md`에 **이미 있으면 덮어쓰지 않는다** (먼저 만든 쪽 보존). 이후 `/explain`을 돌려도 캐시 hit으로 digest가 만든 섹션을 보존한다 ([explain SKILL](../explain/SKILL.md)의 "캐시 동작" 참고).

### Frontmatter publishable

- **새로 만드는 knowledge/ 파일의 frontmatter에 `publishable: true`를 자동으로 박는다.** digest는 공식문서 1차 소스를 기반으로 하므로 외부 노출 가치가 검증된 입력이다.
- 기존 파일에 추가할 때는 frontmatter를 건드리지 않는다 (이미 있는 값 유지).
- 사용자가 명시적으로 false를 원하면 그 결정을 따른다.

### 이미지 alt text 필터링
사용자가 붙여넣은 텍스트에는 이미지의 alt text가 섞여 들어올 수 있다. 주어-서술어 구조가 없는 나열형 문장, 이미지 캡션으로 추정되는 문장은 Official Answer에서 제외한다. 해설 시에도 본문과 구분하여 취급한다.

### 이미지 다이어그램 처리
사용자가 공식문서의 다이어그램/구조도 이미지 URL을 함께 제공하면, 이미지를 로컬에 저장하지 않고 아래 방식으로 처리한다:
1. WebFetch로 이미지 내용을 파악한다
2. 이미지의 구조를 **텍스트 다이어그램**(ASCII art, 코드블록)으로 변환하여 OFF 시 생성하는 explained/ 섹션에 삽입한다
3. 원본 이미지 URL은 explained/ 섹션 하단에 `원본 다이어그램: [URL]` 형태로 기록한다

이렇게 하면 review/explain 시 이미지를 매번 fetch하지 않고 텍스트로 바로 설명할 수 있다.

### 범위 제한
사용자가 제공한 내용의 범위 내에서만 Q&A를 작성한다. 출처 URL을 방문하여 사용자가 언급하지 않은 추가 내용을 가져오지 않는다.

### User/Additional Answer → OA 교체
새 영어 원문이 들어왔을 때, 기존 knowledge/ 문서에서 Official Answer 없이 User Answer 또는 Additional Answer만 있는 질문과 매칭되면 OA로 교체한다.

- 기존 User Answer / Additional Answer는 삭제한다
- 새 영어 원문을 Official Answer로 저장한다
- 교체 전 사용자에게 확인받는다: "X 질문의 User Answer를 OA로 교체할까요?"

### 기존 미완성 질문 매칭

Q&A를 새로 만들기 전에, 기존 knowledge/ 문서에서 미완성 질문을 2단계로 검색한다. 마커 정의는 [document-structure.md](../local/contexts/document-structure.md)의 "미완성 질문 처리" 참고.

**1단계: `[UNVERIFIED]` 질문 검색**
- 사용자 내용과 관련된 마커 질문이 있는지 검색한다
- **매칭됨**: 새 내용으로 답변을 어떻게 채울지 사용자에게 확인 후 처리한다.
  - 공식 출처 원문이 들어왔다 → Official Answer를 채우고 마커 제거 (Questions 목록·본문 헤딩 양쪽). 기존 Additional Answer가 있으면 삭제한다.
  - AI 추정 답변만 가능하다 → AI Answer를 채우고 마커를 `[UNVERIFIED]`로 변경 (이미 `[UNVERIFIED]`이면 답변 보강만)

**2단계: 답변 본문 내 `TODO` 문자열 검색**
- 1단계 이후, 답변 본문에 `TODO` 문자열이 포함된 항목을 추가로 검색한다
- **매칭됨**: 해당 `TODO` 부분만 교체/보충한다. 질문이나 전체 답변을 교체하지 않는다

**매칭 안 됨**: 새 Q&A를 작성한다. 마커는 아래 "마커 결정" 규칙으로 정한다.

답변을 채울 때, 해당 질문에 대한 **직접적인 답**이 되는 내용을 작성한다. 같은 파일 내 다른 답변의 문장을 그대로 복사하지 않는다.

### 마커 결정

새 Q&A를 만들 때 답변 상태에 따라 마커를 결정한다. 마커 사용 조건과 정의는 [document-structure.md](../local/contexts/document-structure.md)의 "미완성 질문 처리" 참고.

| 상태 | 마커 |
|---|---|
| 공식 출처 원문 또는 후보 Reference 확보 | 없음 |
| 출처 단서 없음 | `[UNVERIFIED]` |

답변 섹션은 비어 있어도, User/Additional Answer가 채워져 있어도 무관하다 (마커는 OA/Reference 부재 상태만을 의미).

### Official Answer 길이 관리

OA는 단락 여러 개로 구성될 수 있다 ([content-format.md](../local/contexts/content-format.md)의 "OA 단락 통합"). 길이 룰은 두 단계로 적용한다:

- **단락 단위**: 한 단락이 6문장을 초과하면 문장 정리 또는 단락 분리를 검토한다.
- **전체 단위**: OA 전체가 15문장을 초과하면 질문 분리를 검토한다.
- 테이블·코드블록·리스트는 문장 수에서 제외한다.
- 최종 판단은 사용자가 한다.

## 대화형 학습 (ON/OFF)

### ON — 모드 시작

[production-guide.md](../local/contexts/production-guide.md)의 **Before** 실행.

출처 URL을 기록하고, WebFetch로 해당 페이지의 텍스트 콘텐츠를 가져와 **원본 참조 텍스트**로 보관한다. 이후 단락 검증에 사용한다.

**기존 출처 검사**: 출처 URL이 기존 knowledge/ 문서의 Reference에 이미 등장하는지 검색한다. 등장하면 해당 Q&A 목록을 보여주고 "이 URL은 이미 N건의 Q&A에서 참조됩니다. 계속할까요?"로 확인받는다.

아래 안내를 출력한다:

```
digest 모드 시작.
단락을 붙여넣으면 해설합니다.
기억할 만한 내용은 질문으로 만들어 제안합니다.
승인하면 즉시 파일에 저장합니다.
단락을 붙여넣어 주세요.
```

### 루프 — 단락 처리

사용자가 단락을 붙여넣을 때마다:

#### 0. 원본 페이지 검증

붙여넣은 단락의 핵심 문장이 원본 참조 텍스트에 포함되어 있는지 확인한다.

- **포함됨**: 다음 단계로 진행
- **포함되지 않음**: 해설을 진행하지 않고 아래 안내를 출력한다:

```
이 단락은 현재 URL 페이지에 없는 내용입니다.
다른 페이지의 단락을 붙여넣으셨나요?
새 URL을 알려주시면 해당 페이지로 전환합니다.
```

사용자가 새 URL을 제공하면 WebFetch로 다시 가져와 원본 참조 텍스트를 교체하고, 해당 단락의 해설을 이어서 진행한다.

#### 1. 해설

**단락 전체가 학습 가치 없으면** (하드웨어 표준 세부사항, 관리 체계, 사용자 관심 밖의 전문 분야 등), line-by-line 해설 없이 "넘어간다"로 건너뛴다. 사유도 한 줄 이내로 짧게 쓴다.

**가치 있는 단락**은 [explanation-guide.md](../local/contexts/explanation-guide.md)를 따라 해설하되, 단락 안에 **일부 학습 불필요한 토픽**이 섞여 있으면 그 토픽만 건너뛴다. 건너뛴 토픽은 해설을 다 출력한 뒤 맨 아래에 묶어 통지한다:

```
──── 넘어간 것 ────
· [토픽 이름] — [한 줄 사유] (필요하면 말씀하세요)
```

- **토픽 이름은 구체적으로**. 사용자가 무엇을 건너뛰는지 알아보고 "그건 필요한데"를 그 자리에서 판단할 수 있어야 한다. "기타 세부사항" 같이 뭉뚱그리지 않는다.
- **토픽 단위로 한 줄씩**. 문장마다 달지 않는다 (스킵 로그가 해설만큼 길어진다).
- 건너뛴 토픽은 **질문 제안에서도 제외**한다.
- "학습 불필요" 기준은 §4 「스킵 판단」과 같다 (알 필요 없는 내용. 쉬운 내용·상식은 스킵 사유 아님).

#### 1-0. 누락 방지 검증

해설을 출력하기 전에, 붙여넣은 단락의 모든 문장·코드블록이 다음 둘 중 하나로 처리됐는지 확인한다:

- **해설됨** — 블록쿼트로 인용·설명됨 (이미 아는 발판 개념은 재정의 없이 짧게 짚는 것도 포함)
- **스킵 통지됨** — §1의 "넘어간 것" 묶음에 토픽으로 포함됨

어느 쪽도 아닌 문장 — 즉 *말없이* 빠진 문장 — 이 있으면 둘 중 하나에 배정한 뒤 출력한다.

목적은 "모든 문장을 해설하라"가 아니라 "어떤 문장도 말없이 사라지지 않게 하라"다. 학습 불필요한 내용은 해설하지 말고 스킵 통지로 보낸다.

#### 1-1. 요약

해설이 끝나면 단락 전체의 핵심을 비유와 함께 3줄 이내로 정리한다.

#### 출력 규칙

해설 후 제안 단계(중복 검사, 보충 검토, 니치 분류 등)의 판단 과정은 내부적으로 수행하되 사용자에게 보여주지 않는다. 사용자에게는 최종 결과만 출력한다:
- **제안 있음**: 질문/보충 제안을 바로 보여준다
- **제안 없음**: "넘어간다 — [사유 한 줄]"

#### 2. 중복 검사

해설 후, 기억할 가치가 있는 내용이면 질문을 만들기 **전에** 기존 knowledge/ 문서에 동일한 내용이 있는지 확인한다.

- 태그, 파일명, 폴더 기반으로 검색
- 기존 Official Answer와 **완전히 동일한 문장**이면 스킵한다
- **같은 개념이지만 다른 출처의 다른 표현**(더 쉬운 설명, 다른 관점, 보충이 되는 문장)이면 스킵하지 않고 보충 검토(3단계)로 넘긴다
- 중복으로 판단하여 새 질문을 만들지 않더라도, 보충 검토(3단계)는 반드시 수행하고 결과를 보고한다. 보충할 내용도 없을 때만 "넘어간다"로 최종 스킵한다.

#### 3. 기존 답변 보충 검토

새 질문을 만들기 전에, 현재 단락의 내용이 기존 Q&A의 답변을 보강할 수 있는지 검토한다.

- **다른 출처의 Official 원문**이 기존 Official Answer와 같은 개념을 다른 표현·각도로 설명하면, 기존 OA에 단락으로 이어 붙이는 보충을 적극 제안한다. 새 출처 URL은 Reference 섹션에 추가한다 (OA 본문에 인라인 URL을 박지 않는다 — [content-format.md](../local/contexts/content-format.md)의 "출처 표기는 Reference에만" 참고).
- 기존 답변에 구체적 예시·안티패턴 등 비공식 보충이 필요하면 explained/ 보충을 제안한다. OFF 시 생성하는 explained/ 섹션에 반영한다.
- 새 질문으로 만들 만큼 독립적이지 않지만, 기존 답변을 풍부하게 만드는 내용이 대상이다.
- 보충 제안도 사용자 승인 후 즉시 저장한다.

#### 4. 질문 제안

중복이 아닌 경우, 질문을 제안한다. 질문 작성 시 아래 원칙을 따른다:

- **면접 질문처럼**: 맥락 없이 읽어도 무슨 말인지 이해되어야 한다. 전문 용어로 냅다 시작하지 말고, 상황/배경을 질문 안에 포함한다.
- **문제→해결 구조**: "이런 문제가 있다. 어떻게 해결하는가?" 형태가 recall에 좋다.
- **리스트 암기 금지**: "3가지는?", "핵심 특성은?", "장단점은?" 식의 나열 질문 금지. 사용자에게 키워드 목록을 외우라는 꼴이 된다. 대신 각 항목을 개별 질문으로 분리하거나, 질문에 키워드를 언급하여 이해를 묻는 형태로 만든다.
  - ❌ "BST의 핵심 특성은?" → 사용자가 Ordering Property, Recursive Nature, No Duplicates... 를 줄줄 외워야 함
  - ✅ "BST에서 임의의 노드를 골랐을 때, 왼쪽 서브트리와 오른쪽 서브트리의 값은 어떤 관계인가?" → 키워드(ordering property)를 이해하고 설명하는 질문
- **정답 키워드 누설 금지**: 사용자가 떠올려야 할 답의 핵심 요소(반환값의 필드명·구성 요소·용어 등)를 질문 문장에 미리 적지 않는다. 키워드를 질문이 불러주면 recall 훈련이 무력화된다. 막연히 묻고, 그 키워드 산출을 사용자 몫으로 남긴다.
  - ❌ "next()는 무엇을 반환하며 `value`·`done`은 각각 무슨 의미인가?" → 답인 `value`·`done`을 질문이 미리 노출
  - ✅ "next()는 무엇을 반환해야 하며, 그 반환 객체 안의 필드들은 각각 무엇을 의미하는가?" → 사용자가 `value`·`done`을 스스로 떠올려 설명
  - 단, 그 용어 자체가 답이 아니라 질문의 **전제·상황**일 때는 언급해도 된다 (위 ordering property 예처럼 "무엇을 이해하는지"를 묻는 경우와 구분).

질문 구조:
- **기본 우선 원칙**: 단락에 정의·사용법·비교·응용이 함께 있으면 정의 → 사용법 → 비교/응용 순으로 제안한다. 비교·응용·오개념 타파 질문은 같은 단락의 기본 정의 질문이 모두 제안된 뒤에만 제시한다. "비교가 더 흥미롭다"는 이유로 정의를 건너뛰지 않는다.
  - ❌ (이번 세션 사례) AWS API/RESTful API 단락 첫 제안에서 "REST API와 일반 API 차이?", "DB 직접 공개 안 하는 이유?" 같은 비교/응용 질문을 정의 질문 없이 메인으로 제안 → 사용자 지적: "각각 개념이 뭔지를 먼저 확정하고나서 그런 질문을 던지는게 좋지 않나?"
  - ✅ 교정: "API란 무엇인가?", "RESTful API란 무엇인가?", "클라이언트란?", "리소스란?" 같은 정의 질문을 먼저 제안하고, 비교·응용 질문은 그 뒤에 꼬리로 매단다.
- **새 개념 등장 시 정의 질문 필수**: 문서에서 새로운 기술/개념(예: subagent, agent team)이 처음 등장하면, "X란 무엇인가?" 형태의 기본 정의 질문을 반드시 먼저 제안한다. 심화/메커니즘 질문은 정의 질문 이후에 제안한다.
- **정론 먼저, 꼬리질문은 그 다음**: 먼저 개념 자체를 담백하게 묻는 질문(정론)을 만들고, 그 안에서 꼬집을 수 있는 세부 사항을 꼬리질문으로 붙인다. 오개념 타파 질문을 억지로 만들지 않는다 — 정론이 있어야 꼬리질문이 의미가 있다.
- **위계 구조 일관성 (세션 단위)**: 같은 세션에서 사용자가 "톱레벨 + 꼬리질문" 같은 위계 구조를 확립하거나 명시적으로 요구하면, 이후 단락의 질문 제안에서도 같은 위계로 자발적으로 구성한다. 새 단락이라고 flat 리스트로 돌아가지 않는다. 한 단락에서 질문이 5개 이상 나올 것 같으면 flat 대신 톱레벨 + 꼬리질문 위계로 묶는 것을 기본으로 한다.
  - ❌ (이번 세션 사례) 1단락(API/RESTful API)에서 사용자가 톱레벨 + 꼬리질문 위계를 확립한 뒤, 2단락(REST 5가지 원칙)에서 다시 flat 8개("TODO 매칭 ①②③ + 신규 ①②③④⑤")로 제안 → 사용자 지적: "처음에 큰범위 물어보고 꼬리질문을 여러개 하게 해서 좀 질문을 구조화하자. 아까 위에서 했던거처럼"
  - ✅ 교정: 톱레벨 "REST란?" + 꼬리 2개(프로토콜vs스타일, representation), 톱레벨 "REST 핵심 원칙들은?" + 꼬리 5개(각 원칙)로 위계화.
- 부모 질문과 꼬리질문의 Official Answer는 내용이 겹쳐도 괜찮다. 단, 완전히 동일한 복붙은 피한다.

질문 유형:
- 일반 질문: "X란 무엇인가?"
- 오개념 타파: "X는 Y를 하는가?" (답이 아니오인 질문)
- 실무 연결: "X가 실제로 해결하는 문제는?"
- 문제→해결: "이런 문제가 있었다. 어떻게 해결하는가?"
- **예시 해석**: 단락에 실제 코드/설정/프로토콜 예시가 포함된 경우, 예시를 질문 본문에 그대로 싣고 "각 부분의 역할을 설명하라" 형태로 만든다. 복습 시 예시를 보고 직접 해석한 뒤 답을 확인하는 훈련이 된다.

- **중요도 사전 판단 (내부 전용)**: 질문을 제안할 때, 출처의 성격에 따라 적절한 중요도 기준을 적용하여 **내부적으로** 필터링한다. 중요도 라벨은 사용자에게 **보여주지 않는다**.
  - **라이브러리/프레임워크 공식문서**: "핵심 API" (거의 매번 쓰는 기능), "자주 사용" (알아두면 유용), "고급/니치" (특수 상황에서만 필요) 로 구분한다.
  - **CS 개념/이론 문서**: "면접 핵심", "CS 배경지식", "니치/선택" 으로 구분한다.
  - 출처 URL의 도메인과 경로(예: `/docs/`, API 레퍼런스 등)로 자동 판단한다. 판단이 애매하면 첫 단락 제안 시 사용자에게 확인한다.
  - 한 단락에서 중요도가 섞여 있으면 핵심만 기본 제안하고, 나머지는 선택으로 분리한다.
  - **제안 전 자기 검증**: 각 질문에 대해 두 가지를 자문한다: (1) "이걸 알면 사용자가 뭘 새로 할 수 있는가?" — 새로운 기능/패턴/가능성이 열리는 내용을 우선 제안한다. (2) "이걸 몰랐을 때 실무에서 어떤 문제가 생기는가?" — 답이 약하면 니치로 분류한다. 니치 항목은 기본 제안에서 제외하고, 사용자가 요청할 때만 제안한다.

- **해설 통찰도 질문 후보**: 해설 과정에서 도출된 통찰(왜 이런 설계인가, 기존 방식과의 차이, 실무적 의미 등)도 질문 후보로 검토한다. 통찰에 대응하는 영어 원문을 Official Answer로, 통찰 자체는 OFF 시 생성하는 explained/ 섹션에 보충으로 기록한다.

- **스킵 판단**: 단락마다 무조건 질문을 만들려고 하지 않는다. 스킵 기준은 **"알 필요가 없는 내용"**이다 (하드웨어 표준 세부사항, 관리 체계 등). **"쉬운 내용", "이미 알 만한 내용", "상식"은 스킵 사유가 아니다** — 알아야 하는 내용이면 난이도와 무관하게 질문을 제안한다. **"나중에 더 자세히 배울 것 같다"도 스킵 사유가 아니다** — 필요한 내용이면 지금 추가하고, 나중에 깊이 배우면 그때 분리하거나 보충한다. 가치가 없다고 판단하면 사유와 함께 "이건 ~한 이유로 학습 안 해도 될 것 같은데 넘어갈까요?"로 사용자에게 확인받는다. 최종 판단은 사용자가 한다.

  **사용자 역할 기준 필터링**: 사용자의 역할·배경(예: 프론트엔드 개발자, AI 사용자)이 명확하면 그 역할과 무관한 전문 분야 내용은 기본 skip 후보로 분류한다.
  - 비개발 도메인 적용 사례(의료·반도체·물류 등) → 개발자 사용자에게 skip
  - 구현·운영 레벨 세부사항(모델 내부 회로, 인프라 설계 등) → 일반 사용자·소비자 역할에게 skip

#### 4-1. 제안 출력 포맷

질문을 제안할 때 사용자에게 보여주는 것은 **딱 3가지**다:

1. **한글 질문**: 저장될 질문 문장
2. **한글 해설**: 질문에 대한 답의 핵심을 한글로 풀어 설명 (영어 원문 번역 + 요지). 저장 시 Official Answer가 될 영어 원문은 **이 단계에서 보여주지 않는다** — AI가 해설 작성의 근거로 내부 보관만 한다.
3. **제안 이유**: 이 질문을 왜 제안했는지 한 줄 (recall 가치, 실무 활용, 오개념 교정 등 구체적으로)

**보여주지 않는 것 (내부 처리)**:
- 영어 원문 (Official Answer 후보) — 번역해서 해설로만 노출
- 중요도 라벨 — 내부 필터링에만 사용

사용자가 승인하면 아래 **승인 시 즉시 저장** 절차를 실행한다. 거부하면 버린다.

#### 5. 승인 시 즉시 저장

사용자가 질문을 승인하면 즉시 파일에 저장한다. OFF까지 미루지 않는다.

**세션 첫 승인 시:**
- [production-guide.md](../local/contexts/production-guide.md)의 **After** 전체 실행 (파일 배치, 구조 검증, 태그 검증)
- 대상 파일을 결정하고 Q&A를 작성하여 저장

**같은 파일에 추가 승인 시:**
- 기존 파일에 Q&A를 추가 (Questions 목록과 Answers 섹션 모두)
- 구조 검증만 재실행

**다른 주제/파일 대상 승인 시:**
- 파일 배치를 다시 결정하고 새 파일에 저장

승인 시점에 사용자가 붙여넣은 원문이 대화 맥락에 가까이 있으므로, 원문 대조가 자연스럽게 정확해진다.

### OFF — 모드 종료

모드를 종료한다. 승인된 질문은 이미 파일에 저장되어 있으므로 아래 절차만 수행한다.

1. 이번 세션에서 변경한 knowledge 파일 목록과 요약(새 질문 N건, 보충 N건)을 보고한다
2. **explained 생성 — 별도 에이전트 위임**: 아래 정보를 구성하여 별도 에이전트에 전달하고, 에이전트가 `explained/<rel>.md`를 작성하게 한다. 에이전트는 이 세션의 대화 맥락을 직접 보지 않으므로, 세션 지칭 표현 없이 독자 관점으로 explained를 생성한다.

   **에이전트에 전달할 내용 (세션 요약):**
   - 대상 파일 경로: `explained/<rel>.md`
   - 확정된 질문 목록 (OA 원문 포함)
   - 질문별 사용자 오해·막힌 지점·재설명 내용 — 세션 맥락 표현 없이 **사실만** 요약 (예: "사용자가 collapse와 collision의 차이를 혼동했다. collapse는 여럿이 하나로 줄어드는 것이고 collision은 둘이 부딪혀도 둘이 유지된다는 점을 재설명했다")
   - 데모·이미지 등 생성한 자산 경로 (있으면)
   - 적용 규칙: [explanation-guide.md](../local/contexts/explanation-guide.md), [explain SKILL](../explain/SKILL.md)의 캐시 동작

   **에이전트 지시:**
   - 각 질문에 대해 `# <질문>` H1 + `## 도입` / `## 본문` / `## 종합` 양식으로 explained 섹션 작성
   - 이미 있는 섹션은 덮어쓰지 않고 보존
   - 데모·이미지 자산은 `assets/<rel>/`에서 explained 본문으로 상대 경로 링크
3. [production-guide.md](../local/contexts/production-guide.md)의 **스킬 종료 시** 실행 (분할 경고 등)
4. **학습가치 판단 회고**: 이번 세션에 스킵/keep 판단이 뒤집힌 적(내가 넘긴 걸 사용자가 도로 달라 했거나, 해설·제안한 걸 불필요로 반려)이 있으면 `/pre-exit digest`를 제안한다. 그 번복 유형을 §4 「스킵 판단」 기준에 반영해 다음 세션 판단을 다듬기 위함이다. 번복이 없었으면 제안하지 않는다.
