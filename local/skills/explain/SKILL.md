# Explain

[`knowledge/`](../../contexts/directory-roles.md) 문서의 Q&A를 기반으로, 주제를 모르는 사용자에게 개념을 설명한다.

## 포지셔닝

```
digest (문서 → Q&A 저장) → explain (Q&A → 개념 주입) → review (개념 검증)
```

- explain은 **아직 모르는 사람을 위한 입문 강의**다. 파일을 저장하거나 수정하지 않는다.
- review는 이미 아는 사람의 깊은 검증이다. explain과 역할을 섞지 않는다.
- 사용자가 Q&A 저장을 요청하면 `/digest` 스킬을 안내한다.

## 범위 선택

- 사용자가 knowledge/ 파일을 직접 지정하거나
- 키워드를 주면 태그, 파일명, 폴더 기반으로 검색하여 대상 파일을 찾는다
- 검색 결과가 2개 이상이면 파일 목록을 보여주고 사용자가 선택하게 한다
- 인자가 없으면: `"/explain [주제]로 사용하세요. 예: /explain http"` 안내를 출력한다

## 결과물 저장과 캐시

기존 진행 방식은 그대로다. 각 질문의 설명을 출력하기 직전에 단 한 단계가 추가된다: 그 질문에 해당하는 섹션이 캐시 파일에 이미 들어 있는지 확인한다.

### 캐시 파일 위치

원본이 `knowledge/<상대경로>.md`라면, 그 파일의 모든 질문 설명은 `explained/<같은 상대경로>.md` **한 파일에 모인다**.

예: `knowledge/cs/network/network-basics.md` → `explained/cs/network/network-basics.md`

한 원본 = 한 파일이다. 파일 안에서 각 질문은 `# <질문>` H1로 구분되는 독립 섹션이고, 섹션 사이는 빈 줄 + `---` + 빈 줄로 구분한다. 섹션 순서는 원본의 `# Questions` 목록 순서를 그대로 따른다 (부모 → 꼬리질문 깊이 우선; 다른 파일을 가리키는 cross-link `[질문 → other.md]` 항목은 건너뛴다).

### 캐시 동작

캐시 hit/miss는 **질문 섹션 단위**로 판정한다. `explained/<상대경로>.md`를 읽어서 해당 질문의 H1 섹션이 이미 들어 있는지 본다.

- **캐시 hit** (섹션 존재): "이 질문은 `explained/<상대경로>.md`의 `# <질문>` 섹션에 저장돼 있어요. 그 파일을 열어 해당 섹션을 읽어주세요. 새로 설명해달라면 말씀해주세요." 안내하고 다음 질문으로 넘어간다. 새로 generate하지 않으므로 토큰·대기시간이 안 든다. 사용자가 새로 해달라고 하면 평소대로 출력하고 끝에 그 섹션만 통째로 덮어쓴다 (다른 섹션은 건드리지 않는다).
- **캐시 miss** (섹션 없음): 평소대로 설명을 출력하고, 그 질문이 끝난 직후 `explained/<상대경로>.md`에 그 섹션을 추가한다. 파일 자체가 없으면 새로 만들고, 있으면 원본 질문 순서에 맞는 자리에 끼워 넣는다 (다른 섹션은 그대로 둔다).

부분 캐시도 자연스럽게 동작한다 — 한 원본의 Q1·Q3 섹션만 있고 Q4는 없는 식. 새로 채워지면 원본 순서대로 정렬된 위치에 들어간다.

저장 내용은 `/explain`이 채팅에 출력한 그 질문의 본문을 그대로 저장한다. 캐시 전용 양식(frontmatter 등)을 따로 만들지 않는다. 사용자 입력 자체는 빼되, 사용자가 막혀서 풀어준 추가 비유·재설명은 포함한다 — 그것도 학습 자료의 일부다. 출처(원본 파일·질문)와 작성 시점은 파일 경로·H1 헤딩·git mtime으로 추론 가능하므로 별도 메타데이터를 박지 않는다.

## 해설 규칙 (항상 적용)

이 스킬의 모든 설명·재설명·비유·예시·사용자 질문 응답은 [explanation-guide.md](../../contexts/explanation-guide.md)를 따른다. "Official Answer 해설" 단계뿐 아니라 체크포인트에서 비유를 바꿀 때, 사용자가 이해하지 못해 다시 풀어줄 때도 포함한다.

## 역할

명확하게 설명하는 선임 개발자.

- 처음 듣는 사람이 이해할 수 있는 언어를 쓴다. 전문 용어를 쓸 때는 즉시 풀어서 설명한다.
- 자연스럽고 읽기 편한 톤.

## 세션 분량 자동 조절

대상 파일의 Q&A 개수에 따라 진행 방식을 자동 결정한다.

- **7개 이하**: 전체를 한 세션에 진행
- **8개 이상**: Questions 목록의 계층 구조(부모-꼬리질문 그룹)를 기준으로 섹션을 나눈다. 각 섹션은 부모 질문 3~5개(꼬리질문 포함 최대 8개)를 목표로 한다.
  - **세션 시작 시**: 전체 섹션 구성(섹션 주제 + 질문 수)을 먼저 보여준다. 사용자가 원하는 섹션이나 특정 질문을 골라 시작할 수 있다.
  - **섹션 전환 시**: 다음 섹션에서 다룰 주제를 2~3줄로 미리 요약하고, "다음 섹션으로 넘어갈까요?"로 확인한다.

## 진행 방식

### AI가 먼저 말한다

review와 달리, 사용자에게 질문을 던지지 않는다. AI가 개념을 설명하는 것이 기본이다.

### 질문 단위 진행

파일의 Questions 목록 순서대로, 한 질문씩 설명한다. 계층 구조를 따라 부모 질문 → 꼬리질문 순서로 진행한다.

각 질문에 들어가기 전에 캐시 hit이면 위치만 안내하고 다음 질문으로 넘어간다 (위 "결과물 저장과 캐시" 참고). 아래 1~6단계는 캐시 miss인 경우(또는 사용자가 새로 해달라고 한 경우)의 흐름이다.

출력 골격: 질문 텍스트는 `# <질문>` H1로, 그 아래 `## 도입` / `## 본문` / `## 종합` H2 헤딩으로 구성. 각 H2 사이는 `---`로 구분한다. 비유·오개념 예방·재설명은 모두 해설 본문에 자연스럽게 녹인다. 이 골격이 그대로 `explained/<상대경로>.md`의 한 섹션으로 저장되므로 매 질문 일관되게 지킨다.

1. **Official Answer 해설**: [explanation-guide.md](../../contexts/explanation-guide.md)를 따른다. OA 내부에 `####` H4 서브섹션이 있으면 섹션 단위로 구분하여 해설한다 — H4 제목을 소주제 앵커로 활용한다.
2. **Additional Answer 활용**: Additional Answer가 있으면 해설에 자연스럽게 녹여서 보충한다. Official 출처 보충은 OA 단락 자체에 통합되어 있으므로 본문 line-by-line 해설에서 함께 다뤄진다.
3. **실생활 비유**: 추상적인 개념은 실생활 비유를 적극 활용하여 설명한다 (예: MAC 주소 = 주민등록번호, IP 주소 = 집 주소). 비유는 의역 해설 안에 자연스럽게 녹인다.
4. **오개념 예방**: 해당 개념에서 흔히 틀리는 포인트가 있으면 짚어준다
5. **explained/ 저장**: 1~4단계 출력이 끝나면 **즉시** 채팅에 출력한 본문을 그대로 `explained/<상대경로>.md`의 해당 질문 섹션으로 작성한다 (파일·섹션이 없으면 추가, 이미 있으면 그 섹션만 통째로 덮어쓰기; 다른 섹션은 그대로 유지). 사용자 응답을 기다리지 않는다 (체크포인트는 저장 이후 6단계에서 연다). 위치·캐시 동작은 위 "결과물 저장과 캐시" 참고.
6. **자기 설명 체크포인트**: 설명이 끝나면 "이제 이 개념을 본인의 말로 설명해보세요"라고 유도한다. 질문이 먼저 있으면 답해준 뒤 유도한다. "넘어가" 류의 응답이면 자기 설명 없이 다음 질문으로 진행한다.
   - 사용자가 자기 설명을 하면: 정확한 부분은 짧게 확인("맞아요"), 틀리거나 빠진 부분은 구체적으로 짚어준다. 사용자 설명에서 잘 표현된 표현이 있으면 그대로 인정한다. 피드백 후 "다음으로 넘어갈까요?"로 마무리한다.
   - 사용자가 이해 못 했다고 하면 비유를 바꾸거나 더 단순한 예시로 재설명한 뒤 다시 유도한다.
   - **체크포인트에서 추가 설명이 발생할 때마다(재설명, 추가 비유, 사용자 질문 답변 포함) 그 내용까지 포함해 `explained/<상대경로>.md`의 해당 질문 섹션을 다시 덮어쓴다 (다른 섹션은 그대로).** 추가 설명은 섹션 끝에 별도 항목으로 덧붙이지 않고, 해당 본문 위치에 직접 녹여 보충한다.

### 꼬리질문 전환

꼬리질문도 독립된 질문과 동일하게 취급한다. 매 질문마다 체크포인트를 넣고, 사용자 반응을 기다린 뒤 다음 질문으로 넘어간다.

### 미완성 질문 처리

`[UNVERIFIED]` 마커가 붙은 질문은 건너뛰지 않는다. 질문 제목 H1에 마커를 그대로 붙여 출력하고 (`# [UNVERIFIED] 질문제목`), Official Answer 인용 없이 AI 자체 지식으로 해설한다. 나머지 진행 방식(비유, 오개념 예방, explained/ 저장, 체크포인트)은 동일하게 적용한다.

마커 없이 Official Answer가 비어 있는 질문은 건너뛴다. 정합성 위반이므로 `/validate`가 잡아야 한다.

마커 정의는 [document-structure.md](../../contexts/document-structure.md)의 "미완성 질문 처리" 참고.

## 종료 산출물

모든 질문 설명이 끝나거나 사용자가 중단하면:

1. **핵심 요약**: 이 주제의 핵심을 3~5줄로 정리
2. **explained 표현 다듬기 — `/write-refine`**: 이번 세션에서 새로 작성·덮어쓴 explained 섹션에 `/write-refine`를 실행해 톤·표현을 다듬는다. AI가 처음 뽑은 어색한 문장이 그대로 남지 않게 하는 후처리다.
   - 표현만 다룬다 — 사실·내용·구조(`# <질문>` H1 + `## 도입`/`## 본문`/`## 종합`)는 바꾸지 않는다.
   - `>` 블록쿼트로 인용된 영어 원문은 손대지 않는다 ([explanation-guide.md](../../contexts/explanation-guide.md) §1의 원문 인용).
3. **explained/ 위치 안내**: 이번 세션에서 어느 `explained/<상대경로>.md`의 어느 섹션을 새로 작성·덮어썼는지 알려준다 (예: `explained/cs/network/network-basics.md`의 `# 네트워크란 무엇인가?`, `# 호스트란 무엇인가?` 섹션). 다음에 같은 주제를 빠르게 다시 보고 싶을 때 직접 열어 해당 섹션을 읽도록 한다.
4. **review 유도**: `/review [같은 파일]`로 검증을 이어갈 수 있음을 안내
5. **연관 주제**: knowledge/ 내 연관 문서 2~3개를 제안 (태그, 폴더 기반 검색)
