# 파일과 파일 디스크립터(fd)

> 프로세스 학습 중 "system resources" 항목에서 분기한 보충 노트.
> 프로세스 본질이 아니라 OS 인터페이스 설계 이야기라 explained 본 파일과 분리.

## 배경 — 어디서 막혔는가

프로세스가 화면에 출력하려면 OS를 거쳐야 하고, OS에게 부탁할 때 쓰는 "창구"가 파일 디스크립터(fd). 1번 = 터미널 화면. 여기까지는 이해. 다음이 막힘:

- 파일은 디스크에 저장된 데이터 덩어리인데, "터미널이 파일이다"는 말이 왜 나오나?
- fd 외에 다른 창구들(소켓, 파이프 등)이 있다는데 전체 그림이 안 잡힘
- fd라는 단어 자체가 뭔지, 왜 "디스크립터"라는 이름인지

---

## "파일"이라는 단어의 두 가지 뜻

평소 쓰는 **파일** = 디스크에 저장된 데이터 덩어리 (`a.txt`, `chrome.exe`). 일상적 의미.

근데 Unix(Linux·macOS의 조상)는 "파일"을 훨씬 넓게 정의한다.

> **데이터를 read/write 시스템 콜로 주고받을 수 있는 모든 대상** = 파일

정확히 분리하면:
- **파일 = "대상"** (read/write의 대상)
- **인터페이스 = read/write 시스템 콜 그 자체** (대상에 적용하는 동작)

왜 이런 넓은 정의가 나왔냐면, 1970년대 Unix 설계자들의 발상:

- 프로그램이 데이터를 주고받는 대상이 너무 많다 — 디스크, 키보드, 화면, 프린터, 네트워크 다른 컴퓨터, 다른 프로세스...
- 각각마다 다른 API를 만들면? `readDisk()`, `readKeyboard()`, `recvNetwork()`, `pipeReceive()`, `printerSend()`... 코드가 폭발.
- "전부 같은 인터페이스로 통일하자. 어차피 동작은 read랑 write 두 가지뿐."
- → **everything is a file** 철학.

여기서 "파일"은 **인터페이스(read/write)를 공유하는 대상들의 부류**라는 의미.

> **시스템 콜** = 프로세스가 OS에게 부탁할 때 쓰는 정해진 함수 호출. `read(fd, ...)`, `write(fd, ...)` 등.

---

## "터미널이 파일이다"의 진짜 의미

터미널 = 키보드 입력 받고 화면에 글자 출력하는 장치. 입출력이 일어나는 대상.

Unix는 이걸 "키보드 = 읽을 수 있는 파일, 화면 = 쓸 수 있는 파일"로 본다. **디스크에 저장된다는 뜻이 절대 아님.** "디스크 파일과 똑같이 read/write 시스템 콜로 다룬다"는 의미.

JS로 보면 셋 다 내부적으로 같은 read/write를 fd만 바꿔서 호출:

```js
fs.readFile('a.txt', cb)        // 디스크 파일 읽기  → read(fd=3, ...)
process.stdin.on('data', cb)    // 키보드 읽기       → read(fd=0, ...)
process.stdout.write('hi')      // 화면 쓰기         → write(fd=1, ...)
```

Node.js가 위에서 친절한 API로 가려놨지만, OS 레벨에선 fd 번호만 다른 같은 함수.

---

## fd가 왜 필요한가

Unix가 "모든 자원을 read/write로 다루자"고 결정했다. 그럼 read 함수는 어떻게 생겨야 하나?

```
read(???, 어디에담을지, 몇바이트)
```

이 `???` 자리에 "무엇을 읽을지" 알려줘야 함. 후보:

| 후보 | 문제 |
|---|---|
| 파일 이름 `"a.txt"` | 매 read마다 OS가 파일을 새로 열어야 함 (비효율). 키보드·소켓·파이프는 이름이 모호. 같은 파일을 두 곳에서 읽고 있을 때 "지금 어디까지 읽었는지"를 구분 못 함 |
| 자원의 메모리 주소 | 프로세스가 OS 내부 자료구조를 직접 가리키게 됨. 보안·안정성 망함 |
| **정수 번호표** | ✅ OS가 자원을 미리 "열어"두고 그 자원에 번호를 부여. 프로세스는 그 번호만 들고 read/write 호출 |

세 번째가 fd. **OS가 자원을 열어두고 그 자원에 부여한 정수 번호.**

비유 — 카페 진동벨:
- 음료(자원)는 카운터(OS) 뒤에 있음
- 손님(프로세스)은 음료를 직접 못 듦
- 카운터가 "111번" 진동벨을 줌
- 손님은 "111번 음료 줘" = `read(111, ...)`

진동벨 번호 자체엔 음료 정보가 없지만, 카운터 뒤 시스템은 그 번호로 "라떼, 결제됨, 5분 후 픽업" 같은 정보를 찾을 수 있음. fd가 정확히 이 역할.

---

## fd가 OS 내부에서 가리키는 것

OS 안에는 **"열린 자원 표"** 가 있다. 자원이 하나 열릴 때마다 이 표에 항목이 추가됨. 항목에 적히는 정보:

- 이 자원이 어떤 것인지 (디스크 파일이면 어느 파일, 소켓이면 어느 연결...)
- 현재 어디까지 읽었는지 (디스크 파일의 경우 위치)
- 권한 (읽기 전용? 쓰기 가능?)

프로세스마다 별도로 **fd 테이블**이 있고, 각 슬롯이 OS의 "열린 자원 표" 항목을 가리킴:

```
프로세스 A의 fd 테이블:
┌─────┬─────────────────────────────────┐
│ fd0 │ ──→ [키보드 항목]                │
│ fd1 │ ──→ [화면 항목]                  │
│ fd2 │ ──→ [화면 항목]                  │
│ fd3 │ ──→ [a.txt 항목: 현재 위치 200]  │ ← open("a.txt")로 추가
│ fd4 │ ──→ [google.com 소켓 항목]       │
└─────┴─────────────────────────────────┘
```

`read(3, buf, 100)` 호출 시 OS가 거치는 단계:

1. fd 테이블의 3번 슬롯을 봄
2. "a.txt 항목"을 찾음
3. 거기 적힌 "현재 위치 200"부터 100바이트 읽어 buf에 채움
4. 항목의 위치를 300으로 갱신

다음 `read(3, buf, 100)`은 자동으로 300번지부터 읽힘. **OS가 진행 상황을 기억해주니까 프로세스는 fd 번호 하나만 들고 있으면 됨.** 이게 fd의 진짜 가치.

## 왜 이름이 "디스크립터"인가

`descriptor`의 어원은 라틴어 `describere` = **"기술하다, 묘사하다"**.

OS 안의 "자원 항목" 자체가 그 자원이 무엇인지·어떤 상태인지를 **기술(describe)** 한 데이터:

- 이 자원은 a.txt 파일이고
- 읽기 권한으로 열렸고
- 현재 200번지까지 읽었고
- 소유자는 사용자 X고
- ...

이게 그 자원에 대한 "description(기술서)". 그래서 이 항목 이름이 **file descriptor (파일을 기술하는 것)**.

프로세스가 가지는 정수 번호(fd 0, 1, 3...)는 그 description을 가리키는 번호. 실무에선 그 정수 자체를 fd라고 부르지만, 엄밀히는:

- **(a) OS 안의 기술 데이터 자체** = file descriptor (원래 의미)
- **(b) 그 데이터를 가리키는 정수** = file descriptor (실무 용법)

호텔 카드키 비유:
- 카드키 번호 자체는 의미 없음
- 카드를 대면 호텔 시스템이 "이 카드 = 305호, 5/17까지 유효, 미니바 결제 가능..." 정보(=description)에 접근
- 카드 번호 = fd (실무 용법)
- 호텔 시스템의 카드 정보 데이터베이스 = file descriptor (엄밀 용법)
- 그 카드 정보가 카드를 "기술"하니까 → **descriptor**

## fd 종류 (전부 같은 인터페이스)

| fd | 자원 | JS 예시 |
|---|---|---|
| 0 | 표준 입력 (보통 키보드) | `process.stdin` |
| 1 | 표준 출력 (보통 터미널 화면) | `process.stdout`, `console.log` |
| 2 | 표준 에러 (보통 터미널 화면) | `process.stderr`, `console.error` |
| 3+ | 디스크 파일 | `fs.openSync('a.txt')` → fd 반환 |
| 3+ | 소켓 (네트워크) | `fetch`, `WebSocket`, Node.js `net` |
| 3+ | 파이프 (프로세스 간) | `child_process.spawn().stdout.pipe()` |
| 3+ | 디바이스 (`/dev/...`) | `fs.createReadStream('/dev/...')` |

각 종류 한 줄:

- **소켓** — 네트워크나 같은 컴퓨터의 다른 프로세스와 데이터 주고받는 창구. `fetch()`, `WebSocket`이 내부에서 사용.
- **파이프** — 두 프로세스 사이를 잇는 데이터 통로. 셸 `ls | grep .js`가 `ls`의 stdout fd → `grep`의 stdin fd를 파이프로 연결.
- **디바이스** — `/dev/` 아래의 가상 파일. 키보드, USB, 사운드카드, 블랙홀(`/dev/null`)까지 모두 파일처럼.

---

## JS로 직접 fd 보기

```js
const fs = require('fs');

const fd = fs.openSync('a.txt', 'r');  // OS: a.txt를 열고 fd 번호 반환
console.log(fd);                        // → 3 (또는 다른 작은 정수)

const buf = Buffer.alloc(100);
fs.readSync(fd, buf, 0, 100, null);    // fd가 가리키는 자원에서 100바이트
fs.readSync(fd, buf, 0, 100, null);    // 또 호출하면 그다음 100바이트 (위치 자동 진행)

fs.closeSync(fd);                       // fd 슬롯 비움, OS 자원 해제
```

평소 쓰는 `fs.readFile('a.txt', cb)`는 Node.js가 open → read → close를 한 번에 감싼 것.

## 오개념 예방

- **"터미널이 파일이다" ≠ "터미널 입력이 디스크에 저장된다"**. read/write 시스템 콜을 공유한다는 인터페이스 통일의 의미.
- **fd 0/1/2는 항상 키보드/화면이 아님**. `node app.js > out.txt` 하면 fd 1이 화면 대신 파일. 프로세스 코드는 그냥 `write(1, ...)` 하면 되고 어디로 가는지 알 필요 없음 — 통일 인터페이스의 강점.
- **Windows는 fd 대신 `HANDLE`** 이라는 더 일반적인 개념. "자원을 번호로 통일 식별"이라는 핵심 아이디어는 같음.
- **"프로세스가 화면에 출력하려면 OS를 거쳐야 한다"의 메커니즘** — 프로세스가 픽셀을 직접 못 그리는 게 아니라, fd 1번에 `write` 시스템 콜 호출 → OS 커널이 받아 터미널 드라이버 통과 → 화면 출력. fd가 그 부탁의 창구 번호.

## 자문자답 Q&A

### A. 파일의 정의

**A1.** "터미널이 파일이다"의 정확한 의미는 뭔가? 디스크에 저장된다는 뜻인가?

> 디스크 저장과는 무관. **터미널을 디스크 파일과 똑같이 `read`/`write` 시스템 콜로 다룬다**는 의미. Unix가 "파일"을 "read/write로 다룰 수 있는 모든 대상"으로 넓게 정의했기 때문에 터미널도 파일 부류에 들어간 것.

**A1-1.** 키보드·마우스·USB도 파일인가? 그렇다면 "파일"이라는 분류의 기준은 뭔가?

> 전부 파일. 키보드·마우스는 `/dev/input/...`, USB는 `/dev/sdb` 같은 가상 파일로 표현. 분류 기준은 **"같은 read/write 시스템 콜로 데이터를 주고받을 수 있는가"**. 디스크 저장 여부가 아니라 인터페이스 공유 여부.

**A1-2.** "파일"이라는 단어는 "대상"인가 "인터페이스"인가? 혼동하면 왜 헷갈리는지 한 줄로

> **파일 = 대상**, **인터페이스 = read/write 시스템 콜 그 자체**. 혼동하면 "터미널이 파일이다"를 듣고 "터미널이 read/write라는 함수다"라고 잘못 받아들이거나, 반대로 "터미널이 디스크에 저장된 데이터 덩어리다"라고 받아들여 양쪽 다 의미가 안 맞게 된다.

**A2.** Unix 설계자들이 "everything is a file" 철학을 채택한 동기는?

> 데이터를 주고받는 대상이 너무 다양해서(디스크·키보드·화면·네트워크·프로세스 간 통신 등) **각각 다른 API를 만들면 코드가 폭발하기 때문**. `readDisk()`, `readKeyboard()`, `recvNetwork()`처럼 자원마다 시스템 콜이 따로면 프로그램·OS 양쪽 모두 복잡해진다. "어차피 동작은 read와 write 두 가지뿐, 인터페이스 통일하자"는 결정.

**A2-1.** 자원마다 다른 API였다면 셸 리다이렉션 `node app.js > out.txt`이 자연스럽게 가능했을까?

> 자연스럽게는 불가능. 리다이렉션이 가능한 이유는 **프로세스가 `write(1, ...)`라는 같은 시스템 콜만 호출하고, 셸이 그 fd 1번을 화면 대신 파일로 미리 연결해놓기 때문**. 만약 `writeScreen()`·`writeFile()`이 따로면 프로세스 코드 안의 `writeScreen()` 호출을 셸이 어떻게 가로채서 파일로 보낼 수 있겠나? 코드 자체를 바꿔야 하므로 외부에서 못 가로챈다.

### B. fd의 필요성

**B1.** read 함수의 첫 인자에 fd가 아니라 파일 이름(문자열)을 넣는다면 어떤 문제가 생기나?

> 세 가지.
> - **매번 OS가 파일을 새로 열어야 함** — 이름만 받으면 매 read마다 OS가 "이 파일 어디? 권한 있나?"를 처음부터 확인. 비효율.
> - **키보드·소켓·파이프는 이름이 모호** — 키보드는 `/dev/tty`가 있지만 "google.com에 연결된 특정 소켓 하나"는 부를 이름이 없다.
> - **진행 상황 추적 불가** — 같은 파일을 두 곳에서 동시에 읽고 있을 때 파일 이름만으로는 "이쪽이 어디까지 / 저쪽이 어디까지"를 구분할 수 없다.

**B1-1.** "현재 어디까지 읽었는지"를 OS가 기억해야 하는 이유 — 같은 파일을 두 곳에서 동시에 읽는 시나리오로

> `a.txt` 1000바이트. 프로세스 A는 0→500 읽고 나머지 읽고 싶음, 프로세스 B는 0→300 읽고 나머지 읽고 싶음.
> - 위치를 OS가 기억 안 하면? A·B의 두 번째 read 모두 "어디부터"를 호출자가 직접 줘야 하고, A와 B가 서로의 위치를 망친다.
> - 위치를 OS가 fd별로 기억하면? A의 fd 항목 = "위치 500", B의 fd 항목 = "위치 300"으로 독립. 각자 read 호출이 자기 위치부터 자동으로 이어진다. **fd별 위치 독립 관리가 동시 읽기를 깨끗하게 한다.**

**B2.** fd가 정수 번호인 이유? 메모리 주소가 아니라.

> 두 가지.
> - **보안**: 메모리 주소를 직접 넘기면 프로세스가 OS 커널 내부 자료구조를 직접 가리키게 됨. 잘못된 주소나 의도적 조작으로 OS를 망가뜨릴 수 있다. 정수는 OS가 fd 테이블을 통해 한 번 검증·번역하므로 안전.
> - **추상화**: OS 내부 구현이 바뀌어도(자료구조 위치가 옮겨져도) fd 번호는 그대로. 프로세스 코드를 다시 컴파일할 필요 없다.

### C. fd의 내부 구조

**C1.** "프로세스의 fd 테이블"과 "OS의 열린 자원 표"는 각각 뭐고 어떤 관계?

> - **프로세스의 fd 테이블**: 각 프로세스가 따로 가지는 작은 배열. 슬롯 번호 = fd 번호. 슬롯 내용 = OS 자원 표 항목을 가리키는 포인터.
> - **OS의 열린 자원 표**: OS 커널 전체에 하나만 있는 큰 표. 시스템 전체에서 열린 모든 자원의 정보(어느 파일, 현재 위치, 권한 등)가 들어 있다.
> - **관계**: fd 테이블 슬롯이 자원 표 항목을 가리킨다. 진동벨 번호 ↔ 카운터 뒤 음료 시스템과 동일한 구조.

**C1-1.** `read(3, buf, 100)` 호출 시 OS 내부 단계 4가지

> 1. **fd 테이블 조회** — 현재 프로세스의 fd 테이블에서 3번 슬롯을 찾는다.
> 2. **자원 항목 접근** — 그 슬롯이 가리키는 "열린 자원 표 항목"을 찾는다. 거기에 "a.txt, 현재 위치 200, 읽기 권한 있음" 같은 정보.
> 3. **실제 데이터 읽기** — 디스크에서 a.txt의 200번지부터 100바이트를 읽어 프로세스의 buf에 복사.
> 4. **위치 갱신** — 자원 항목의 "현재 위치"를 300으로 갱신. 다음 호출이 자동으로 그다음 바이트부터.

**C1-2.** 두 번째 `read(3, buf, 100)`이 자동으로 다음 100바이트가 읽히는 이유 — "어디에" 무엇이 있어서?

> **OS의 "열린 자원 표 항목" 안에 "현재 위치"가 기록되어 있어서**. 프로세스가 위치를 들고 다니는 게 아니라 OS가 fd가 가리키는 항목에 보관·갱신. → **fd는 단순 식별자가 아니라 "상태가 있는 핸들"**.

**C2.** 자원 표 항목에 적히는 정보 3가지와, 각각 없으면 생기는 문제

> - **이 자원이 어떤 것인가** (디스크 파일이면 어느 파일, 소켓이면 어느 연결) — 없으면 OS가 read 받아도 "어디서 데이터 가져오라는 거지?"를 모른다.
> - **현재 어디까지 읽었는지** (위치 포인터) — 없으면 연속 read가 매번 처음부터. 동시 읽기에서 서로 위치를 망친다.
> - **권한** (읽기 전용? 쓰기 가능?) — 없으면 읽기 전용으로 연 파일에 write가 통해버려서 보안·데이터 무결성 망가짐.

### D. 디스크립터 어원

**D1.** 왜 이름이 "descriptor"인가? 어원과 OS 자료구조 연결

> descriptor의 어원 = 라틴어 `describere` ("기술하다, 묘사하다"). OS 안의 "열린 자원 표 항목"이 그 자원에 대한 **description(기술서) 그 자체** — 자원이 뭔지·어떤 상태인지를 적어둔 데이터. 그 description 데이터가 곧 "file descriptor (파일을 기술하는 것)"라는 이름의 정체. 프로세스에게 보이는 정수 번호는 그 description을 가리키는 핸들.

**D1-1.** "file descriptor"는 (a) OS 안의 자료구조 자체와 (b) 그걸 가리키는 정수 중 어느 쪽? 실무 용법은?

> 엄밀히는 둘 다. 실무에선 거의 대부분 (b) 정수 번호 쪽 — "fd 3", "stdin은 fd 0이다". (a)를 따로 구분할 땐 "open file description"이라는 더 긴 용어를 쓰기도 한다 (Linux man page에서 실제로 구분).

**D1-2.** 호텔 카드키 비유 — "카드 번호" / "카드 정보 DB" / "방 자체" 각각 fd 모델에서 무엇?

> - **카드 번호** = 프로세스가 들고 있는 fd 정수 (예: 3). 자체엔 의미 없는 식별자.
> - **카드 정보 데이터베이스의 한 행** = OS의 "열린 자원 표 항목". 카드가 어느 방·언제까지·어떤 권한인지 기술.
> - **방 자체** = 실제 자원 (디스크의 a.txt, 네트워크 연결, 키보드 디바이스 등). 카드 정보가 가리키는 진짜 대상.

### E. 자원 종류와 통일 인터페이스

**E1.** `node app.js > out.txt` 실행 시 프로세스 안의 `console.log('hi')`는 어디로 가나? 프로세스가 그 사실을 알아야 하나?

> **`out.txt` 파일로 들어간다**. `console.log`는 내부에서 `write(1, "hi", ...)` 호출, 셸이 띄울 때 그 프로세스의 fd 1번 슬롯을 화면 대신 `out.txt` 파일 항목으로 미리 연결해놓아서. **프로세스 코드는 그 사실을 전혀 모르고 알 필요도 없다.** 그냥 fd 1에 write할 뿐.

**E1-1.** 이게 왜 "통일 인터페이스의 강점"인지 한 줄로

> 프로세스가 자원 종류별 함수(`writeScreen`·`writeFile`)가 아니라 **자원 종류와 무관한 단일 함수(`write(fd, ...)`) 하나만 쓰기 때문에, 그 fd가 무엇과 연결됐는지를 외부(셸)에서 자유롭게 바꿔 끼울 수 있다**.

**E2.** 디스크 파일 fd와 `socket()` fd, 프로세스 입장에서 다른 점? read/write 호출 코드는?

> **프로세스 입장에서는 동일**. 만들어진 방식만 다를 뿐(`open("a.txt")` vs `socket(...)`), 한 번 fd를 받고 나면 `read(fd, buf, n)`, `write(fd, buf, n)`로 똑같이 호출. Node.js에서 `fs.createReadStream`과 `net.createConnection`이 둘 다 비슷한 Stream API를 갖는 게 그 반영.

**E2-1.** "디스크 vs 네트워크"의 차이는 OS의 어디서 생기나?

> **OS 커널 내부의 드라이버 단계**. read가 들어오면 OS가 fd 항목을 보고 "디스크 파일 항목이네 → 디스크 드라이버 → SATA/NVMe로 가져옴" / "소켓 항목이네 → 네트워크 드라이버 → 이더넷/Wi-Fi로 패킷 받음" 식으로 분기. 프로세스에겐 그 분기가 숨겨져 있다.

**E3.** `/dev/null`에 write하면 데이터가 사라지는 게 "파일 = 인터페이스 통일"의 어떤 강점 예?

> `/dev/null`은 진짜 디스크 파일도 아니고 데이터를 저장하지도 않는데 **read/write라는 같은 인터페이스로 다룰 수 있다**. 덕분에 `command > /dev/null`로 "출력 다 버려"를 자연스럽게 표현. 프로그램 코드는 한 줄도 안 바뀜. **"파일 = read/write 가능한 대상"이라는 추상화가 디스크 저장과 무관한 가상 자원(null, zero, random 등)까지 자연스럽게 받아들이는** 강점의 예.
