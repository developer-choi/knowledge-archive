# Answers

## 테스트 코드가 필요한 이유는 무엇인가?

### User Answer
수정한 코드가 문제가 없는지를 검증하기 위해서다.
사람이 매번 같은 시나리오를 손으로 확인할 수는 없으므로, 코드로 코드를 검증한다.

---

## QA가 있는데도 왜 테스트 코드가 필요한가?

### User Answer
테스트 코드는 QA를 대체하는 것이 아니라 **보완**하는 것이다.
두 활동은 목적, 시점, 비용이 다르므로 서로 다른 종류의 결함을 잡는다.

| 항목 | QA | 테스트 코드 |
|---|---|---|
| 주체 | 사람(QA 엔지니어) | 자동화된 코드 |
| 시점 | 기능 완성 후 | 코드 작성과 동시에 / CI |
| 검증 대상 | 사용자 시나리오, UX, 시각적 요소 | 함수·컴포넌트 단위 동작, 회귀 |
| 반복 비용 | 매번 사람이 다시 수행 | 한 번 작성하면 자동 반복 |
| 잡는 결함 | 사용성·UX·예상 못한 흐름 | 로직 오류·회귀 |

## 어떤 기능이 테스트코드를 작성할 만큼 중요한가?

### User Answer
모든 기능에 동일한 비중으로 테스트를 작성할 수는 없으므로, 다음 4가지 기준으로 우선순위를 정한다.

- **비즈니스적으로 더 중요한 기능**: 예를 들어 랭디의 경우 수업신청이 커뮤니티보다 비즈니스 영향이 크다.
- **코드가 복잡한 기능**: 조건문이 많아 경우의 수가 많은 코드.
- **과거에 한 번 이상 오류가 발생했던 기능**: 회귀 가능성이 높다.
- **많은 곳에서 사용하는 공통 함수**: 영향 범위가 넓어 깨질 때 파급력이 크다.

## 부모 함수가 자식 함수를 호출할 때, 테스트를 어떻게 분배해야 하는가?

### User Answer
부모 함수가 자식 함수를 내부적으로 호출하는 구조라면, 다음 3단계로 테스트를 분배한다.

- **부모 함수를 많이 테스트한다**: 자식 함수의 동작을 포함하여 실제 사용 흐름을 검증한다.
- **자식 함수에서는 부모로부터 잘 전파되는지만 짧게 확인한다**: 부모에서 이미 검증한 케이스를 자식에서 중복 테스트하지 않는다.
- **자식 함수에서 확장한 부분만 따로 테스트한다**: 자식 고유의 추가 로직만 별도로 커버한다.

> #### User Annotation:
> 참고: validateIncludeString()이 내부적으로 validateString()을 호출하는 케이스 — validateString()에서 테스트하는 것을 validateIncludeString()에서 중복할 필요는 없고, 잘 전파되는지만 보면 된다. (https://github.com/developer-choi/react-playground/commit/585ed1fb6c3fd217441e402886c755d0c3f186d5)

> #### User Annotation:
> 참고: calculateTotal() / calculateNumberTotal() — 자식인 calculateNumberTotal()만 테스트하려 했다가 부모 쪽 흐름을 놓칠 뻔한 사례. (https://github.com/developer-choi/react-playground/commit/54fddbfa65e823df90e7b3f0b662c06419b4bca7)
