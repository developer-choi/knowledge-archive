# Answers

## SSR/SSG/CSR을 어떤 기준으로 선택해야 하는가?

### User Answer
1. 핵심 데이터를 SSG 또는 SSR로 렌더링하고,
2. 그 외 데이터를 CSR한다.
3. 단, 예외로 메인 콘텐츠 API 속도가 지나치게 느리면 Full CSR한다. (예시: 온라인 쇼핑몰 상품리스트 페이지)

모든 데이터를 전부 빠르게 보여줄 수는 없다. 뷰포트에 보이는 부분부터 최대한 빠르게 보여주고, 나머지는 차차 보여준다.

---

## CSR은 어떤 콘텐츠에 적합한가?

### User Answer
웹페이지에서 중요한 UI가 아닌 경우에 사용한다. (메인 콘텐츠가 아닌 경우, 뷰포트보다 밑에 있거나 모달처럼 떠있는 경우)

대표적인 예시는 유튜브 동영상 1개 보는 페이지인데, 동영상과 관련 동영상만 먼저 보이고 댓글목록은 나중에 사용자가 스크롤을 내려서 보려고 할 때 불러온다는 특징을 갖고 있다.

이것을 전부 SSR로 하게 될 경우 동영상·관련동영상·댓글목록이 모두 응답될 때까지 사용자는 기다릴 수밖에 없게 된다.

---

## SSG를 사용할 수 있는 조건은 무엇인가?

### User Answer
모든 사용자에게 같은 URL이어도 페이지의 유니크한 데이터만큼은 동일하게 보이는 페이지만 SSG를 사용할 수 있다.

모든 데이터가 모든 사용자에게 동일하지 않아도 되고, 해당 페이지만의 유니크한 데이터만 모든 사용자에게 동일하게 보이는 웹페이지면 된다.

게시글목록 페이지가 1페이지부터 수백 페이지가 있다고 했을 때, 유니크한 데이터는 게시글 목록 데이터만 해당하고, 유니크하지 않은 데이터는 필터목록·정렬목록 등이 해당한다. 최근 검색어목록도 유니크하지 않은 데이터에 포함된다 (유저마다 다름).

---

## Private page에 SSG를 쓸 수 없는 이유는?

### User Answer
로그인 안 한 사용자는 접근을 못해서 비회원/회원 간의 보여지는 정보가 다르다. 회원 간에도 서로 보여지는 데이터가 모두 다르기 때문에 안 된다. (예: 마이페이지 정보)

---

## 진입 조건이 있는 페이지에 SSG를 쓸 수 없는 이유는?

### User Answer
Private page의 연장선이다. 다음 경우에 SSG를 사용할 수 없다.

- 특정 등급의 회원만 입장할 수 있는 경우
- 내가 허용한 사용자만 접근할 수 있는 페이지 (유튜브 동영상 상세페이지의 visibility 기능)
- 특정 기간 동안에만 접근할 수 있는 페이지 (응모하기 기능있는 이벤트 상세 페이지)

---

## 실시간으로 최신화되어야 하는 페이지에 SSG가 맞지 않는 이유는?

### User Answer
실시간으로 최신화돼야 하는 페이지는 굳이 SSG를 사용해도 얻을 수 있는 이득이 없다. (예: 암호화폐 거래소)

---

## 사용자가 직접 콘텐츠를 생성·수정하는 페이지에 SSG를 쓸 수 없는 이유는?

### User Answer
특정 조건에 따라 즉시 최신화돼야 하는 웹페이지에는 SSG를 쓸 수 없다.

사용자가 글을 쓸 수 있는 게시판(예: 네이버 카페)에서 사용자가 글을 쓰고 난 뒤 목록·상세·수정 페이지로 이동하면, 해당 페이지에 쓴 글이 즉시 반영되어 있어야 한다. 따라서 목록·상세·수정 페이지는 미리 만들 수 없다. 사용자가 액션을 취한 다음 즉시 반영되어야 하는 페이지이기 때문이다.

---

## 관리자가 만든 콘텐츠라도 SSG가 안 되는 경우가 있는가?

### User Answer
단순히 보는 것으로 끝나는 상세페이지면 SSG가 맞는데, 응모이벤트 상세페이지처럼 응모를 하는 페이지(CREATE)이거나, 관리자가 노출 여부를 설정할 수 있는 페이지라면 SSG로 만들면 안 된다.

사용자가 뭔가를 생성하는 페이지이고, 노출을 설정할 수 있는 페이지(이벤트 상세페이지)라면 그 페이지는 SSG로 만들면 안 된다. 노출이 적용되지 않으니까.

이벤트 리스트 페이지도 관리자에서 이벤트마다 노출을 설정할 수 있는 기능이 있으면, 당장 내리고 싶어도 SSG를 사용하면 안 된다.

---

## SSR TTFB에 가장 크게 영향을 주는 요인은 무엇인가?

### User Answer
HTML TTFB에 영향을 주는 것은 API TTFB보다 HTML 만드는 데 들어가는 비용이 더 크다.

API TTFB 시간이 42ms여도 HTML 만드는 데에만 700ms 이상 걸리는 사례가 있었다.

SSG는 한번만 빌드해 두면 (또는 revalidate) 그 뒤로는 항상 빠르게 응답되지만, SSR은 매 요청마다 HTML을 빌드해야 해서 더 타격이 컸다.

즉, 당장 뷰포트에 보이는 중요한 데이터만 Server Side 단계에서 렌더링시키는 것이 성능상 옳다. (나머지는 CSR)
