# Answers

## 프로세스간 통신이 뭔가요?

### User Answer
프로세스들은 단독으로 역할을 수행하고 종료될 수도 있지만, 타 프로세스와 협력하는 경우가 종종 생긴다.

프로세스가 협력을 허용하는 이유는 몇 가지가 있다.

1. 계산 가속화
- 작업을 빨리 실행하기 위해서 서브 태스크들로 잘게 나눈 다음 다른 서브 태스크들과 병렬로 실행해야 한다.
- 이게 가능하려면 서브 태스크들끼리 서로 협력할 수밖에 없다.
- 이 병렬 동시 처리는 코어 수가 여러 개일 때 가능하다.

2. 편의성
- 음악을 들으면서 워드 파일을 편집할 수 있는 것처럼, 이런 작업들을 병렬로 하기 위함도 있다.

이런 이유로 프로세스 간의 통신이 필요할 때, 프로세스 간에 데이터를 주고받을 수 있는 기법을 IPC(Inter Process Communication)라고 부른다.
크게 공유메모리 / 메시지전달 2가지 모델이 있고, 서로 장단점과 특징이 있으므로 적절한 상황에 응용하면 된다.

## 생산자, 소비자 문제가 뭔가요? (Producer–consumer problem)

### User Answer
생산/소비 프로세스 간에 생산 속도와 소비 속도가 달라서 생기는 문제다.

생산자 프로세스는 데이터를 만드는 프로세스, 소비자 프로세스는 데이터를 소비하는 프로세스라고 단어를 먼저 정의한다.

생산하는 속도와 소비하는 속도는 다를 수밖에 없다.
예를 들어 생산자가 소비자가 소비할 수 있는 것보다 너무 빨리 생산하면 소비자에게 전부 다 전달할 수 없는 상황이 생긴다.
반대로 소비자가 소비를 너무 빨리 해버리면 생산자가 생산할 때까지 소비자는 그냥 기다려야 한다.

이것을 해결할 수 있는 방법은 몇 가지가 있지만, 가장 이상적인 결과는 생산자 프로세스와 소비자 프로세스가 모두 기다리지 않고 평행하게 실행되는 것이다.

어쨌든, 방법 중에는 버퍼를 사용하는 방법이 있다.

1) 무한 버퍼
- 저장할 수 있는 데이터의 크기에 한계가 없어서, 생산자는 무한정 생산할 수 있다.
- 소비자는 버퍼에 데이터가 없으면 대기하겠지만, 있다면 소비하는 구조다.
- 이것도 방법이긴 하지만, 현실에서 버퍼가 무제한인 건 쉽지 않다. 공장으로 따지면 창고 크기가 무제한인 공장은 없다.

2) 유한 버퍼
- 크기가 고정된 버퍼를 사용하는 방법이다.
- 버퍼가 비어 있으면 소비자는 대기하게 되고, 버퍼가 다 채워져 있으면 생산자가 대기하게 된다.
- 이 방법은 이러한 trade-off가 있다.

## 공유메모리 방식(Shared Memory Model)에 대해 설명해주세요.

### User Answer
협력하는 두 프로세스들 간에 공유되는 메모리 영역을 따로 구축하는 방식이다.

원칙적으로는 프로세스가 다른 프로세스의 메모리에 접근하는 것을 금지하고 있지만, 공유메모리 같은 경우에는 이 제약 조건을 해제하는 것을 전제로 한다.

공유메모리 영역 구축은 OS에서 해주지 않고, 통신하는 프로세스들이 직접 공유메모리 영역을 구축해야 한다.
그리고 동일한 메모리 위치에 쓰기를 하지 않아야 하는 것도 주의해야 할 점 중 하나다.

## 메시지전달 방식(Message Passing Model)에 대해 설명해주세요.

### User Answer
프로세스 간에 메시지들을 통해 통신이 이루어지는 방식이다.
만약 프로세스들이 통신을 원하면 반드시 서로 메시지를 보내고 받아야 한다.
OS가 프로세스 간의 통신 수단을 제공해 주는 방법이다.

공유 메모리처럼 주소 공간을 제공하지 않아도 되고, 이 동작들을 동기화할 수 있는 기법을 제공한다.

컴퓨터 여러 대가 네트워크로 연결된 시스템에서 특히 유용한 방법이다.
이들 사이에는 통신 연결 링크(Communication Link)라고도 부르는 링크를 통해서 설정이 되어야 한다.
바로 이 링크를 다양한 방법들로 구현할 수 있다.

1. 직접 통신 / 간접 통신
2. Blocking 방식 통신 / Non-Blocking 방식 통신
3. 버퍼링 방식의 통신

OS가 제공하는 연산은 기본적으로 Send와 Receive 2가지 연산이 있다.
그리고 메시지는 고정 길이일 수도 있고 가변 길이일 수도 있다.

1. 고정 길이
- 시스템이 구현하는 건 간단하다.
- 하지만 프로그래밍할 때 고정해서 정보를 보내야 하기 때문에 힘들다.

2. 가변 길이
- 시스템이 구현하는 건 어렵다.
- 하지만 프로그래머는 아무렇게나 전송하면 되어서 편하다.

그래서 메시지의 길이는 서로 장단점이 있다.

## 메시지 전달방식 중 직접 통신에 대해 설명해주세요.

### User Answer
직접 통신은 메시지 전달 방식 중 하나이고, 프로세스 2개가 한 쌍으로 통신하는 방식이다.
직접 통신이라는 말 그대로 중간에 거치는 것 없이 바로 통신한다는 특징을 갖고 있다.

대칭성 또는 비대칭성을 갖고 있는 경우가 있다.

여기서 대칭성이란, 보내는 프로세스가 받는 프로세스를 알고 있고, 받는 프로세스도 보내는 프로세스를 알고 있는 경우다.

비대칭성이란, 보내는 프로세스만 받는 프로세스에 대해 알고, 받는 프로세스는 보낸 프로세스에 대해 모르는 경우다.
받는 곳만 정확하게 지정된다면 보낼 수는 있다.

## 메시지 전달방식 중 간접 통신에 대해 설명해주세요.

### User Answer
간접 통신과 직접 통신의 가장 큰 차이는 메일함이라는 중간자가 있다는 부분이다.
프로세스들에 의해서 메시지들이 중간자인 메일함에 넣어진다.

메일함마다 고유의 ID를 갖고 있고, 이 ID로 프로세스가 여러 메일함을 통해 다른 프로세스들과 통신이 가능하다.

간접 통신은 다음과 같은 성질을 갖고 있다.

1. 메일함이 있어야 프로세스들 사이에 연결이 구축된다.
2. 연결은 두 개 이상의 프로세스와 연결 가능하다.
3. 각 프로세스들의 쌍에는 서로 다수의 서로 다른 연결이 존재할 수 있다. ⇒ 꼭 A와 연결되어 있다고 해서 B와 연결이 불가능한 것은 아니다.

## 간접 통신 과정에서 송신자가 보낸 메시지를 받을 때 수신자 프로세스들이 서로 충돌하는 경우 어떻게 해결할 수 있을까요?

### User Answer
P1 프로세스가 송신자, P2·P3 프로세스가 수신자인 상황에서 P1이 메시지를 보내면 반드시 충돌이 발생한다.
하나의 요청을 동시에 처리하는 꼴이 된다.

그래서 미리 대책을 세워야 한다.
- 방법 1. 하나의 링크는 최대 두 개의 프로세스가 링크를 공유하도록 허용한다.
- 방법 2. 한 번에 한 프로세스만 receive 연산을 수행하도록 한다.
- 방법 3. 송신자가 미리 수신자를 선택할 수 있도록 한다.

## 메시지전달 방식 중 Blocking, Non-Blocking 방식에 대해 설명해주세요.

### User Answer
- Blocking 모드
  - Send: 송신자 프로세스는 메시지를 보낸 뒤 수신될 때까지 대기한다.
  - Receive: 수신자 프로세스는 이용 가능한 메시지가 있을 때까지 대기한다.

- Non-Blocking 모드
  - Send: 송신자 프로세스는 메시지를 보낸 뒤 다른 작업을 처리한다. (Blocking처럼 송신자가 대기하지 않는다.)
  - Receive: 수신자 프로세스는 유효한 메시지를 받거나 null 메시지를 수신한다.

## 메시지전달 방식 중 버퍼링 방식에 대해 설명해주세요.

### User Answer
통신 방식이 직접적이든 간접적이든, 통신 프로세스 간 교환되는 메시지는 임시 대기열(Queue)에 존재하게 된다.
이 Queue를 구현하는 방식에는 총 세 가지가 있다.

1. 최대 길이를 0으로 설정하는 방법
- 발신자는 수신자가 메시지를 받을 때까지 대기하는 방식이다.
- 용량이 없어서 메시지가 대기할 수 없기 때문이다.

2. 제한된 용량(유한 길이)
- 수신자는 버퍼로부터 메시지를 전달받는다.
- 하지만 버퍼가 가득 차면 송신자는 대기열 공간이 확보될 때까지 대기한다.

3. 무한 용량
- 발신자는 절대 대기하지 않는다.
- 하지만 용량이 무한이라는 부담이 있다.

## 공유메모리 방식과 메시지전달 방식의 장단점이 어떻게 되나요?

### User Answer
1. 구현 난이도: 공유메모리 > 메시지전달
- 공유메모리는 같은 곳에 쓰기를 하면 안 되는 문제가 있지만, 메시지전달 모델은 충돌 회피를 할 필요가 없다. 메시지가 Queue에 쌓이는 방식이기 때문이다.
- 분산 시스템에서도 공유 메모리보다 메시지 전달이 구현하기가 더 쉽다.

2. 속도: 공유메모리 > 메시지전달
- 메시지전달은 중간에 시스템 콜을 거치지만, 공유메모리는 중간에 시스템 콜을 거치지 않기 때문에, 공유메모리 방식이 메시지전달 방식보다 속도가 빠르다.
- 메시지 큐는 커널 위에 있어서 커널까지 가야 한다. 하지만 공유메모리는 커널까지 안 간다. 그래서 빠르다.

3. 분산 환경: 공유메모리 < 메시지전달
- 서버가 많을 경우 공유하는 데이터를 여러 서버 간에 일치시키려고 하다가 성능이 떨어지는 문제가 발생할 수 있다.
- 그래서 분산 환경에서는 메시지전달 방식이 낫다.
