# Answers

## 컴퓨터 시스템의 기본 구성이 어떻게 되나요?

### User Answer
사용자는 사람·기계·컴퓨터 모두 가능하다.
기계가 명령을 했을 때도 유저의 명령을 받는다고 할 수 있다.

수많은 사용자들이 Application을 통해 H/W에게 수많은 요청을 보낸다.
그 중간 매개체가 바로 OS다.

OS = Application과 H/W의 중간자 역할.

---

## OS는 무슨 일을 하나요?

### User Answer
**하드웨어를 효율적으로 사용할 수 있도록 관리한다.**
메모리, CPU 등의 리소스는 한정되어 있는데, 수많은 Application들이 서로 요청을 한다. (= 경쟁)
이 때 OS는 최대한 효율적이고 공정하게 배분한다.

- 리소스 할당을 OS가 한다. 모든 리소스 관리를 OS가 한다고 말할 수 있을 정도.
- 어떠한 프로세스 하나가 너무 많은 시간을 점유하지 않도록 관리한다.
- 반대로 프로세스가 중요하다면 그만큼 더 많은 시간을 점유하도록 보장한다.

**프로세스에 문제가 생기기 전에 예방하고, 문제가 생기면 해결한다.**
프로세스들이 OS에게 요청을 하다가 문제가 생길 수 있다.
OS는 데드락 현상을 방지하고 사전에 이런 일이 생기지 않도록 막는 역할도 한다.

## 컴퓨터의 부팅 과정에 대해 설명해주세요.

### User Answer
한 줄 요약: 컴퓨터의 전원 버튼을 누른 순간부터, OS가 메모리에 적재되는 순간까지.

- 사용자가 PC의 전원 버튼을 누른다.
- 메인보드에 전력이 공급된다.
- CPU가 ROM에 저장되어 있던 BIOS를 실행해서, BIOS 루틴이 시작된다.
- BIOS가 주변 하드웨어에 문제가 없는지 확인한다.
  - 이 부분에서 문제가 생기면 컴퓨터를 켜다가 갑자기 화면이 보이지 않고 부팅이 멈춰버린다든지, 비프음이 발생한다든지, 블루스크린이 발생한다든지 같은 상황이 있다.
- MBR에 저장된 부팅 정보를 읽어온다.
- RAM에 Bootloader를 적재한다.
- 디스크에 있는 OS 커널 코드를 복사해서 메모리에 적재한다.

> #### User Annotation:
> - MBR은 메모리가 아니라 디스크에 있다.
> - 메모리(RAM)는 휘발성 저장장치이기 때문에, 컴퓨터가 꺼졌다가 나중에 다시 켤 때는 아무것도 없는 상황이다. 그래서 부팅 과정에서 MBR이 메모리에 있을 수가 없다.

## 인터럽트가 뭔가요?

### User Answer
인터럽트 = 끼어들기.
우선순위가 더 높은 작업이 생겼으니 이것부터 처리해달라는 요청.
CPU가 어떤 작업을 하고 있는데 입출력 같은 예외 상황이 발생하여 처리가 필요한 경우에 CPU로 신호를 보내는 것.

> #### User Annotation:
> 인터럽트 구분 방법은 H/W vs S/W도 있고, 외부 vs 내부도 있고, Exception 인터럽트 vs I/O 인터럽트 등 구분 방법이 많다.

## 하드웨어 인터럽트와 소프트웨어 인터럽트의 차이는 무엇인가요?

### User Answer
**하드웨어 인터럽트**
- 하드웨어가 발생시키는 인터럽트. CPU 외의 하드웨어가 CPU에게 신호를 보내는 것.
- 전원 이상일 수도 있고, 오버플로우가 발생했을 때도 해당된다.

**소프트웨어 인터럽트**
- 소프트웨어가 발생시키는 인터럽트. supervisor call (SVC)이 있다.
- 모든 데이터 삭제 같은 위험한 요청을 하면 문제가 생기기 때문에, SVC가 자동으로 관리 감독을 해서 중간에 가로챈다.

## 인터럽트의 Flow에 대해 설명해주세요.

### User Answer
- Interrupt 발생
- CPU는 수행 중인 작업을 중지
- 현재 수행 중이던 명령어의 주소값을 Program Counter에 저장
- CPU는 인터럽트가 요청한 위치(주소값)로 할 일을 변경
- (처리 중)
- 중단되었던 작업을 재개. (수행 중이던 명령어의 주소값을 저장했기 때문에 가능)

**I/O 리퀘스트에 의한 인터럽트 흐름**
- I/O Request 시작
- I/O Request 전송 완료 (transfer done)
- CPU는 수행 중이던 명령어의 주소값을 PC에 적재 (적재하는 만큼 딜레이 있음)
- 인터럽트 작업을 수행
- 다시 복귀해서 원래 하던 작업을 이어서 한다.
- (이후 반복)

## 운영체제가 사용자의 편의를 위해 제공하는 기능은 뭐가 있나요?

### User Answer
- GUI (PC, Mobile은 터치스크린)
- CLI
- 프로그램 실행
  - OS는 메모리에 프로그램을 적재시키고, 실행하고, 종료되도록 지원한다.
- I/O 수행
  - 실행 프로그램에 입출력이 필요할 경우 지원한다.
- 파일 시스템 조작
  - 파일/디렉토리 CRUD 지원, 사용자 권한 관리도 지원한다.
- 리소스 할당
- 로깅: 어떤 프로그램이 얼마나 많은 자원을 사용했는지 추적
- 기타

## 시스템 콜이 뭔가요?

### User Answer
운영체제에게 요청을 보내는 수단이다.

시스템 콜이 운영체제를 둘로 갈라놓고 있다고 이해하면 좋다.

```
유저의 요청 (From CLI, GUI)
       ↕
   <<< 시스템 콜 >>>
       ↕
OS가 제공하는 서비스들 (프로그램 실행, 파일 시스템 조작, 통신, 리소스 할당 등)
```

"이 파일 삭제해주세요", "이 프로그램 실행시켜주세요" 같은 요청(GUI에서도 할 수 있고, CLI에서도 할 수 있음)을 보내는 수단이 바로 시스템 콜이다.

시스템 콜은 2가지 모드가 있다: **사용자 모드, 커널 모드.**
사용자가 직접 리소스에 접근하면 문제가 생길 가능성이 많기 때문에 이렇게 나눠 놓았다.
(예를 들어 메모리의 몇 번지에 운영체제가 돌아가고 있고 몇 번지에 다른 중요한 프로세스가 돌아가고 있는데, 사용자가 직접 메모리에 접근하려고 하면 문제가 발생할 가능성이 높다.)

그래서 사용자는 원하는 요청이 있으면 직접 접근을 하지 않고, 시스템 콜이라는 수단을 통해 커널에게 요청을 보내도록 되어 있다.
그리고 커널이 요청을 처리하고 결과를 응답한다.

## cp in.txt out.txt 명령어를 입력하면 구체적으로 어떤 일이 발생하나요?

### User Answer
- 사용자는 "cp in.txt out.txt"라는 요청을 시스템 콜을 통해 커널에게 보낸다.
- 커널은 그 요청을 처리하기 위해 다음과 같은 절차를 수행한다.
  - 입력 파일명과 출력 파일명을 받는다.
  - 입력된 파일명에 해당하는 파일을 연다.
    - 이 때 파일명으로 된 파일이 없으면 에러가 발생한다.
  - 출력될 파일명에 해당하는 파일을 생성한다.
    - 이 때 파일명으로 된 파일이 이미 존재하면 에러가 발생한다.
  - 입력 파일을 한 줄 읽고, 출력 파일에 한 줄 쓴다.
    - 이 과정을 더 이상 읽을 게 없을 때까지 반복한다.
  - 두 파일을 닫는다.
  - 완료됐다는 메시지를 사용자에게 반환한다.

> #### User Annotation:
> 굳이 이렇게 사용자가 시스템 콜을 통해 요청을 보내는 이유도 결국 보안 문제다.
> - 사용자가 직접 in.txt 파일을 열고 out.txt 파일을 만들고 한 줄씩 복사하다 보면 문제가 생길 가능성이 높다.
> - 그래서 복사 요청만 시스템 콜을 통해 커널에게 보내고, 그 결과만 돌려받는 것.

## 시스템 콜들은 어떤 것들이 있는지 나열해주세요.

### User Answer
- **프로세스 제어**
  - 프로세스 생성·종료·로드·실행, wait/signal, 메모리 할당
- **파일 관리**
  - 파일 열기/닫기, 쓰기/수정, 생성·삭제·복사·이동, 파일 권한 관리
- **장치 관리**
  - 장치 연결 / 분리
- **통신**
  - 통신 연결 생성, 송수신, 전송 상태 확인

→ 거의 대부분을 사용자가 직접 하지 않는다.
→ 시스템 콜이 없으면 거의 대부분의 작업을 못 한다.
