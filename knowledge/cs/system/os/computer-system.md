## 시스템 소프트웨어란 무엇인가?

### User Answer
운영체제, 부트로더, 쉘 같은 것들을 말한다.

---

## 응용 소프트웨어란 무엇인가?

### User Answer
일반적으로 사람이 사용하는 프로그램들을 말한다.
컴퓨터 자체의 동작과는 관련이 없다.
예: 크롬 브라우저.

---

## 컴퓨터의 하드웨어 구성 요소는 무엇인가?

### User Answer
겉으로 보이는 것: 디스플레이, 키보드, 마우스.
본체 안: CPU, 주기억장치(= 메모리), 보조기억장치(HDD, SSD), Power Supply 등.

---

## 컴퓨터의 가장 중요한 3요소는 무엇이며, 컴퓨터의 기본 구조는 어떻게 되는가?

### User Answer
메모리, CPU, I/O Device 이 3가지가 가장 중요하다.
이 세 가지가 시스템 버스로 통신하는 구조로 되어 있다.

---

## CPU란 무엇인가?

### User Answer
프로그램의 실행과 데이터 처리를 담당하는 부품이다.
주로 입출력 장치로부터 데이터를 받아서 처리한 후 결과를 기억장치에 저장하는 일을 한다.

---

## 기억장치(메모리)란 무엇인가?

### User Answer
CPU에서 처리된 결과를 저장하는 장치다.

기억장치(메모리)는 주기억장치와 보조기억장치로 구분할 수 있다.
주기억장치는 전원 공급이 중단되면 데이터가 사라지며, 데이터를 읽고 쓰는 데 걸리는 시간이 빠르다.
보조기억장치는 전원 공급이 중단되어도 데이터가 유지된다는 특징이 있으며, 데이터를 읽고 쓰는 데 걸리는 시간이 비교적 느리다.

---

## I/O Device란 무엇인가?

### User Answer
사용자와 컴퓨터의 상호작용을 위한 장치다.
사용자로부터 입력을 받아서 컴퓨터로 전달하고, 컴퓨터에서 나온 출력을 사용자에게 보여준다.

---

## 컴퓨터가 다루는 정보의 종류에는 무엇이 있는가?

### User Answer
Program Code, 그리고 2진수로 표현된 데이터 2가지가 있다.

---

## Program Code의 종류는 무엇인가?

### User Answer
고급언어 코드, 어셈블리 코드, 기계어 코드 3종류가 있다.

고급언어 코드는 JavaScript 같은 고급언어로 작성되며, 개발자가 이해하기는 쉽지만 컴퓨터가 이해할 수는 없다.
그래서 컴파일러를 통해 컴퓨터가 이해할 수 있는 기계어로 번역한다.
기계어는 CPU마다 다르기 때문에, 고급언어와 기계어의 중간인 어셈블리어가 존재한다.

---

## 컴파일러란 무엇인가?

### User Answer
고급언어로 작성된 코드를 **기계어 또는 어셈블리언어로** 변환해주는 소프트웨어다.
컴파일러는 크로스 컴파일러와 네이티브 컴파일러 2가지로 분류할 수 있다.

---

## 크로스 컴파일러와 네이티브 컴파일러의 차이는?

### User Answer
크로스 컴파일러는 컴파일러가 실행되는 컴퓨터와 컴파일 결과물(기계어)이 실행되는 컴퓨터가 다른 경우에 사용하는 컴파일러다.
네이티브 컴파일러는 같은 컴퓨터에서 컴파일하고 실행되는 경우에 사용하는 컴파일러다.

예를 들어, 윈도우 OS에서 작성한 소스코드를 안드로이드 OS에서 실행될 파일로 컴파일하려면 크로스 컴파일러를 사용해야 한다.

## 인터프리터란 무엇인가?

### User Answer
고급언어를 한 줄씩 읽고 바로 실행해주는 소프트웨어다.

---

## 인터프리터와 컴파일러의 장단점은 무엇인가?

### User Answer
전체를 다 **실행**하는 속도만 따진다면 인터프리터보다 컴파일러가 더 빠르다.
하지만 컴파일하는 시간까지 함께 따져본다면, 긴 코드를 컴파일하느라 실행하지 못하고 있을 때 인터프리터는 즉시 한 줄씩이라도 실행할 수 있다는 차이점이 있다.

---

## 고급언어에서 기계어로 번역되는 과정은 어떻게 되는가?

### User Answer
고급언어 코드의 의미는, 기억장치의 주소값 X에 저장된 값과 주소값 Y에 저장된 값을 더해서 주소값 Z에 저장하라는 뜻이라고 하자.

이에 대응되는 어셈블리 코드의 의미는 다음과 같다.
기억장치의 주소값 X에 저장된 값을 레지스터의 주소값 A에 로드(적재)한다.
기억장치의 주소값 Y에 저장된 값과 레지스터의 주소값 A에 저장된 값을 더한다.
레지스터의 주소값 A에 저장된 값을 기억장치의 주소값 Z에 저장한다.

이에 대응되는 기계어 코드는 8bit를 예시로 들 수 있다.
앞 3비트는 연산코드 필드, 뒤 5비트는 오퍼랜드 필드로 나뉜다고 가정하면,
**연산코드** 필드는 CPU가 수행할 연산(LOAD, ADD, STOR 등)을 나타내고,
오퍼랜드 필드는 **명령어 실행에 필요한** 데이터가 저장된 주소값을 나타낸다.

실제로는 CPU마다 명령어 형식(= 연산코드 필드, 오퍼랜드 필드를 지정하는 방식)이 다르다.
위 예시에서는 연산코드 필드가 3자리이므로 2^3 = 8가지의 연산을 가질 수 있고,
오퍼랜드 필드가 5자리이므로 기억장치의 주소를 2^5 = 32개 가질 수 있다는 것을 나타낸다.

## 시스템 버스는 무엇이며 하위 버스는 어떻게 구성되는가?

### User Answer
시스템 버스 = 통신하기 위한 경로.
시스템 버스는 3가지 하위 버스인 Control Bus, Address Bus, Data Bus로 나눌 수 있다.

- **Control Bus**: 제어 신호를 전달하는 Bus.
- **Address Bus**: 주소값을 보내기 위한 Bus. 주로 메모리의 주소값이나 I/O 디바이스의 포트 번호가 이에 해당된다.
- **Data Bus**: 말 그대로 데이터를 보내기 위한 Bus.

---

## CPU가 시스템 버스와 통신하는 과정은?

### User Answer
주소 버스는 단방향이라서 CPU → I/O Device나 CPU → Memory로 주소값을 전송하는 것만 가능하고, 그 반대는 불가능하다.
데이터 버스와 제어 버스는 양방향성이라서 위의 경우에 보내기와 받기(= 읽기, 쓰기)가 모두 가능하다.

각 버스의 비트 수는 한 번에 보낼 수 있는 정보의 크기를 뜻한다.
예를 들어 주소 버스가 16bit라면, 2^16 = 약 64k개의 주소를 지정할 수 있기 때문에, 해당 CPU에 접속될 수 있는 기억장치의 최대 용량이 64KB라는 것을 의미한다.
마찬가지로 데이터 버스가 32비트라면 CPU가 한 번에 32비트씩 데이터를 전송할 수 있다는 뜻이다.
각 버스의 비트 수는 CPU마다 다르다.

---

## CPU가 Memory와 통신하는 과정은?

### User Answer
CPU와 주소 버스는 단방향이므로 화살표가 단방향이고, 데이터 버스는 양방향이므로 화살표도 양방향이다.

시스템 버스 화살표의 굵기가 쓰기 신호·읽기 신호 화살표보다 두꺼운 이유는 비트 수가 차이가 나기 때문이다.
쓰기·읽기 신호는 1비트만 있으면 되고 0과 1만 있으면 된다.
이에 비해 시스템 버스의 비트 수는 CPU마다 다르지만 1비트보다는 더 크다.

CPU가 Memory에 읽을 때와 쓸 때는 서로 과정에 차이가 존재한다.

---

## CPU가 Memory에 데이터를 쓸 때의 과정은?

### User Answer
CPU가 데이터를 Memory(기억장치 통칭)에 저장할 때는 다음 순서로 진행된다.

먼저 저장할 데이터의 주소값을 주소 버스로 보내고, 저장할 데이터를 데이터 버스로 **동시에** 보낸다.
그리고 동시에 쓰기 신호를 활성화(= 1로 보냄)한다.
이 쓰기 신호는 기억장치에 쓰기가 완료될 때까지 유지된다.

그래서, 기억장치 쓰기 시간(= Memory Write Time)은 CPU가 기억장치로 주소와 데이터를 보낸 순간부터 저장이 완료될 때까지의 시간을 의미한다.

---

## CPU가 Memory로부터 데이터를 읽을 때의 과정은?

### User Answer
CPU가 메모리로부터 데이터를 읽어올 때는 다음 순서로 진행된다.

읽을 데이터가 저장된 주소값을 주소 버스로 보내고, 동시에 읽기 신호를 활성화한다.
그다음 **일정 시간이 지난 후** 기억장치에서 읽힌 데이터를 데이터 버스로 보내서 CPU가 받게 된다.

여기서 일정 시간(= 지연 시간)이 걸리는 이유는 기억장치에서 읽을 데이터를 꺼내는 데 시간이 필요하기 때문이다.

그래서 기억장치 읽기 시간(= Memory Read Time)은 CPU가 기억장치로 주소값을 보낸 순간부터 읽기가 완료될 때까지의 시간을 의미한다.

---

## CPU와 I/O Device가 통신하는 과정은?

### User Answer
CPU가 I/O Device와 통신하는 과정은 Memory와 약간 차이가 있다.
I/O Device와 통신할 때에도 똑같이 시스템 버스를 이용하긴 하지만, I/O Controller를 추가로 이용해야 한다.

I/O Device는 종류도 다양하고 데이터 형식·제어 방식·속도 같은 것들도 모두 차이가 있어서, 시스템 버스가 I/O Device와 바로 연결되는 것이 불가능하다.
그래서 시스템 버스에서 I/O Device로 가기 전에 I/O Controller를 거친다.

I/O Controller는 CPU로부터 입력·출력 명령을 받아서 I/O 장치를 제어하여 명령을 수행한다.
I/O Controller 안에는 Status Register와 Data Register가 있다.

- **Status Register**: 현재 I/O 장치의 상태를 나타내는 bit를 저장한다.
- **Data Register**: CPU와 I/O Device 간에 이동되는 데이터를 임시적으로 저장하는 데 사용한다.

---

## 키보드로 데이터를 입력하는 과정은?

### User Answer
키보드로 어떤 키를 누르면, 그 키에 대한 아스키 코드 값이 데이터 레지스터에 저장되고, 동시에 상태 레지스터의 In Ready 비트가 1이 된다.

CPU는 상태 레지스터 값을 읽어서, 만약 1이 아닌 경우 대기했다가 다시 확인하는 과정을 반복한다.
만약 In Ready 비트가 1이면 데이터 레지스터에 있는 데이터를 전달받게 된다.

---

## 프린터로 출력하는 과정은?

### User Answer
키보드로 입력하는 과정과 비슷하다.
CPU가 프린터 I/O Controller 안에 있는 상태 레지스터의 Out Ready 비트가 1인지 확인하고, 아니면 대기하고 있다가 1이 되면 출력할 데이터를 보내서 프린터 I/O Controller의 데이터 레지스터에 저장시킨다.

그러면 프린터 I/O Controller는 데이터 레지스터에 저장된 데이터를 프린터로 보내고, 프린터 하드웨어를 제어하여 인쇄가 된다.

---

## 프로그램이 실행되는 과정을 간단히 설명하면?

### User Answer
CPU가 주기억장치로부터 프로그램 코드를 읽어서(= CPU와 메모리가 주소 버스, 데이터 버스, 읽기 신호로 통신해서) 실행하는 것이다.
여기서 프로그램은 고급언어로 작성된 소스코드일 수도 있고, 기계어로 작성된 실행 파일일 수도 있다.

CPU로 프로그램을 실행한 그 결과의 데이터를 주기억장치에 저장하게 된다.

---

## 폰노이만 아키텍처란 무엇인가?

### User Answer
현재 범용적으로 가장 많이 쓰이는 컴퓨터 아키텍처다.

크게 3가지로 구분하면 Input Device / CPU / Output Device 로 나눌 수 있다.
그리고 CPU는 다시 CU와 ALU로 나눌 수 있으며, CU는 메모리에서 데이터를 읽어서 ALU로 명령을 보내는 역할을 한다.

폰노이만 구조는 만들기가 단순하고 비용도 비교적 적게 든다.
하버드 구조는 만들기가 복잡하고 비용도 많이 든다.
그래서 하버드 구조는 고속의 I/O가 필수적으로 필요한 시스템에 아주 제한적으로 사용되는데, 대표적인 예가 DSP에 사용되는 마이크로프로세서를 만들 때다.

**장점**
- 컴퓨터에 다른 작업을 시키려고 할 때 하드웨어를 재배치할 필요 없이 소프트웨어(프로그램)만 교체하면 되어 범용성이 좋다.
